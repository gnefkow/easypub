# 5.4 Annotations, Highlights, and Reading Position

When you read a paper book, you can underline a sentence, scribble a note in the margin, fold a corner, or slip in a bookmark. Those markings are literally part of the object you own. EPUBs feel similar when you use them: you highlight a paragraph, add a note, and the app shows that highlight every time you open the book. But under the hood, the EPUB file usually stays unchanged. Your highlights and annotations live somewhere else.

This matters for two reasons. First, it changes what “owning” your reading data means. If the EPUB file itself does not contain your notes, then moving the book file to another device does not automatically move your highlights. Second, it makes “where exactly is this highlight?” a technical problem. In a web page, you can point at an element with an `id` or a CSS selector. In an EPUB, content is split across multiple XHTML documents, can reflow into different “pages” depending on font size and screen size, and may be modified by the reading system (for example, for pagination or accessibility).

This chapter explains the ecosystem reality: what highlights and annotations are, where they are stored, how reading systems address precise locations using CFIs (Canonical Fragment Identifiers), how the W3C Web Annotation Data Model relates to EPUB, and why exporting and syncing notes is harder than it looks. We’ll also cover reading position and bookmarks, which are closely related: they are “annotations” too, just with a different user experience.

## Key vocabulary (for this chapter)

An **annotation** is any user-created note attached to a location in a text, such as a comment, a sticky note, or a tagged remark. A **highlight** is a specific kind of annotation where the user marks a range of text (often with a color) to indicate importance. A **bookmark** is a saved location in a book that the user can return to, typically representing a single point rather than a text range. **Reading position** (sometimes called “last read location”) is the current location the reader app tracks so it can resume where you left off. A **reading system** is the app/device software that opens and renders the EPUB (Apple Books, Kobo, Google Play Books, Thorium, etc.). A **CFI** (**EPUB Canonical Fragment Identifier**) is a standardized way to address a precise position inside an EPUB’s content, including a character offset within a text node. The **W3C Web Annotation Data Model** is a web standard for representing annotations as structured data (often JSON-LD), with a “target” that identifies what is being annotated and a “body” that contains the annotation content.

## What highlights and annotations “are” in the EPUB ecosystem

At a user-interface level, highlights and annotations are simple: the reader selects some text and asks the app to remember it. Some apps let the user add a note (a text comment) to a highlight. Others allow standalone notes attached to a point in the text (like dropping a pin). Many apps also let the user categorize highlights (colors) or assign tags.

At a data level, every highlight needs at least:

- A way to identify *which book* the highlight belongs to.
- A way to identify *where* in the book it belongs (a location or range).
- Optional metadata: highlight color, creation time, last updated time, the selected text itself (useful for display and search), and user note text.

In print, “where” is a page number plus a location on that page. In EPUB, page numbers are unstable because reflow changes the number of screens. So the ecosystem uses two broad strategies:

- Use an internal location scheme that stays stable across reflow (CFI is the standard option).
- Use a “best effort” scheme based on the selected text (quote matching) plus surrounding context, then re-find it later.

Most modern systems use some combination of the two, because each approach has failure modes.

## Where highlights are stored: reading system responsibility, not EPUB content

The EPUB specification does not define a required way to store user highlights and notes *inside the EPUB file itself*. In practice, reading systems treat annotations as user data that belongs to the app/account, not to the book file. That means:

- Your highlights are typically stored in an app-specific local database (on device) and/or synced to a vendor’s cloud.
- They may be tied to your login account, subscription, or DRM licensing state.
- They are often not portable by default, even if the EPUB file is portable.

There are good reasons for this design:

- Changing the EPUB file would make it no longer match the distributor-provided artifact, which can be problematic with signatures, DRM wrappers, or “content integrity” expectations.
- User data often needs privacy protections, encryption, and account-level sync, which is outside the EPUB container’s scope.
- Multiple users can read the same book file on a shared device; reading systems want per-user annotation sets.

But there is a trade-off: if you copy `book.epub` to a new app, your highlights will not appear unless the new app supports importing annotations from the old one, or the old one exports them in some interoperable format.

## Addressing specific locations: CFIs (Canonical Fragment Identifiers)

If a reading system stores user data externally, it still needs a stable pointer into the book. That is where CFIs come in.

An EPUB Canonical Fragment Identifier is like a URL fragment identifier, but specialized for EPUB’s structure. A normal fragment like `chapter1.xhtml#section3` points to an element with an `id`. That can work when the target element has a stable `id`, but it breaks down when you need to point to “the 14th character in the 3rd paragraph of this chapter,” or when the content does not have stable `id` attributes everywhere.

CFIs solve this by describing a path through:

- The EPUB package document’s spine (which content document is being referenced).
- The content document’s DOM tree (which nodes inside the XHTML).
- A precise character offset (or range) inside a text node.

CFIs are usually written with an `epubcfi(...)` wrapper. Here are some simplified examples:

```text
epubcfi(/6/2[chapter1]!/4/2/6)
```

This is not meant to be human-friendly. It is meant to be machine-stable. You can think of it as “walk to the 6th spine item, then into the referenced XHTML, then down through child nodes until you reach the target.”

Highlights need to represent a *range*, not just a point. CFIs can represent ranges in a couple of ways. A common pattern is “start CFI, end CFI” stored as a pair. Some systems also use a CFI “range” syntax that encodes both endpoints in one string.

### Why CFIs are useful (and why they are not perfect)

CFIs are designed to survive reflow, because they address the logical structure of the content rather than “page 53.” If the user increases the font size, your CFI still points into the same paragraph and character range.

But CFIs can be fragile across *content changes*. If the underlying XHTML changes (for example, you ship a revised edition, fix typos, or the reading system injects markup), the exact DOM paths and offsets may shift. The CFI might still resolve to a nearby location, or it might fail, or it might resolve to the wrong text.

That is why many systems also store the selected text (the quote) and sometimes its surrounding context. If the CFI fails, the system can fall back to “search for this quote in this chapter and pick the best match.” This is similar to how browser-based annotation systems work on ordinary web pages.

### CFIs and “which book is this?”

An annotation needs a stable book identifier in addition to a location identifier. In vendor ecosystems, the “book ID” is usually vendor-specific (a store item ID, an account library ID, or a DRM license identifier). For sideloaded EPUBs, the reading system may use a hash of the file, the OPF `dc:identifier`, the file path, or a combination.

The key practical point is that “same content” is not always treated as “same book.” If you have two different EPUB files with the same text (or the same EPUB file but packaged slightly differently), a reading system might treat them as different books and keep annotations separate.

## The W3C Web Annotation Data Model (and how it relates to EPUB)

The Web Annotation Data Model is a W3C standard that describes annotations in a way that is not tied to a single app. Conceptually, it is very similar to what EPUB reading systems do internally:

- The **target** says what is being annotated.
- The **body** contains the annotation content (for example, a note the user typed).
- Metadata captures author, timestamps, motivation (“highlighting”, “commenting”), and so on.

In practice, many reading apps do not export annotations as Web Annotation objects. But the model is still important because it provides a shared vocabulary for interoperability projects, research tools, academic platforms, and standards-adjacent work.

Here is a simplified JSON example showing how a highlight with a note might be represented:

```json
{
  "@context": "http://www.w3.org/ns/anno.jsonld",
  "id": "urn:uuid:9e6e3b2f-3f1a-4e2b-8b5f-9b0c45d7d6e1",
  "type": "Annotation",
  "motivation": ["highlighting", "commenting"],
  "created": "2026-02-18T12:00:00Z",
  "body": [
    {
      "type": "TextualBody",
      "purpose": "commenting",
      "value": "This definition connects to the earlier chapter."
    }
  ],
  "target": {
    "source": "urn:epub:bookid:example",
    "selector": [
      {
        "type": "FragmentSelector",
        "conformsTo": "http://www.idpf.org/epub/linking/cfi/epub-cfi.html",
        "value": "epubcfi(/6/2[chapter1]!/4/2/6)"
      },
      {
        "type": "TextQuoteSelector",
        "exact": "A canonical fragment identifier",
        "prefix": "In EPUB, ",
        "suffix": " is used to"
      }
    ]
  }
}
```

Notice two things:

First, the target uses a **FragmentSelector** with a CFI value. That is the EPUB-specific precision pointer.

Second, it also includes a **TextQuoteSelector** (exact text plus surrounding context). That gives resilience when the DOM path changes, or when you want to re-anchor the annotation in a different rendering pipeline.

This “multiple selectors” approach matches the real-world lesson: no single anchoring mechanism is perfect.

## How reading apps handle annotation storage and sync

Different reading systems treat annotation storage as a product feature, not as a pure file-format concern. This is why behavior varies so much:

- Some apps store annotations only on the device (local-only). This is common in lightweight EPUB viewers and some desktop tools.
- Some apps sync annotations through a cloud account. This is common in vendor ecosystems and any product with cross-device reading.
- Some apps sync only within their own environment (for example, only to the same vendor’s apps) and do not provide full export.

Sync sounds straightforward (“upload your highlights”), but it has hard edges:

- **Versioning**: What happens if the book file changes? Does the system treat it as the same book? Can it migrate annotations?
- **Conflict resolution**: If you highlight on two devices while offline, what wins?
- **Identity**: Is the identity “user account + vendor book ID”? If so, sideloaded EPUBs often have weaker identity guarantees.
- **Privacy and DRM**: Notes can reveal what a user is studying or thinking. Some platforms restrict export for privacy or business reasons, and DRM ecosystems add additional constraints.

If you are building a reading system yourself (or a feature that touches this world), the big architectural insight is that annotations are more like “user documents” than like “book content.” They deserve a data model, migrations, backup, and robust sync logic, just like any other user-generated data.

## Exporting and sharing annotations: why portability is hard

Users often want to take their highlights out of a reading app: to study, to write notes, to cite sources, or to move to a new platform. This is where the difference between EPUB-as-a-file and reading-system-as-a-service becomes painfully visible.

Portability is hard because “a highlight” is not just text. It is text plus a location that makes sense in a particular edition of a particular book. When you export highlights, you have to decide what to include:

- The selected text (quote).
- The note body (if any).
- A stable location pointer (CFI, spine href + fragment, or a page/loc number).
- Context (chapter title, nearby text, headings).
- A book identifier (title/author plus a stable ID if available).

If you include only the selected text, it is readable but hard to trace back to the exact location later. If you include only a CFI, it is precise but often meaningless outside the app that can resolve it against the exact same EPUB.

The most practical export formats in the wild tend to be “human-friendly”: a text file, CSV, HTML, or a vendor-specific format that prioritizes readability over perfect re-anchoring. Interoperable machine-friendly exports (like Web Annotation JSON-LD) are possible, but they only help if there is an importing system that understands them and can resolve the selectors.

## Reading position and bookmarks: how readers track where you left off

Reading position is the most common “annotation” a user never thinks about. When you close a book and reopen it, the app needs to jump to “where you were.” Unlike a highlight, reading position is usually:

- A single point, not a range.
- Updated frequently (often every page turn, scroll, or time interval).
- Tied to a particular reading mode (scrolling vs paginated) and sometimes to a specific font size/layout.

Good reading systems separate “user intent” from “layout artifacts.” That means they store a location in a layout-independent way (often a CFI point), and then at render time they compute whatever is needed to show that location on the current device in the current settings.

Bookmarks are similar, but more deliberate. A bookmark is a user-created saved point that should remain stable even as the reading position moves on. Bookmarks are commonly represented as a CFI point plus some display metadata (like the text of the paragraph, the chapter title, or a short excerpt shown in the bookmark list).

Some ecosystems also expose “locations” or “percent read” metrics. These can be derived from spine progression and are helpful for UI (“you’re 43% done”), but they are not reliable as a precise pointer for annotations. Two reading systems can compute “location 1234” differently even for the same EPUB.

## Common pitfalls and gotchas

The first pitfall is assuming highlights live inside the EPUB file. In most real reading apps, they do not. If your product or workflow depends on portability, plan explicitly for export/import and do not rely on “just move the EPUB.”

The second pitfall is over-trusting any single location mechanism. CFIs are powerful, but they can break when content changes, when the DOM differs across conversion pipelines, or when an app injects markup. Quote-based anchoring is resilient, but it can produce ambiguous matches in repetitive text (“Chapter 1”, “Introduction”, common phrases). Storing both is usually safer.

The third pitfall is weak book identity. Two EPUBs with the same text may not be “the same book” to a reading system. If you are building annotation features, you need a clear strategy for identifying books and for migrating annotations across editions or repackaged files.

The fourth pitfall is confusing “page numbers” with stable locations. In reflowable EPUBs, “page” is a rendering artifact, not a property of the book. If you show page numbers, treat them as a UI convenience and store an underlying location that survives reflow.

The fifth pitfall is ignoring privacy and security. Annotations are user-generated content. They may contain personal notes, study habits, and sensitive information. If you sync annotations, treat them like any other user data: authenticate properly, consider encryption, handle deletion, and be careful about what is shared by default.

The sixth pitfall is failing to test annotation behavior across devices and rendering modes. A highlight that looks correct in a desktop reader might wrap differently on a phone, and a range that anchors correctly in paginated mode might behave differently in continuous scrolling mode. Test the anchoring logic against the variability that real users will produce.

When you treat annotations, highlights, bookmarks, and reading position as a layer above the EPUB file—anchored into it, but not stored inside it—you can reason about portability, sync, and interoperability more clearly. EPUB provides tools like CFIs to make precise anchoring possible, and the broader web ecosystem provides models like Web Annotation to represent the data. The remaining challenge is product reality: many systems are not designed to share, so portability is as much about ecosystem choices as it is about standards.
