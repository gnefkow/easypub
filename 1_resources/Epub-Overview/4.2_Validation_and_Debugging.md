# 4.2 Validation and Debugging

If you have ever built a web page, you already know the emotional shape of debugging: something looks wrong, you change a line, it fixes one thing and breaks another, and you gradually narrow in on the real cause. EPUB debugging feels similar, but with one extra layer that makes it confusing at first. You are not only debugging HTML and CSS; you are also debugging a package format with strict rules about how files are stored, referenced, and declared.

That is why EPUB problems can feel “mysterious.” The EPUB might open in one reading system but not another. The cover might show in a library view but not appear as a page. A chapter might render blank because of a single XML character error. The best way to remove the mystery is to develop a repeatable workflow: validate, unpack, inspect, fix, repackage, validate again, and then test in a couple of real readers.

This chapter introduces that workflow and the tools that support it, with an emphasis on how to interpret validator output and translate it into concrete fixes.

## Key vocabulary (for this chapter)

**Validation** is the process of checking whether an EPUB conforms to the rules of the EPUB specifications (and the underlying container format). **EPUBCheck** is the official open-source validator that most of the publishing world relies on. A **well-formed** document is valid XML at the syntax level: tags must be properly nested, attribute values must be quoted, special characters must be escaped, and empty elements must be closed. An **error** is a problem that makes the EPUB invalid or likely to fail in many reading systems. A **warning** is a suspicious pattern that might work but is risky or non-ideal. An **info** message is a diagnostic note. A **manifest entry** is an `<item>` in the OPF manifest; the manifest is the inventory of all resources in the EPUB package. A **media type** (or MIME type) is the string that declares what a file is (for example, `application/xhtml+xml` for an XHTML content document, or `image/svg+xml` for an SVG image).

## What EPUBCheck is (and what it is not)

EPUBCheck is best thought of as a spec-focused, ecosystem-tested “lint tool” for EPUB packaging. It can tell you when your EPUB violates rules like “a file referenced by the spine is missing,” “a resource is not declared in the manifest,” or “this XHTML is not well-formed XML.”

It is not a guarantee that every reading system will render your book identically. Validation is necessary, but not sufficient. Some reading systems are forgiving and will display an EPUB that is technically invalid. Others are strict and will refuse. And there are many “valid but ugly” outcomes that a validator cannot catch, like fonts being ignored or CSS being overridden. Still, if you are trying to ship a robust EPUB, you almost always start with EPUBCheck, because it catches the highest-impact failures early.

## Running EPUBCheck: command line, GUI wrappers, and online validators

EPUBCheck is available as a command-line tool (commonly distributed as a `.jar`), and many other tools wrap it. For example, some editors include a “Validate EPUB” feature, some build pipelines run EPUBCheck automatically, and some websites let you upload an EPUB and see the results.

If you run it from the command line, the core idea is: you point EPUBCheck at a `.epub` file, and it produces a report that includes file paths and line/column positions for problems inside XML-based files.

A typical command looks like this:

```bash
java -jar epubcheck.jar path/to/book.epub
```

You may also see a version flag and an output format option, depending on the distribution you are using. The details vary, but the mental model stays the same: EPUBCheck reads the container, follows the references, parses the XML, and then emits messages with enough context to locate the broken part.

If you prefer a GUI, the trade-off is usually convenience versus control. A GUI can be easier for occasional checks, but the command line is easier to integrate into repeatable workflows and easier to run “again and again” while you iterate. In practice, many teams do both: GUI during editing, command line in CI or export pipelines.

## Understanding EPUBCheck output: errors vs warnings vs info

The single most important debugging skill is to not panic when you see a wall of validator output. Many errors cascade. One broken reference can cause a dozen “file not found” messages, and one malformed XHTML document can lead to follow-on errors because the parser cannot reliably understand what comes after the first syntax failure.

A practical rule is: fix the first root-cause error in the output first, then rerun validation. Repeat until the remaining messages are truly independent problems.

EPUBCheck output generally includes three pieces of information: the severity (error, warning, info), the file involved (often a path inside the EPUB), and a short description of what was wrong. When the issue is inside XHTML, OPF, NCX, or nav documents, you often also get a line number and column number. Treat those line/column positions as “where the parser noticed something wrong,” not always “the exact location of the original mistake.” XML parsers sometimes report the error at the point where the structure becomes impossible, which can be a few characters after the real cause.

It is also common for EPUBCheck to include a short code for the type of problem. The exact codes vary by version, but the pattern is stable: the code tells you which subsystem complained (resource, OPF, nav, etc.), and the message tells you what was wrong. A simplified example might look like this:

```text
ERROR(RSC-005): text/chapter-03.xhtml(214,37): Error while parsing file: The entity name must immediately follow the '&' in the entity reference.
ERROR(OPF-014): content.opf(78,52): Item 'images/fig-01.png' is referenced but not declared in the manifest.
WARNING(HTM-012): text/chapter-05.xhtml(12,1): The 'lang' attribute should be present.
```

The first error is a classic XHTML well-formedness issue: an unescaped `&` somewhere near line 214. The second error is an OPF bookkeeping problem: a file is used but the manifest does not list it. The warning is a “quality” hint rather than a hard failure: the document will often render, but language metadata is important for pronunciation and accessibility. When you learn to map these messages to categories, validator output stops feeling like noise and starts feeling like a set of very specific instructions.

## The debugging workflow: unzip → inspect → fix → re-zip → validate → repeat

When an EPUB is failing, the fastest way to understand it is to unpack it and look at the actual files. That is not a special trick; it is the nature of the format. An EPUB is a ZIP archive, so you can unzip it and inspect the folder structure directly.

On macOS and many other systems, you can unzip from the command line:

```bash
unzip book.epub -d book-unpacked
```

Once you have a folder on disk, you can open `META-INF/container.xml` to see which OPF file is the “entry point” for the package. You can open the OPF and follow its manifest and spine to see which resources exist and which are supposed to be in reading order. And you can open the XHTML documents to check whether they are well-formed and whether their references to images and stylesheets are correct.

When you fix something, you then need to repackage the EPUB correctly. This is where EPUB debugging differs from “just editing HTML.” Packaging rules matter, especially the `mimetype` file rule: it must be the first file in the ZIP and it must be stored uncompressed. If you re-zip casually, you can accidentally create an EPUB that looks fine as a ZIP but fails on strict reading systems.

One common repackaging pattern is:

```bash
cd book-unpacked
zip -X0 ../book-fixed.epub mimetype
zip -Xr9D ../book-fixed.epub META-INF OEBPS
```

This does two intentional things. First, it adds `mimetype` without compression (`-0`) and does it first. Second, it adds the rest of the directory tree after. The additional flags are there to reduce platform-specific ZIP metadata and directory entries that can cause compatibility quirks. The exact flags you use may vary by tooling, but the principle should not: make `mimetype` first and uncompressed, then include everything else.

After repackaging, you rerun validation. If you are working on a book, you will do this loop many times. A reliable loop is more valuable than a clever one-off fix.

## Common validation errors and how to fix them

Validators tend to complain about the same categories of problems because the EPUB format has a few central “linkage points.” When one of those linkage points is wrong, the whole book can become unstable.

### Malformed XHTML: “not well-formed” means XML rules, not HTML rules

Many EPUB content documents are XHTML, which is XML. That means the HTML habits you can get away with in browsers become real errors in EPUB. A `<br>` must be `<br />`. Attribute values must be quoted. Tags must nest properly. And special characters like `&` must be escaped unless they are part of a valid entity.

For example, this will often render in a browser but is not well-formed XHTML:

```xml
<p>Research & development</p>
```

In XHTML, the ampersand must be escaped:

```xml
<p>Research &amp; development</p>
```

When EPUBCheck reports a well-formedness error, fix that first, because a single XML syntax error can make an entire chapter unreadable or blank in some readers.

### Missing manifest entries: “resource is not declared”

The OPF manifest is the inventory of every file that belongs to the publication. If you add a new stylesheet or image file and forget to add it to the manifest, some reading systems will still load it, but you should not rely on that. Validation will often complain about resources that are referenced from XHTML but not listed in the OPF.

The fix is to add an `<item>` entry with the correct `href` and `media-type`:

```xml
<item id="diagram" href="images/diagram.svg" media-type="image/svg+xml" />
```

Two details matter here. The `href` is resolved relative to the OPF file, not relative to the XHTML file. And the media type string must match the actual format. This is a common place where typos cause hours of confusion.

### Incorrect media types: tiny strings with big consequences

Media types look like boring metadata, but they are part of how reading systems decide how to parse and handle a resource. For example, an SVG should be declared as `image/svg+xml`. A common mistake is to write `image/svg` or `image/xml`. Another common mistake is to declare JPEG as `image/jpg` instead of `image/jpeg`.

Validators catch many of these, and you should treat those messages as actionable. Even when a particular reader seems to “figure it out,” other readers may not.

### Broken references: spine points to missing items, or links point to missing files

The spine is the list of content documents in reading order. Each `<itemref>` points to an `idref`, which must match an `<item>` in the manifest. If you rename a file and update the manifest but forget to update the spine, you can end up with a spine reference that points nowhere. That often looks like a missing chapter in a reader.

Similarly, within XHTML, a link like `href="../images/fig-01.png"` might be correct relative to the XHTML file, but if the file was moved or renamed, the link becomes broken. Validators often catch missing resources when they are part of the manifest/spine graph, but not every internal fragment link is validated equally. This is why the unzip-and-inspect workflow is so helpful: you can see the actual folder structure and verify the paths.

### The `mimetype` problem: “it’s a ZIP” does not mean “any ZIP is fine”

One of the most common “why does this EPUB fail on device X?” bugs is a packaging mistake. The `mimetype` file must be the first file in the archive and it must be stored uncompressed. If you build an EPUB with a generic ZIP workflow (or by zipping the folder in a file manager), you can accidentally compress it or place it later in the archive.

The fix is not to edit XHTML; the fix is to rebuild the EPUB with correct ZIP options. This is also why many production workflows use a dedicated packaging step rather than manual zipping.

### Navigation document issues: present, declared, and referenced correctly

In EPUB 3, the nav document is a real XHTML file that must be present in the package and declared in the manifest with the correct `properties` value (`nav`). If it exists as a file but is not declared correctly, some reading systems will not recognize it as navigation, and validators will complain.

The fix is usually in the OPF manifest, not inside the nav XHTML itself:

```xml
<item id="nav" href="nav.xhtml" media-type="application/xhtml+xml" properties="nav" />
```

The nav document itself should contain the expected `<nav epub:type="toc">` structure (Chapter 2.5), but the most common validation mistake is simply forgetting the `properties="nav"` declaration.

## Tools for inspecting EPUB internals (and why web dev tools still matter)

Even though EPUB is not “the web,” many of the tools you already know are useful, because EPUB content documents are XHTML and CSS. Once you unpack an EPUB, you can open a content document in a browser to inspect the markup and styles. You do have to be aware of differences: some CSS works in a browser but not in a reading system, and some reading systems apply their own styles and pagination model. But browser dev tools are still a fast way to answer questions like “is this class applied?” or “why is this margin showing up?”

XML tools are also valuable because they understand strict well-formedness. A browser might silently correct malformed markup; an XML parser will not. If you are debugging a blank page, a fast sanity check is to run the XHTML through an XML validator or parser. The goal is not to become an XML expert, but to quickly distinguish “this is a styling issue” from “this document is not parseable.”

Finally, do not underestimate simple text search and careful reading. Many EPUB issues are just broken paths, inconsistent capitalization (`Images/` vs `images/`), or mismatched IDs between manifest and spine. Validators can point you at the symptom, but you still need to follow the references and make the underlying graph consistent.

## A realistic test mindset: validate, then test in a couple of readers

After you have a clean EPUBCheck run, you still want to open the EPUB in at least a couple of reading systems. Validation can tell you whether the EPUB is structurally sound, but it cannot tell you whether your CSS is being overridden aggressively, whether embedded fonts are ignored on a device, or whether pagination breaks create awkward layouts.

The goal is not to test on every device on earth. It is to test enough to catch the major engine differences. A common strategy is to test on one Readium-based reader (like Thorium) plus one “big ecosystem” reader you care about (Apple Books, Kobo, Google Play Books, or Adobe Digital Editions depending on your target).

## Common pitfalls and gotchas

The first gotcha is treating EPUBCheck output as a to-do list you can fix in any order. Many messages cascade from one root problem, so you save time by fixing the earliest parse failures first (especially XML well-formedness) and rerunning validation frequently.

The second gotcha is mixing up path resolution rules. In XHTML, `src` and `href` paths are relative to the XHTML file. In the OPF manifest, `href` paths are relative to the OPF file. A book can look “internally consistent” in one place and still be broken because you updated one set of paths but not the other.

The third gotcha is case sensitivity. Your development environment might treat `Images/cover.jpg` and `images/cover.jpg` as the same path, but many reading systems do not. If a validator says “file not found” and you are sure it exists, check the exact capitalization.

The fourth gotcha is packaging errors during re-zipping. You can fix every XHTML and OPF bug in the world and still ship a broken EPUB if `mimetype` is compressed or not first. If a book validates when unpacked and inspected but fails when packaged, suspect your ZIP step.

The fifth gotcha is believing that “valid” means “consistent across readers.” Validation is a baseline, not a finish line. Readers differ in CSS support, font handling, and navigation UI. The gap between “the spec allows it” and “what real devices do” is where the last 10% of quality work happens.

The sixth gotcha is debugging only inside a reading system. Reader apps often hide the underlying file structure, which makes it hard to see what is truly happening. When you are stuck, unpack the EPUB and debug the files directly. The EPUB format was designed to make this possible, and it is often the fastest path to clarity.

Once you have the workflow in your hands—validate, unpack, inspect, fix, repackage, validate again—EPUB debugging stops feeling like black magic. It becomes a set of repeatable checks on a small number of critical invariants: well-formed XHTML, consistent manifest/spine references, correct media types, correct navigation declarations, and correct packaging.
