# 2.4. The OPF Package Document (`content.opf`)

## Key vocabulary

**OPF (Open Packaging Format)** is the specification (and, informally, the file) that describes an EPUB publication: its metadata, its resources, and its reading order.

**Package document** is the OPF XML file that defines the publication. It is usually named something like `content.opf` or `package.opf`, but the filename is not special; the container points to it.

**Package** (in EPUB) is the logical “publication unit” described by the OPF: a set of resources plus rules for how to present them as a book.

**Metadata** is structured information about the publication (title, author, language, identifiers) plus optional data used by stores, libraries, and reading systems.

**Dublin Core** is a widely used set of metadata terms used by EPUB (for example, `dc:title`, `dc:creator`, `dc:language`, `dc:identifier`).

**Identifier** is an ID that uniquely identifies a publication. In EPUB, it often shows up as a `dc:identifier` value, and the OPF marks which identifier is the “unique identifier.”

**UUID** is a type of globally unique identifier string (for example, `urn:uuid:...`). It’s commonly used when a book doesn’t have an ISBN.

**ISBN** is an International Standard Book Number. It’s common for commercially published books, but not required for EPUB validity.

**Manifest** is the inventory of resources in the EPUB package: every XHTML, CSS, image, font, nav document, NCX, etc. Each item has an `id`, an `href`, and a `media-type`.

**Media type** (MIME type) is a standardized string describing the type of a resource, such as `application/xhtml+xml` (XHTML) or `text/css` (CSS). Reading systems use this to decide how to parse a resource.

**Spine** is the ordered list of content documents that form the primary reading flow. The spine points to manifest items by `id`.

**Itemref** is an entry inside the spine (`<itemref idref="...">`) that says “this manifest item is part of the reading order.”

**Properties** is an attribute used on manifest items (and sometimes elsewhere) to mark special roles, like `nav` for the EPUB 3 navigation document or `cover-image` for the cover image resource.

**Guide** is an EPUB 2 legacy section that points to common “landmark” pages like cover, ToC, and start of text. EPUB 3 prefers navigation landmarks, but you still see `<guide>` in the wild.

## Why the OPF is the “brain” of the EPUB

In Chapter 2.3 we built a mental model of EPUB as three layers: the container layer (ZIP + `META-INF/container.xml`), the package layer (the OPF), and the content layer (XHTML, CSS, images, fonts). If the container layer answers “where is the book’s definition?”, the OPF answers “what is this book?”

The reason the OPF feels like the brain is that reading systems don’t treat an EPUB as “a folder of HTML pages.” They treat it as a publication with declared parts. The OPF is the declaration. It tells the reading system what resources exist, what they are, and what order they should be read in.

That means two practical things.

First, you can have perfectly well-formed XHTML and still have a broken EPUB if the OPF is wrong. The reading system might not be able to find a chapter, might skip an image, or might refuse to open the book if the package document violates constraints.

Second, almost every meaningful structural edit to an EPUB eventually touches the OPF, even if you don’t edit it by hand. If you add a stylesheet, the OPF needs a manifest entry. If you split a chapter into two files, the spine needs to change. If you rename a content document, the manifest and spine references need to change.

## Where the OPF lives (and why its filename doesn’t matter)

Beginners often look for `content.opf` at a fixed path like `OEBPS/content.opf`. Many books do use that pattern, but it’s a convention, not a rule.

The only reliable way to find the OPF is to start at `META-INF/container.xml` and read the `full-path` attribute of the `<rootfile>` entry. That value is the authoritative location of the OPF inside the EPUB.

This is worth repeating because it affects tooling. If you write software that assumes the OPF path, it will work for some EPUBs and then fail on others. If you follow `container.xml`, your tool will work across many different “house styles” of directory layout.

## The three big sections: `<metadata>`, `<manifest>`, `<spine>`

The outline of the OPF is simple enough that you can carry it in your head. Almost all OPF files revolve around three sections.

The `<metadata>` section describes the publication.

The `<manifest>` section lists the resources included in the package.

The `<spine>` section defines the reading order by referencing manifest items.

These sections are separate on purpose. Metadata is for identification and rendering behavior. The manifest is for completeness and type correctness. The spine is for book flow. Keeping them separate makes the format easier to validate and easier to reason about.

## A realistic annotated OPF example

OPF is XML, and that can look intimidating at first. The good news is that most of it is predictable and repetitive. The best way to learn it is to look at a full example and understand what each part does.

The example below is not “the only correct OPF,” but it includes the pieces you will see repeatedly, and it’s small enough to be readable.

```xml
<?xml version="1.0" encoding="UTF-8"?>
<package
  xmlns="http://www.idpf.org/2007/opf"
  xmlns:dc="http://purl.org/dc/elements/1.1/"
  version="3.0"
  unique-identifier="pub-id">

  <metadata xmlns:opf="http://www.idpf.org/2007/opf">
    <dc:identifier id="pub-id">urn:uuid:2c6b8c1e-1b7a-4efb-9c7a-9b10a2f7d1c1</dc:identifier>
    <dc:title>Example Book</dc:title>
    <dc:creator id="creator">Alex Author</dc:creator>
    <dc:language>en</dc:language>
    <dc:publisher>Example Press</dc:publisher>
    <dc:date>2026-01-01</dc:date>

    <meta property="dcterms:modified">2026-01-01T12:00:00Z</meta>
  </metadata>

  <manifest>
    <item id="nav" href="nav.xhtml" media-type="application/xhtml+xml" properties="nav"/>
    <item id="css" href="styles/book.css" media-type="text/css"/>

    <item id="cover-image" href="images/cover.jpg" media-type="image/jpeg" properties="cover-image"/>
    <item id="titlepage" href="text/titlepage.xhtml" media-type="application/xhtml+xml"/>
    <item id="ch1" href="text/chapter-01.xhtml" media-type="application/xhtml+xml"/>
    <item id="ch2" href="text/chapter-02.xhtml" media-type="application/xhtml+xml"/>
  </manifest>

  <spine>
    <itemref idref="titlepage"/>
    <itemref idref="ch1"/>
    <itemref idref="ch2"/>
  </spine>

</package>
```

Let’s unpack what you’re seeing, starting at the top.

The root element is `<package>`. Think of it as “this XML file defines a publication package.” It declares namespaces (which are XML’s way of avoiding naming collisions), a version, and a `unique-identifier` attribute.

The `unique-identifier="pub-id"` attribute points to an element inside `<metadata>` that has `id="pub-id"`. This is how the OPF says “this identifier is the canonical one.” The identifier itself can be an ISBN or a UUID (or something else), but it should be stable. If you generate EPUBs programmatically, choosing a UUID is common when no ISBN exists.

Inside `<metadata>`, you see several `dc:*` elements. These are the Dublin Core terms. EPUB doesn’t require you to use a huge set of metadata, but it does expect a sensible baseline like title, language, and identifier.

The `<meta property="dcterms:modified">...</meta>` line is especially important for EPUB 3. Many validators and toolchains expect it, and some reading systems use it as a “package changed” signal. If you edit an EPUB and forget to update the modified timestamp, you can still have a readable book, but you may get validation warnings and, in some systems, confusing caching behavior.

Next is the `<manifest>`. This is a complete list of resources in the package. Each `<item>` has three attributes that matter almost everywhere.

The `id` is the internal identifier used by the spine and other references.

The `href` is the path to the resource, relative to the OPF file’s location.

The `media-type` tells the reading system how to interpret the resource.

Some items have an additional `properties` attribute. In the example, the nav document (`nav.xhtml`) is marked with `properties="nav"`, which tells EPUB 3 reading systems “this is the navigation document.” The cover image is marked with `properties="cover-image"`, which is the EPUB 3 way to designate the cover image resource.

Finally, the `<spine>` lists the linear reading order using `<itemref idref="...">`. These `idref` values must match `id` values from manifest items. The spine is intentionally indirect: it references items by ID rather than repeating file paths. This allows validators and reading systems to confirm that every spine item is actually declared and has a known media type.

In the example, the spine starts with `titlepage`, then `ch1`, then `ch2`. A reading system will treat this as the primary flow when the user reads forward and backward through the book.

## The manifest is “everything,” not just chapters

One of the easiest mistakes for new EPUB authors is to treat the manifest as “the list of chapters.” It isn’t. It’s the list of resources in the package, including resources that are not part of the reading flow.

That includes stylesheets, images, embedded fonts, the navigation document, and sometimes auxiliary pages (like an index) that might not be spine-linear. It also includes “infrastructure” resources like the NCX file in EPUB 2 or compatibility hybrid EPUB 3 files.

From an engineering perspective, the manifest is a lot like a strict asset pipeline. It’s not enough for the file to exist. The publication needs to declare it, type it, and reference it consistently.

## The spine is “what you read,” and it’s stricter than your table of contents

The spine is the reading system’s primary instruction for what to render in sequence. It is not the table of contents. It is closer to a playlist of content documents.

This difference matters because many EPUBs have pages that are read but not shown in the ToC (copyright pages, section breaks, half-title pages). Those pages might still be in the spine. Conversely, a ToC entry might point to an anchor inside a chapter rather than being a distinct spine item.

If you’re building tooling that manipulates EPUB structure, the spine is the place where “book logic” lives. The moment you split a chapter into two files, you’re making a spine change: you’ll add a new manifest item and insert a new spine itemref.

## EPUB 2 features you still see: `toc` and `<guide>`

The OPF file exists in both EPUB 2 and EPUB 3, but there are a few EPUB 2-era features that still appear in real books and that are useful to recognize.

In EPUB 2, the spine often has a `toc` attribute that points to the manifest ID of the NCX file. It looks something like `toc="ncx"` and corresponds to a manifest item like `<item id="ncx" href="toc.ncx" media-type="application/x-dtbncx+xml"/>`.

EPUB 2 also includes an optional `<guide>` section that points to landmark pages such as the cover, the ToC, and the start of the main text. The guide is not the same as the ToC; it’s a set of shortcuts.

In EPUB 3, the nav document and landmarks nav are the preferred mechanisms, but you still see `<guide>` because older tools and workflows produce it, and some publishers keep it for compatibility.

The most important mindset here is compatibility pragmatism: the OPF is the authoritative package definition, but the ecosystem contains a lot of “carryover” structure to support older reading systems.

## How OPF mistakes show up in the real world

When the OPF is wrong, the failure modes can be surprising because they don’t always look like a clean error message. Sometimes the book simply opens but has missing content. Sometimes the ToC appears empty. Sometimes a reader refuses to open the book at all.

As a junior developer, it’s useful to map these failures to the underlying structural issues.

If a chapter is missing, the first suspect is the spine. The chapter file might exist, but if it isn’t in the spine, many reading systems won’t treat it as part of the primary reading flow. It might still be accessible by a link, but it will feel “lost.”

If an image is missing, the first suspect is the manifest. The content document might reference `images/figure-1.png`, but if the OPF doesn’t declare that file (or declares it with the wrong media type), reading systems vary in how forgiving they are. Some will still load it. Some will not. Validators will complain either way.

If navigation is broken, the manifest is again a suspect: EPUB 3 nav documents must be present and declared with `properties="nav"`, and EPUB 2 NCX must be referenced correctly when used. Navigation problems often come down to a resource being present but not “announced” correctly through the OPF.

## Common pitfalls and gotchas

The most common OPF pitfall is failing to keep the manifest, spine, and the actual filesystem in sync. If you rename `text/chapter-01.xhtml` but forget to update the manifest `href`, the book can break in ways that look like “random reader bug.” In reality it’s a broken reference. Similarly, if you add a new file and forget to add it to the manifest, validators will flag it and some reading systems will not reliably load it.

Media types are another high-impact gotcha. A manifest item must declare the correct `media-type`. Declaring an XHTML file as `text/html`, or declaring an SVG as `image/svg` instead of `image/svg+xml`, is the kind of tiny mismatch that can work in one reading system and fail in another. The OPF is where those types are defined, so it’s where these mistakes live.

The `href` paths are also relative to the OPF file’s location, not the container root. If your OPF lives at `OEBPS/content.opf`, then `href="text/chapter-01.xhtml"` resolves to `OEBPS/text/chapter-01.xhtml`. This sounds obvious, but it’s a frequent source of broken links when EPUBs are reorganized or when tooling assumes a different root.

For EPUB 3 specifically, forgetting `dcterms:modified` is an easy mistake. EPUBCheck will warn about it, and some pipelines expect it. If you are rewriting EPUBs in a batch process, always update the modified timestamp when you produce a new output.

Another subtle pitfall is mixing EPUB 2 and EPUB 3 conventions without understanding the consequences. Hybrid navigation (nav + NCX) is common and often helpful, but you want to be deliberate: if you include an NCX, it needs a correct manifest item and, in EPUB 2-style packaging, the spine may reference it. If you include a nav document, it should have `properties="nav"` and actually contain a ToC structure. Half-doing both can leave you with a book that “sort of” works but fails validation and behaves inconsistently across readers.

Finally, don’t assume that because a book opens in one app it is structurally correct. Many reading systems are forgiving, and that forgiveness varies widely. The OPF is designed to be validated, so the best practice is to validate after structural changes and treat OPF warnings as real engineering work, not cosmetic issues. The spec exists to reduce ecosystem variance, but only if you actually follow it.
