# 3.7 Multimedia and JavaScript in EPUB 3

EPUB 3 is built on web technologies, so it’s tempting to assume an EPUB can do anything a modern web page can do: play video, run JavaScript apps, fetch data from APIs, and build rich interactive experiences. The spec does move in that direction. EPUB 3 adds native audio and video elements, a standardized “read-aloud” system called media overlays, and explicit support for scripting.

The reality, though, is that EPUB is not “one platform.” It’s a file format consumed by many reading systems with different rendering engines, security policies, battery constraints, DRM constraints, and UI expectations. Multimedia and JavaScript sit exactly at the intersection of those differences, which is why this chapter is as much about engineering judgment as it is about markup.

This chapter explains how to include audio and video in EPUB content documents, how fallbacks work when a format isn’t supported, what media overlays are (and what SMIL has to do with them), and what scripting looks like in EPUB 3—including why “allowed by the spec” often still means “doesn’t work reliably in real readers.” We’ll end with progressive enhancement patterns that keep your EPUB usable when multimedia or JavaScript is unavailable.

## Key vocabulary (for this chapter)

**Multimedia** in EPUB usually means **audio** and **video** resources packaged inside the EPUB and referenced from content documents. A **fallback** is alternate content provided for cases where a reading system can’t play a media file or doesn’t support a particular feature. A **media overlay** is EPUB’s mechanism for synchronizing text with audio narration so a reading system can highlight text while playing audio (“read-aloud”). **SMIL** (Synchronized Multimedia Integration Language) is the XML format EPUB uses to describe that synchronization.

In EPUB scripting discussions you’ll see **Scripted Content Document**, which is an EPUB content document that contains scripts and/or HTML form elements and is identified in the OPF manifest with `properties="scripted"`. A **container-constrained script** is JavaScript that is limited to the EPUB container’s own resources and context (think “runs inside the book”), while **spine-level scripting** is scripting that affects the reading flow more broadly across spine items. Finally, **progressive enhancement** is the approach of building content that works without scripting or rich media, then enhancing it when a capable reading system is available.

## Audio and video in EPUB content documents

EPUB 3 content documents are HTML5-based (typically XHTML), so the basic elements for media are the familiar `<audio>` and `<video>` elements. The simplest version of each is straightforward.

Here is an audio example:

```xml
<audio controls="controls">
  <source src="../audio/intro.mp3" type="audio/mpeg" />
  <p>Your reading system does not support audio playback.</p>
</audio>
```

And here is a video example:

```xml
<video controls="controls" width="640" height="360">
  <source src="../video/demo.mp4" type="video/mp4" />
  <p>Your reading system does not support video playback.</p>
</video>
```

Even in these small snippets, there are a few EPUB-specific details worth calling out.

The `controls` attribute is important because it gives the reading system permission to show its standard playback UI. Without it, some reading systems will still play media, but many will not provide a usable control surface.

The `<source>` element includes both a `src` path and a `type`. The `type` is the media type hint. On the web, browsers can sniff formats, but EPUB reading systems often rely more heavily on declared types, and they vary in how forgiving they are. The safest approach is to declare the type and include fallbacks.

Finally, the paragraph inside the media element is a fallback. It’s what the reader sees if the media element can’t be played. Fallback content is not just “nice.” It’s how you keep the EPUB meaningful on devices that simply don’t support video (which is common on e-ink readers) or where playback is disabled by policy.

## Declaring multimedia resources in the OPF manifest

Just like images and fonts, audio and video files should be declared in the OPF manifest with correct media types. Here’s a small illustrative snippet:

```xml
<manifest>
  <item id="ch1" href="text/chapter-01.xhtml" media-type="application/xhtml+xml" />

  <item id="audio-intro" href="audio/intro.mp3" media-type="audio/mpeg" />
  <item id="video-demo" href="video/demo.mp4" media-type="video/mp4" />
</manifest>
```

This does two things. It tells validators and reading systems “these resources are part of the publication,” and it forces you to be explicit about media types, which helps reduce cross-reader ambiguity.

## Fallbacks: treating playback as optional

If you are coming from web development, you might assume media playback is available unless the user blocks it. In EPUB, you should assume the opposite: playback is optional and device-dependent.

That affects your content design. A narrated pronunciation clip can be a nice enhancement. A chapter that *requires* video to be understood is much riskier unless you control the reading environment (for example, an educational platform with a known app).

In practice, the best EPUB multimedia design follows two rules:

First, include meaningful text fallbacks for every media element. “Unsupported” messages are better than nothing, but often you can do better: include a short transcript, a description of what the audio conveys, or a pointer to an alternate representation inside the book.

Second, don’t rely on network access. EPUBs are files. Many reading systems run with restricted networking, offline assumptions, or privacy policies. If you want reliable playback, package the media in the EPUB and reference it with relative paths.

## Media overlays: synchronized text + audio narration

Media overlays are one of the most “book-native” multimedia features in EPUB 3. Instead of embedding audio players in the text, media overlays describe how an audio narration aligns with specific elements of the text so a reading system can play audio while highlighting the corresponding words or phrases.

The “overlay” itself is typically expressed in a SMIL document (`.smil`). The SMIL file does not contain the text; it contains timing and references to text fragments (by `id`) and to audio clip segments.

Here is a simplified conceptual example. Suppose your XHTML has explicit `id` values on the parts you want to synchronize:

```xml
<p>
  <span id="w1">EPUB</span>
  <span id="w2">is</span>
  <span id="w3">a</span>
  <span id="w4">container</span>
  <span id="w5">format.</span>
</p>
```

Now a SMIL file can point at those fragment identifiers and describe the timing against an audio file:

```xml
<smil xmlns="http://www.w3.org/ns/SMIL" version="3.0">
  <body>
    <seq>
      <par>
        <text src="chapter-01.xhtml#w1" />
        <audio src="../audio/ch01.mp3" clipBegin="0:00:00.000" clipEnd="0:00:00.350" />
      </par>
      <par>
        <text src="chapter-01.xhtml#w2" />
        <audio src="../audio/ch01.mp3" clipBegin="0:00:00.350" clipEnd="0:00:00.500" />
      </par>
      <!-- ...more segments... -->
    </seq>
  </body>
</smil>
```

This is not meant as a full production-ready SMIL file, but it shows the core idea: the SMIL file maps “this text fragment” to “this time range in the audio file.”

### Packaging media overlays in OPF

Media overlay documents are also declared in the OPF manifest, typically with `media-type="application/smil+xml"`. The content document that has the narration can reference its overlay via the OPF `media-overlay` linkage.

Conceptually, you might see something like:

```xml
<manifest>
  <item id="ch1" href="text/chapter-01.xhtml" media-type="application/xhtml+xml" media-overlay="mo-ch1" />
  <item id="mo-ch1" href="text/chapter-01.smil" media-type="application/smil+xml" />
  <item id="audio-ch1" href="audio/ch01.mp3" media-type="audio/mpeg" />
</manifest>
```

The key principle is consistency: stable `id` targets in XHTML, a SMIL file that references those targets, and an OPF manifest that declares all the pieces.

Media overlays are powerful when they work, but support varies. Some readers implement read-aloud beautifully; others ignore overlays entirely. As with every EPUB feature, the safest mindset is: overlays are an enhancement, not the only way the content can be consumed.

## JavaScript in EPUB 3: what’s possible vs what’s reliable

EPUB 3 allows JavaScript, but many reading systems restrict it. Restrictions exist for good reasons: security (books from the internet shouldn’t behave like arbitrary apps), privacy (no silent tracking or fingerprinting), battery/performance, and DRM policy.

When JavaScript is supported, it typically runs inside a constrained environment, closer to a sandboxed webview than to a full browser. Network access may be limited. Persistent storage may be limited. Some APIs you expect on the web may be missing. And some readers disable scripting entirely.

That’s why scripting in EPUB is best treated as progressive enhancement. Use it to add optional interactivity—like a glossary popover, a quiz, or a small visualization—but keep the core reading experience intact without it.

### Declaring scripted content in the OPF manifest

If a content document is scripted (contains scripts and/or HTML form elements), EPUB provides a standard manifest signal: the `scripted` property.

For example:

```xml
<item id="interactive"
      href="text/interactive.xhtml"
      media-type="application/xhtml+xml"
      properties="scripted" />
```

This doesn’t force scripts to run. It tells reading systems and tools that this resource expects scripting capability, so they can make better decisions (for example, warning users, adjusting security posture, or falling back).

### Container-constrained vs spine-level scripting (a useful mental model)

In practice, you can think of two broad categories of scripting.

Container-constrained scripting is local. It enhances a single document or a small component: toggling visibility, running a quiz, highlighting terms, or showing a modal. It should not assume network access, and it should not assume it can control global reading system UI. This is the most realistic form of EPUB scripting.

Spine-level scripting is global. It assumes it can coordinate across multiple spine items, manipulate navigation, or behave like a multi-page web app. This is the least reliable form of EPUB scripting, because reading systems manage spine navigation, pagination, and memory aggressively. Many readers unload documents when you navigate away, and they do not want a book to behave like a background application.

If you want the best chance of an interactive feature working across readers, design it to be container-constrained and local, with graceful fallback to plain content.

## Progressive enhancement patterns that work in EPUB

Progressive enhancement is not a buzzword in EPUB. It’s survival.

If you include audio, also include text that communicates the same core information (transcript, summary, or alternate explanation). If you include video, include a description and ensure the surrounding chapter still stands alone.

If you include JavaScript, make sure the content still functions as a normal XHTML document. For example, build a glossary as a normal linked section first, then add JavaScript to show popovers when supported. Build quizzes as normal “question then answer” content first, then add JS to reveal answers interactively.

This approach also helps with validation and debugging. When a feature doesn’t work in a particular reading system, the book is still readable. The interactivity becomes a nice extra, not a failure mode.

## Common pitfalls and gotchas

The first gotcha is assuming modern browser support. Many reading systems are not “latest Chrome.” Even when a reader supports JavaScript, it may be an older engine or a constrained environment. Keep scripts small, avoid heavy dependencies, and avoid relying on cutting-edge APIs.

The second gotcha is relying on network access. A script that fetches JSON from an API, or a video tag that points to `https://...`, may work on your laptop and fail on a device. If the media matters, package it inside the EPUB. If external content is optional, treat it as an enhancement.

The third gotcha is missing or incorrect manifest metadata. Multimedia files and SMIL documents should be declared in the OPF manifest with correct media types. Scripted documents should be marked with `properties="scripted"` when appropriate. When your metadata is incomplete, different reading systems behave differently, and debugging becomes guesswork.

The fourth gotcha is fragile fragment identifiers. Media overlays depend on stable `id` values in your XHTML. If your editing workflow regenerates content documents and changes IDs, your SMIL references silently break. Treat IDs like an internal API: stable names other parts of the book depend on.

The fifth gotcha is performance. Audio and video balloon file size. Large media also increases decode cost and battery usage. On constrained devices, that can mean stuttering playback or crashes. Optimize media and keep it optional.

The sixth gotcha is accessibility. A video without captions or a transcript is a barrier. An interactive widget that only works with touch and doesn’t expose a meaningful reading order is a barrier. EPUB’s ecosystem includes screen readers and keyboard navigation on some platforms, and your enhancements should not make the book harder to read.

Finally, remember the gap between the spec and the real world. EPUB 3 can do multimedia and JavaScript. The question is not “can I include it?” but “will it work in the reading systems my audience actually uses?” When you build multimedia and scripting features as progressive enhancement—with clear fallbacks—you get EPUBs that remain solid everywhere and feel impressive where support is strong.
