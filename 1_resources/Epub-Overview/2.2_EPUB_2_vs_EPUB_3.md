# 2.2. EPUB 2 vs EPUB 3

## Key vocabulary

**EPUB 2** is the older major version of the EPUB standard (widely associated with EPUB 2.0/2.0.1 era). It is centered around XHTML content documents, OPF packaging, and NCX navigation.

**EPUB 3** is the modern major version of the EPUB standard (EPUB 3.0 and later revisions like 3.1, 3.2, and 3.3). It aligns EPUB more closely with the modern web platform and introduces the XHTML navigation document (“nav doc”) as the primary navigation mechanism.

**OPF** (Open Packaging Format) is the package document format used to describe a publication: its metadata, manifest (resource inventory), and spine (reading order). Both EPUB 2 and EPUB 3 use an OPF package document, but the details differ in important ways.

**NCX** is the EPUB 2 table-of-contents format (`.ncx`), an XML file that contains a `<navMap>` made of nested `<navPoint>` entries.

**Navigation document (nav doc)** is the EPUB 3 XHTML file that expresses navigation using normal HTML structures (lists and links) plus EPUB semantics like `epub:type="toc"`.

**Media overlay** is an EPUB 3 feature for synchronizing text and audio narration, typically using SMIL (Synchronized Multimedia Integration Language) files.

**Fallback** (in EPUB context) means a compatibility mechanism where an EPUB 3 publication includes older structures (most commonly an NCX) so that older reading systems can still navigate the book.

## Why the version distinction matters everywhere

It’s tempting to treat “EPUB 2 vs EPUB 3” as trivia, like “which year was the standard published.” In practice, the version affects almost every decision you make as an EPUB author, tool builder, or debugger.

The reason is that EPUB is a stack of layers, and version differences show up at more than one layer. At the packaging layer (the OPF), version changes affect which navigation structures are required or expected. At the content layer (the chapter XHTML), version changes affect what markup is safe to rely on and what readers are likely to support.

Even if you never say “EPUB 2” out loud, your EPUB pipeline still makes version decisions implicitly. If you generate an NCX, you are taking on EPUB 2-era navigation. If you generate a nav document, you are taking on EPUB 3 navigation. If you assume JavaScript will run, you are assuming an EPUB 3-capable and permissive reading system (and even then you’re often betting against reality).

So this chapter has one goal: give you a mental model for what EPUB 2 established, what EPUB 3 changed, and how to choose a target version in 2026 without being surprised later.

## The shared foundation: container + package + content

A reassuring thing to know up front is that EPUB 2 and EPUB 3 share the same foundational “shape.”

Both are ZIP-based containers (Chapter 2.1).

Both have `META-INF/container.xml`, which points to the OPF package document (Chapter 2.3).

Both use an OPF package document to list resources and define the spine (Chapter 2.4).

Both use (X)HTML content documents, CSS, and assets like images and fonts (Part III).

This is why EPUB 3 doesn’t feel like a totally different format. It’s more like “EPUB 2 grew up and adopted modern web practices,” while keeping enough continuity that older tooling and content pipelines could evolve rather than being replaced overnight.

## What EPUB 2 introduced (and why it still matters)

If you open an older book from the late 2000s or early 2010s, you’ll likely find an EPUB 2 publication. EPUB 2 is the version that made EPUB broadly interoperable across many devices and publishers. Three components are especially characteristic.

The first is the OPF package document as the central “inventory + reading order” file. EPUB 2 established the idea that a book is not “a folder of HTML pages,” but a publication with declared parts.

The second is the NCX file. This is the EPUB 2 navigation format. Reading systems in the EPUB 2 era expected a `.ncx` file to build the table of contents UI. If you are shipping to older readers, NCX is often the difference between “the book opens” and “the book is navigable.”

The third is the older content-document worldview. EPUB 2 content documents are usually written as XHTML and treated as XML, with stricter well-formedness rules than the open web. Browsers forgive a lot. Many EPUB reading systems do not.

You’ll sometimes see **DTBook** mentioned in EPUB 2 discussions and in historical tooling. DTBook is an XML-based document format originally associated with DAISY (a digital talking book standard). In practice, most EPUB 2 content you encounter today looks like XHTML rather than DTBook, but DTBook’s presence in the ecosystem helps explain why EPUB developed such a strong “XML toolchain” flavor early on.

## EPUB 3’s philosophy shift: align with the modern web

EPUB 3’s big theme is alignment with the Open Web Platform. Not “turn books into websites,” but “use web technologies to represent documents, with book-specific constraints.”

That alignment shows up in a handful of major feature areas:

Navigation moves from the specialized NCX format to an XHTML navigation document.

Content documents are defined in terms of HTML5-era semantics (while often still authored as XHTML for strictness and tool compatibility).

The spec explicitly accommodates richer media (audio/video) and synchronization features (media overlays).

The format includes semantic enrichment patterns (like `epub:type`) that support accessibility and better reading system behavior.

The practical lesson is that EPUB 3 gives you more expressive power, but it does not magically make reading systems behave like Chrome or Safari. It gives you web-like primitives inside a constrained, inconsistent runtime.

## How to tell which version an EPUB is targeting

The most direct signal is in the OPF package document’s `<package>` element.

An EPUB 2 package often includes `version="2.0"`:

```xml
<package
  xmlns="http://www.idpf.org/2007/opf"
  unique-identifier="BookId"
  version="2.0">
  <!-- metadata / manifest / spine -->
</package>
```

An EPUB 3 package often includes `version="3.0"` (or sometimes later minor revisions depending on tooling):

```xml
<package
  xmlns="http://www.idpf.org/2007/opf"
  unique-identifier="pub-id"
  version="3.0">
  <!-- metadata / manifest / spine -->
</package>
```

The version attribute isn’t just descriptive. It sets expectations for what’s required (for example, EPUB 3 expects a nav document) and for how reading systems should interpret certain structures.

## Navigation: NCX vs nav doc (this is the biggest daily difference)

If you only remember one concrete difference between EPUB 2 and EPUB 3, remember navigation.

EPUB 2 navigation is primarily an NCX file, which is a purpose-built XML vocabulary for table-of-contents trees. Here’s a small NCX snippet:

```xml
<navMap>
  <navPoint id="navpoint-1" playOrder="1">
    <navLabel><text>Chapter 1</text></navLabel>
    <content src="text/chapter-01.xhtml"/>
  </navPoint>
  <navPoint id="navpoint-2" playOrder="2">
    <navLabel><text>Chapter 2</text></navLabel>
    <content src="text/chapter-02.xhtml#section-a"/>
  </navPoint>
</navMap>
```

The meaning is straightforward: each navPoint is a ToC entry. The user-visible label is in `<navLabel>`, and the link target is in `<content src="...">`. Targets can include fragment identifiers, which is how you link to specific sections inside a chapter.

EPUB 3 navigation is primarily a nav document, an XHTML file that looks like normal HTML navigation. Here is the same “idea” expressed as an EPUB 3 ToC nav:

```xml
<nav epub:type="toc" id="toc">
  <h1>Table of Contents</h1>
  <ol>
    <li><a href="text/chapter-01.xhtml">Chapter 1</a></li>
    <li><a href="text/chapter-02.xhtml#section-a">Chapter 2 (Section A)</a></li>
  </ol>
</nav>
```

This feels friendlier to web developers because it is literally a list of links. The trick is that it must carry EPUB semantics. The `epub:type="toc"` marker is what tells a reading system “this nav block is the table of contents.”

There’s one more crucial wiring detail: the nav document must be declared in the OPF manifest with `properties="nav"`. If you forget that, some reading systems will ignore the nav doc even if the HTML looks correct.

```xml
<item id="nav"
      href="nav.xhtml"
      media-type="application/xhtml+xml"
      properties="nav"/>
```

This manifests a common theme in EPUB: a file can “look right” but still be treated as irrelevant unless the OPF declares it with the right metadata.

## EPUB 3 additions: landmarks, page lists, and better semantics

EPUB 2 navigation mostly focuses on a hierarchical ToC. EPUB 3’s nav document expands the idea of navigation to include more “maps” of the book, notably landmarks and (optionally) print page mappings.

Landmarks are a way to identify functional locations like cover, start of body matter, bibliography, and so on. They can improve accessibility and can help reading systems present better UI shortcuts. They also give you a more explicit way to represent things that EPUB 2 often represented through the OPF `<guide>` section.

EPUB 3 also emphasizes semantic enrichment through attributes like `epub:type` in content documents. This becomes especially important once you care about accessibility, but it also matters for basic reading system features like recognizing footnotes, endnotes, or the main content region.

The important mindset is not “EPUB 3 has more tags,” but “EPUB 3 gives you a vocabulary for intent.” When reading systems recognize that intent, the user experience can be better. When they don’t, you still need the book to remain readable.

## Content documents: XHTML strictness vs HTML5-era semantics

Both EPUB 2 and EPUB 3 are strongly associated with XHTML content documents, and in practice you will see lots of EPUB 3 books whose chapters are still authored as XHTML. That’s because XHTML’s strictness (well-formed XML) makes validation and tooling more predictable.

What changes in EPUB 3 is the conceptual model: content documents are aligned with HTML5 semantics, and EPUB 3 introduces additional namespaces and attributes that don’t show up in EPUB 2.

For example, in EPUB 3 you often see the `epub` namespace declared so that `epub:type` is valid:

```xml
<html xmlns="http://www.w3.org/1999/xhtml"
      xmlns:epub="http://www.idpf.org/2007/ops"
      xml:lang="en">
  <head>
    <title>Chapter 1</title>
  </head>
  <body>
    <section epub:type="chapter">
      <h1>Chapter 1</h1>
      <p>...</p>
    </section>
  </body>
</html>
```

If you’re new to namespaces, don’t panic. You can read this as: “this is XHTML, and we’re enabling EPUB-specific attributes.” The `epub:type="chapter"` is semantic enrichment. A browser will mostly ignore it. A reading system might use it for navigation or accessibility, and validators will expect namespace correctness.

In EPUB 2 content documents, you generally won’t see EPUB semantics expressed the same way, and you’ll more often see older patterns shaped by the XHTML 1.1 era.

The crucial practical point is that EPUB readers are usually less forgiving than browsers. Whether you call it “EPUB 2 XHTML” or “EPUB 3 XHTML,” you should assume XML-style well-formedness matters unless you’re deliberately targeting a specific reading system with known tolerance.

## Media overlays: synchronized text + audio (powerful, but niche)

Media overlays are one of the headline “EPUB 3 is richer than EPUB 2” features. The idea is simple: you can ship audio narration and synchronize it to the text, enabling read-aloud experiences where the current phrase is highlighted while audio plays.

The synchronization is typically represented with SMIL files. A minimal SMIL snippet might look like this:

```xml
<smil xmlns="http://www.w3.org/ns/SMIL" version="3.0">
  <body>
    <seq>
      <par>
        <text src="text/chapter-01.xhtml#p1"/>
        <audio src="audio/ch01.mp3" clipBegin="0:00:00" clipEnd="0:00:05"/>
      </par>
    </seq>
  </body>
</smil>
```

You can read that as: “the text node `#p1` corresponds to the first five seconds of the audio file.” In a perfect world, a reading system uses this to drive highlighting and playback.

In the real world, media overlay support varies. It can work well in some environments and be ignored in others. That doesn’t mean you shouldn’t use it; it means you should treat it like an advanced feature with a careful testing burden.

## Scripting and interactivity: allowed by spec, often blocked by reality

EPUB 3 allows JavaScript and more interactive experiences in theory. In practice, many reading systems restrict or disable scripting for security, performance, and content-integrity reasons. Even when scripting is allowed, the runtime environment can be constrained and inconsistent.

This produces a recurring “spec vs reality” pattern: EPUB 3 gives you the vocabulary to build interactive books, but the ecosystem still heavily favors readable, mostly-static publications. If your project is about broad compatibility (most books are), you should avoid depending on JavaScript for essential reading, navigation, or accessibility.

## Backwards compatibility: why “EPUB 3 + NCX” is common

One of the most pragmatic practices in publishing is producing EPUB 3 publications that include EPUB 2-style fallbacks, especially NCX navigation.

This is not contradictory. Think of it as shipping multiple navigation maps so that more reading systems can build a ToC.

In that hybrid approach, the EPUB 3 nav document is present and marked with `properties="nav"`, and an NCX file is also present in the manifest:

```xml
<manifest>
  <item id="nav" href="nav.xhtml" media-type="application/xhtml+xml" properties="nav"/>
  <item id="ncx" href="toc.ncx" media-type="application/x-dtbncx+xml"/>
  <!-- ... content and assets ... -->
</manifest>
```

And you may also see EPUB 2-era wiring via a spine `toc="ncx"` attribute in some packages:

```xml
<spine toc="ncx">
  <!-- itemref entries -->
</spine>
```

The important thing is not the exact syntax; it’s the intent. The book is EPUB 3 in structure and capability, but it carries a compatibility navigation artifact so that older or conservative reading systems can still present a ToC.

## Which version should you target today?

If you are producing new EPUBs in 2026 and you care about modern reading systems, you should treat EPUB 3 as your default target. It is the version that matches today’s expectations for navigation, semantics, and accessibility, and it is the version most contemporary toolchains are built around.

However, “target EPUB 3” does not mean “use every EPUB 3 feature.” Most publishers and robust pipelines target EPUB 3 while intentionally sticking to a conservative subset of features that are widely supported: XHTML content documents, sensible CSS, a nav doc, and often an NCX fallback. They avoid scripting dependence and they treat advanced features (like media overlays) as special cases that require dedicated testing.

If you care about maximum device compatibility, especially with older e-ink devices or older software, including an NCX file is often a practical decision even in an EPUB 3 package. The cost is extra maintenance: if you change chapter filenames or fragment IDs, you must update both nav structures.

## Common pitfalls and gotchas

The biggest gotcha is assuming “EPUB 3 means web browser.” EPUB 3 aligns with web standards, but reading system support is uneven. CSS support varies, font support varies, and scripting support is often limited or disabled. If you design an EPUB like a web app, you can end up with a book that “works in one reader” and fails silently in another.

Navigation is the second big gotcha. In EPUB 3, a nav document that looks correct can still be ignored if it isn’t declared in the OPF with `properties="nav"`, or if the nav block is missing `epub:type="toc"`. In hybrid EPUBs, it’s also easy for nav and NCX to drift out of sync after edits, producing broken ToC links that only show up in some readers.

XHTML well-formedness is another high-impact trap. Many EPUB content documents are parsed as XML. A single unclosed tag can cause an entire chapter to disappear or render unpredictably, and the failure mode can differ across reading systems. If you’re coming from forgiving HTML-in-the-browser, this is one of the most surprising transitions.

Media overlays and scripting are classic “advanced feature” pitfalls. They can be valid per spec, but they are not uniformly implemented. If they are essential to the reading experience, you must treat reader support as a product requirement and test accordingly, not assume compatibility.

Finally, don’t rely on the version label alone. Real-world EPUBs are often hybrids. An EPUB 3 package may contain an NCX. An EPUB may claim a version but contain legacy structures. The reliable way to understand a book is to inspect the OPF, identify navigation resources, and then look at the actual content documents and their semantics. The version is a starting point for expectations, not a guarantee of behavior.
