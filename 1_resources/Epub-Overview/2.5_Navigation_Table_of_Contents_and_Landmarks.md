# 2.5. Navigation: Table of Contents and Landmarks

## Key vocabulary

**Navigation** (in EPUB) is the set of structures that tell a reading system how to present “where am I in the book?” and “what parts does the book have?” Navigation is not just a table of contents; it can also include landmarks and page mappings.

**Table of contents (ToC)** is the hierarchical list of chapters/sections presented to the reader. In EPUB, the ToC is built from either an EPUB 2 NCX file or an EPUB 3 navigation document (and sometimes both).

**NCX** is the EPUB 2 table-of-contents format (`.ncx`), an XML file that contains a `<navMap>` made of nested `<navPoint>` items.

**navMap** is the element inside an NCX file that contains the ToC tree structure.

**navPoint** is a single entry in an NCX ToC. navPoints can nest, which is how you represent parts, chapters, and sections.

**Navigation document** (nav document) is the EPUB 3 XHTML file that expresses navigation using normal HTML structure (typically lists) plus EPUB-specific semantics. It is usually named `nav.xhtml`, but the filename isn’t special; the OPF marks it as the nav doc.

**`epub:type`** is an EPUB 3 attribute used to add semantic meaning, especially in navigation documents (for example, a `<nav epub:type="toc">` for the ToC and `<nav epub:type="landmarks">` for landmarks).

**Landmarks** are navigation entries that identify “special places” in a publication: cover, table of contents, start of main content, bibliography, index, and so on. They help reading systems provide quick jumps and accessibility features.

**Page list** is a navigation structure that maps locations in the EPUB to print page numbers (useful in textbooks, academic books, and any content that references “page 123”).

**Fragment identifier** is the part after `#` in a URL-like reference, such as `chapter-01.xhtml#section-2`. It points to an element with that `id` inside the content document.

## What navigation is trying to solve

When you read a printed book, navigation is “built in.” You have a table of contents printed on paper, page numbers, and a physical sense of where you are. In a digital reflowable book, most of that disappears. Pages are not fixed. Font size can change. A “page” can mean wildly different amounts of content on a phone versus a tablet versus an e-ink device.

EPUB solves this by requiring the book to ship with machine-readable navigation structures that reading systems can use to build UI. The book isn’t responsible for drawing the ToC sidebar; the reading system is. But the book must provide the data that makes that UI accurate.

This is why navigation lives at the “package definition” layer rather than being an optional convenience. In Chapter 2.4 we saw that the OPF is the brain of the EPUB. Navigation is one of the OPF’s most important consequences: it is how the reading system gives the reader a map.

## How reading systems build the ToC

A reading system doesn’t usually infer a table of contents by scanning headings in your chapters. Some systems might offer that as a fallback, but it is not the reliable, spec-driven mechanism.

Instead, reading systems look for explicit navigation resources declared in the OPF.

In EPUB 2, the primary ToC mechanism is the **NCX** file. The OPF typically points to it through a manifest entry and, often, a spine `toc` attribute.

In EPUB 3, the primary ToC mechanism is the **navigation document** (an XHTML file) declared in the OPF manifest with `properties="nav"`.

Many real-world EPUB 3 files include both a nav document and an NCX for compatibility. When both exist, a reading system will usually prefer the nav document (because it’s the modern mechanism), but older EPUB 2-era systems might only understand NCX.

## EPUB 2 navigation: the NCX file

An NCX file is an XML document whose purpose is to represent a hierarchical table of contents. It does not contain the actual content of the book; it contains links to the content.

Conceptually, an NCX is a tree. At the root is `<navMap>`. Under it are nested `<navPoint>` elements, each representing a ToC entry. A navPoint has a label (what the user sees) and a target (where it links).

Here is a small but realistic NCX snippet that shows nesting:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<ncx xmlns="http://www.daisy.org/z3986/2005/ncx/" version="2005-1">
  <head>
    <meta name="dtb:uid" content="urn:uuid:2c6b8c1e-1b7a-4efb-9c7a-9b10a2f7d1c1"/>
  </head>

  <docTitle>
    <text>Example Book</text>
  </docTitle>

  <navMap>
    <navPoint id="navpoint-1" playOrder="1">
      <navLabel><text>Chapter 1</text></navLabel>
      <content src="text/chapter-01.xhtml"/>
    </navPoint>

    <navPoint id="navpoint-2" playOrder="2">
      <navLabel><text>Chapter 2</text></navLabel>
      <content src="text/chapter-02.xhtml"/>

      <navPoint id="navpoint-2-1" playOrder="3">
        <navLabel><text>Chapter 2, Section A</text></navLabel>
        <content src="text/chapter-02.xhtml#section-a"/>
      </navPoint>
    </navPoint>
  </navMap>
</ncx>
```

The most important part to understand is the `<content src="...">` attribute. That `src` is the target location, usually a relative path inside the EPUB, and optionally a fragment identifier. When you see `chapter-02.xhtml#section-a`, it means “go to the element with `id="section-a"` inside that file.”

The `playOrder` values are a historical detail that you’ll still see. They were meant to indicate reading order through the ToC. In practice, reading systems vary in how much they care. The spine is the real reading order; NCX ordering is the ToC ordering. Ideally those are consistent, but they are not the same concept.

## EPUB 3 navigation: the XHTML nav document

EPUB 3 replaced NCX as the primary navigation mechanism with something that looks much more like the web: an XHTML file containing `<nav>` elements. The nav document is still “data,” not “content,” but it is expressed using normal HTML structure.

The key pattern is a `<nav epub:type="toc">` element that contains an ordered list of links. Each list item is a ToC entry. Nesting is represented by nested lists.

Here is a small nav document ToC snippet that corresponds to the same structure as the NCX example above:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<html xmlns="http://www.w3.org/1999/xhtml"
      xmlns:epub="http://www.idpf.org/2007/ops"
      xml:lang="en">
  <head>
    <title>Navigation</title>
  </head>
  <body>
    <nav epub:type="toc" id="toc">
      <h1>Table of Contents</h1>
      <ol>
        <li><a href="text/chapter-01.xhtml">Chapter 1</a></li>
        <li>
          <a href="text/chapter-02.xhtml">Chapter 2</a>
          <ol>
            <li><a href="text/chapter-02.xhtml#section-a">Chapter 2, Section A</a></li>
          </ol>
        </li>
      </ol>
    </nav>
  </body>
</html>
```

This is one of the reasons EPUB 3 feels friendlier to web developers: the ToC is literally a list of links.

The important EPUB-specific detail is `epub:type="toc"`. This tells the reading system that this `<nav>` is not just “a navigation block,” but specifically the table of contents navigation. Without that semantic marker, some reading systems won’t treat it as the ToC, even if the links look right.

## Side-by-side: NCX vs nav doc (same “idea,” different representation)

At a conceptual level, NCX and the nav document are doing the same job: they define a hierarchy of labels that link into content.

The difference is the representation and the ecosystem expectations.

NCX is a specialized XML vocabulary designed for ToCs. It has concepts like `navMap`, `navPoint`, and `navLabel`. The nav document is ordinary XHTML with ordinary lists and links, plus EPUB semantics via `epub:type`.

That representation difference affects tooling. Web tools can parse and manipulate an XHTML nav document with “regular HTML” techniques. NCX often needs an XML-specific parser and a bit of format-specific knowledge.

It also affects how you debug. If a nav doc link is broken, you debug it like a broken web link. If an NCX is broken, you debug it like an XML data file: namespace correctness, element structure, and link paths.

## Landmarks: navigation that is not a ToC

A table of contents is a map of the book’s structure. Landmarks are a map of the book’s important *places*.

Think about what reading systems need for good user experience and accessibility. A user might want “go to cover.” Screen reader users might want “jump to start of main content.” A reading system might want to expose “table of contents” as a quick jump. These aren’t necessarily chapters. They’re functional landmarks.

In EPUB 3, landmarks are typically expressed as another `<nav>` block inside the nav document:

```xml
<nav epub:type="landmarks" id="landmarks">
  <h2>Landmarks</h2>
  <ol>
    <li><a epub:type="cover" href="text/cover.xhtml">Cover</a></li>
    <li><a epub:type="toc" href="nav.xhtml#toc">Table of Contents</a></li>
    <li><a epub:type="bodymatter" href="text/chapter-01.xhtml">Start of Content</a></li>
  </ol>
</nav>
```

Notice that landmarks use `epub:type` in two places: on the `<nav>` itself (`landmarks`) and on the individual links (`cover`, `bodymatter`, etc.). Those per-link types tell the reading system what role that location plays.

In EPUB 2, you’ll sometimes see similar intent expressed through the OPF `<guide>` section (Chapter 2.4). The guide is older and less flexible, but the purpose is related: provide shortcuts to key places.

## Page lists: mapping to print page numbers

Because EPUB content reflows, “page numbers” are not inherently stable. However, some books rely on referencing print pages, especially academic and educational content. This is where a **page list** comes in.

In EPUB 3, a page list can be represented as another nav block:

```xml
<nav epub:type="page-list" id="page-list">
  <h2>Pages</h2>
  <ol>
    <li><a href="text/chapter-01.xhtml#p1">1</a></li>
    <li><a href="text/chapter-01.xhtml#p2">2</a></li>
    <li><a href="text/chapter-02.xhtml#p23">23</a></li>
  </ol>
</nav>
```

This only works if the targets exist. That means your content documents need stable `id` attributes at the correct points, often around page break markers created during conversion from print layouts.

Many EPUBs do not include a page list because it’s extra work and not needed for novels. But when you need it, it’s the mechanism that makes “go to page 23” meaningful in a reflowable environment.

## Including both NCX and nav for compatibility

If you are shipping to a mixed ecosystem, it is common to include both:

- An EPUB 3 nav document (for modern reading systems)
- An EPUB 2 NCX file (for older systems and conservative pipelines)

When you do this, you need to wire both into the OPF correctly.

The nav document must be in the manifest with `properties="nav"`, for example:

```xml
<item id="nav" href="nav.xhtml" media-type="application/xhtml+xml" properties="nav"/>
```

The NCX file must also be in the manifest with the correct media type:

```xml
<item id="ncx" href="toc.ncx" media-type="application/x-dtbncx+xml"/>
```

And, in EPUB 2 style packaging, the spine may reference it via `toc="ncx"`:

```xml
<spine toc="ncx">
  <!-- itemref entries -->
</spine>
```

The important mindset is that “including both” is not just dropping extra files into the ZIP. Both navigation resources are only useful if they are declared in the package and internally consistent with your content documents.

## Common pitfalls and gotchas

The most common navigation pitfall is broken links caused by renames and splits. If you split `chapter-01.xhtml` into `chapter-01.xhtml` and `chapter-01b.xhtml` but you don’t update the nav doc and/or NCX, the reading system’s ToC will point into missing files or outdated fragment identifiers. This can look like “the ToC does nothing,” but the root cause is just a bad link.

Fragment identifiers are a special category of this problem. A link like `chapter-02.xhtml#section-a` only works if the target element exists and has an exact matching `id="section-a"`. If you change headings, regenerate HTML, or run a cleanup step that modifies IDs, you can silently break navigation. Some reading systems fail gracefully; others dump the user at the start of the chapter; others do nothing.

Another common pitfall is forgetting that the nav document has to be declared as the nav document. In EPUB 3, a nav-looking XHTML file is not enough. The OPF manifest item must include `properties="nav"`, and the document should include the correct `epub:type` semantics (like `toc`, `landmarks`, and `page-list`). Without those, reading systems may ignore the navigation entirely.

For NCX, namespace and structure errors are common. NCX is XML with a specific namespace. If you produce NCX by hand or with a buggy converter, you can end up with subtle structural invalidity that some reading systems ignore and others reject. When NCX is present for compatibility, it needs to be correct, not merely present.

Landmarks are also a place where “spec vs reality” shows up. Many reading systems provide some landmarks UI, but support varies. Some systems only recognize a few landmark types. Others ignore landmarks completely. The best practice is to include landmarks because they help accessibility and advanced readers, but don’t depend on them as the only way to reach important content.

Finally, don’t confuse the spine with the ToC. The spine is the reading order. The ToC is the navigation UI map. You want them to agree at a human level, but they can legitimately differ (for example, when front matter is in the spine but omitted from the ToC). When debugging, check both: “is the chapter in the spine?” and “does the ToC link point to the right place?” They answer different questions, and mixing them up is one of the fastest ways to chase the wrong bug.
