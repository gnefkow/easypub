# 3.3 Links and Cross-References in EPUBs

Links are one of the quiet superpowers of EPUB. At a basic level, links let a reader tap a table-of-contents entry and jump to a chapter. At a more “bookish” level, links power footnotes, endnotes, citations, “see also” cross-references, indexes, and internal navigation between sections of the book. If you already know HTML, the syntax will feel familiar: EPUB uses the same `<a>` element and the same idea of URLs and fragments. What changes in EPUB is what those URLs point to and how a reading system resolves them inside a ZIP file.

This chapter will help you build a reliable mental model for links in EPUBs: how internal links work between content documents, what a fragment identifier actually targets, how external links behave across reading systems, and the standard patterns for footnotes and endnotes in EPUB 3.

## Key vocabulary (for this chapter)

In EPUB linking, a **relative path** is a path like `../text/chapter-02.xhtml` that is interpreted relative to the file that contains the link. A **fragment identifier** is the part after `#` in a link, like `#note-12`, and it targets a specific element in the destination document. A **cross-reference** is any link that jumps to another part of the same book, often used for “see Chapter 5” or “as shown in Figure 2.” A **noteref** is a note reference link—usually a superscript number in the main text that points to a footnote or endnote. A **footnote** is a note that is conceptually attached to the page or location where it’s referenced; an **endnote** is a note collected elsewhere (often at the end of a chapter or the end of the book).

## The `<a>` element in EPUB content documents

The anchor element in EPUB XHTML is the same one you know from the web. A minimal link looks like this:

```xml
<a href="chapter-02.xhtml">Go to Chapter 2</a>
```

The visible, tappable text is the content inside the `<a>...</a>` element. The `href` attribute contains the destination. Because EPUB content documents are typically XHTML (well-formed XML), you still need to follow the XHTML rules you learned in the previous chapter: attribute values must be quoted, and the markup must be correctly nested and closed.

The biggest practical difference from the web is that your destination is usually another file inside the EPUB package, not a page served from a web server. In other words, your “site” is a folder tree inside a ZIP file.

## Internal links: linking within the book

Internal links are the most common kind in EPUBs. They connect one content document to another, or one location inside a content document to another location inside the same document.

To make this concrete, imagine an EPUB that looks like this when unzipped:

```text
OEBPS/
  text/
    chapter-01.xhtml
    chapter-02.xhtml
    notes.xhtml
  styles/
    book.css
```

If `chapter-01.xhtml` wants to link to `chapter-02.xhtml`, the `href` must be a relative path from the location of chapter 1 to the location of chapter 2. Since they are in the same folder (`text/`), the simplest link is:

```xml
<a href="chapter-02.xhtml">Next chapter</a>
```

If chapter 1 wanted to link to a notes page that is also in `text/`, it would be similar:

```xml
<a href="notes.xhtml">Notes</a>
```

If your EPUB uses different folders (for example, `Text/` instead of `text/`, or `chapters/`), the same principle applies. EPUB linking is boring in a good way: it is mostly just file paths.

### Linking within a chapter using fragment identifiers

The more interesting case is when you want to link to a specific spot inside a file. That’s where fragment identifiers come in. A fragment identifier targets an element with a matching `id` attribute.

Suppose `chapter-02.xhtml` contains a section heading:

```xml
<h2 id="the-turning-point">The Turning Point</h2>
```

Now you can link directly to that heading from another place:

```xml
<a href="chapter-02.xhtml#the-turning-point">Jump to “The Turning Point”</a>
```

Here’s how to read that `href`:

The part before the `#` (`chapter-02.xhtml`) is the file to open. The part after the `#` (`the-turning-point`) is the `id` of the element to scroll to.

You can also use fragments within the same file by omitting the filename:

```xml
<a href="#the-turning-point">Jump down</a>
```

That says “stay in this file, but scroll to the element with `id="the-turning-point"`.”

### What reading systems actually do with internal links

When a reading system renders an EPUB, it usually builds an internal map of the package: it knows where the OPF is, it knows which files are content documents, and it can load any referenced file from the ZIP as needed. When you tap an internal link, the reader resolves the `href` relative to the current content document, loads the destination document from the package, and then scrolls to the fragment target (if there is one).

This is why correct relative paths matter so much. There is no server-side redirect, no helpful 404 page, and no browser URL bar to hint what went wrong. If the path is wrong, the link just fails or does nothing, depending on the reader.

## Cross-references: “see Chapter 7” links that survive edits

A cross-reference is just an internal link with a human meaning: it points the reader somewhere else for context. For example, a sentence might say:

```xml
<p>
  For the packaging rules, see
  <a href="chapter-10.xhtml#packaging-rules">Chapter 10</a>.
</p>
```

One of the most important habits in EPUB editing is choosing fragment targets that remain stable over time. If you use `id="packaging-rules"` on the heading, the link can survive paragraph edits, line breaks, and text formatting changes. If you link to “the third paragraph” without an `id`, you have nothing stable to aim at.

You can think of `id` attributes as your internal API. They are the names other parts of the book can call. When you rename or remove an `id`, you are making a breaking change.

## External links: linking to the web

External links use the same `href` syntax as the web:

```xml
<a href="https://www.w3.org/publishing/epub3/">EPUB 3 on the W3C site</a>
```

What varies is how readers handle them. Some reading systems open external links in an in-app browser. Some open the system browser. Some show a warning. Some disable external navigation entirely (especially in locked-down or DRM-heavy environments). On e-ink devices, external browsing may be limited or unpleasant.

For that reason, external links in EPUBs should be treated as optional enhancements. They can be valuable for references and citations, but the book should still make sense if the reader never opens them.

If you are editing EPUBs for accessibility and clarity, it’s usually better to write link text that describes the destination instead of displaying raw URLs, because raw URLs are difficult to read aloud and harder to scan:

```xml
<a href="https://example.com/research-paper">Research paper on reflowable layout</a>
```

## Footnotes and endnotes in EPUB 3

Footnotes and endnotes are where EPUB linking becomes more standardized. EPUB 3 introduced conventions using the `epub:type` attribute to mark up both the link in the main text and the note itself. The goal is not just “make the link work,” but “help the reading system understand that this link is a note.”

To use `epub:type`, the XHTML document typically declares the `epub` prefix on the root `<html>` element, like this:

```xml
<html xmlns="http://www.w3.org/1999/xhtml"
      xmlns:epub="http://www.idpf.org/2007/ops"
      xml:lang="en"
      lang="en">
```

Now let’s look at a common footnote pattern. In the main text, you place a note reference (often a superscript number). The link uses `epub:type="noteref"`:

```xml
<p>
  EPUB is a container format.<a href="notes.xhtml#note-1" epub:type="noteref" id="noteref-1">1</a>
</p>
```

This snippet is doing several things at once. The `href` points to another file (`notes.xhtml`) and to a fragment inside it (`#note-1`). The `epub:type="noteref"` tells compatible reading systems “this is a note reference.” The `id="noteref-1"` gives you a stable target so the note can link back to the exact place the reader came from.

In the notes file, you create the note itself. EPUB 3 commonly uses `epub:type="footnote"` on an `<aside>` element:

```xml
<aside id="note-1" epub:type="footnote">
  <p>
    <a href="chapter-01.xhtml#noteref-1">↩</a>
    A “container format” is a file format that wraps other files, like a ZIP.
  </p>
</aside>
```

This note has `id="note-1"` so the fragment `#note-1` can target it. The `epub:type="footnote"` tells the reader “this thing is a footnote,” which can enable special behaviors like pop-up notes. The return link (`href="chapter-01.xhtml#noteref-1"`) is not strictly required, but it is a very good practice because it helps the reader get back to the right place.

### Endnotes: same mechanics, different organization

Endnotes use the same link mechanics as footnotes. The difference is editorial: endnotes are typically collected at the end of a chapter or the end of the book rather than near each reference.

You might keep endnotes in `endnotes.xhtml` and link to them:

```xml
<a href="endnotes.xhtml#note-12" epub:type="noteref" id="noteref-12">12</a>
```

And then define the note similarly:

```xml
<aside id="note-12" epub:type="footnote">
  <p>
    <a href="chapter-05.xhtml#noteref-12">↩</a>
    This endnote is stored separately for a cleaner chapter layout.
  </p>
</aside>
```

Even though the endnote is not “at the foot” of the page, many EPUB workflows still mark the note itself as `epub:type="footnote"` because the attribute describes the semantic role (a note) rather than literal physical placement. The important idea is consistency: pick a pattern, use stable `id`s, and make sure every noteref has a matching note target.

### Where notes live in the EPUB (files, manifest, and spine)

From a file-structure perspective, footnotes and endnotes are not a special file type. They are normal XHTML content documents and normal anchors. You can keep the note content in the same chapter file, or you can move it into a separate “notes” document and link to it. Both approaches are valid EPUB.

If you are just getting started, a simple and common folder layout is to keep chapters and notes together under a `text/` folder and keep other assets in sibling folders:

```text
OEBPS/
  content.opf
  text/
    chapter-01.xhtml
    chapter-02.xhtml
    notes.xhtml
  styles/
    book.css
  images/
    figure-1.png
```

That layout keeps relative links boring. A chapter can link to a note with `href="notes.xhtml#note-1"` because the chapter and the notes file live in the same folder.

The part that is easy to forget is that any notes file you create is still part of the EPUB package, which means it must be declared in the OPF manifest. A minimal manifest entry looks like this:

```xml
<item id="notes" href="text/notes.xhtml" media-type="application/xhtml+xml"/>
```

Whether the notes file should also be listed in the OPF spine depends on the reading experience you want.

If you include the notes file in the spine, the reading system can reach it via “next/previous” navigation, but you might not want it to appear as part of the primary reading flow. In that case, a common pattern is to include it with `linear="no"`:

```xml
<itemref idref="notes" linear="no"/>
```

If you do not include the notes file in the spine, the book can still work perfectly well because the noteref links will navigate to it. The trade-off is that some reading systems treat spine items as “first-class” reading locations, and non-spine documents may be harder to reach except through links.

As the notes collection grows, it can also help to add a semantic wrapper around the notes section itself. For example, a notes document might contain a container marked as rear notes, and then individual notes marked as footnotes:

```xml
<section epub:type="rearnotes">
  <h1>Notes</h1>
  <aside id="note-1" epub:type="footnote">
    <p><a href="chapter-01.xhtml#noteref-1">↩</a> …</p>
  </aside>
</section>
```

Finally, think about ID collisions early. If you keep all notes in a single global `notes.xhtml`, it is easy to accidentally reuse `id="note-1"` in multiple chapters. A boring, scalable convention is to namespace IDs by chapter, like `ch01-noteref-3` linking to `notes.xhtml#ch01-note-3`.

## Common pitfalls and gotchas

The first and most common problem is broken relative paths. EPUB links are resolved relative to the current XHTML file, not relative to the OPF, and not relative to the ZIP root in a “web server” sense. If you move files between folders or change capitalization (`Text/` vs `text/`), you can silently break links in ways that only show up on certain devices.

The second gotcha is forgetting that fragment identifiers require unique `id` values. If you accidentally reuse `id="note-1"` twice in the same document, different reading systems may jump to different places. If you forget to add the `id` at all, the link will load the file but not scroll to the intended location, which can feel “random” to the reader.

The third gotcha is assuming all readers treat notes the same way. Some reading systems recognize `epub:type="noteref"` and `epub:type="footnote"` and show a pop-up. Others will simply navigate like a normal link. Your markup should work in both cases, which is why return links and clear note formatting matter.

The fourth gotcha is styling noteref links in a way that harms usability. It’s common to make note references superscript and smaller, but if you make them too small or remove underline and color cues entirely, they become hard to tap on a phone. EPUB design has to consider touch targets and small screens more than print does.

The fifth gotcha is external links in locked-down environments. A link to `https://...` may be blocked, may prompt the user, or may open a browser that loses the reader’s place. External links are fine, but the book should not depend on them for core meaning.

Finally, remember that EPUB content documents are typically XHTML. That means your `<a>` tags and attributes must stay well-formed. A missing quote in `href`, an unescaped `&` inside a URL query string, or invalid nesting can turn “a link issue” into “the document won’t parse,” which is much harder to debug.
