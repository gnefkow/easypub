# 4.1 Accessibility in EPUBs

Accessibility is often described as “making content usable for people with disabilities,” but that description can sound abstract until you connect it to what an EPUB actually is. An EPUB is a structured set of documents that a reading system turns into a reading experience: text that can be resized, colors that can be inverted, content that can be spoken aloud, navigation that can be surfaced as a table of contents, and semantics that can help a reader understand what they are looking at.

When you make an EPUB accessible, you are not bolting on a special “screen reader mode.” You are making the underlying content more truthful about what it is. A heading is really a heading, not just a bold paragraph. A list is really a list, not a set of lines with hyphens. An image either has an equivalent text story, or it is explicitly decorative. The payoff is bigger than any single assistive technology: accessibility improvements also help reflow, search, note-taking, highlighting, reading on small screens, and long-term maintainability.

Accessibility also matters for reasons that have nothing to do with technology. Depending on where you publish and who your audience is, you may have legal requirements (for example, Section 508 in the US federal context, the ADA in some US publishing scenarios, and the European Accessibility Act in the EU market). Even when you are not legally required, accessible EPUBs are simply better products. They reach more people, they avoid exclusion, and they tend to age better as reading systems improve.

## Key vocabulary (for this chapter)

**Accessibility** is the practice of making content usable by people with a wide range of abilities and assistive technologies. **WCAG** (Web Content Accessibility Guidelines) is the main set of principles and success criteria used to evaluate web-like content, and EPUB content documents inherit much of the same reality because they are built from HTML/CSS. **Semantic markup** means using elements and attributes that express meaning (like headings, lists, tables, and landmarks) rather than only visual styling. **ARIA** (Accessible Rich Internet Applications) is a set of roles and attributes that can add semantics and improve assistive technology behavior when native HTML alone is not enough; **DPUB-ARIA** is a publishing-focused set of ARIA roles that can be useful in documents. The `epub:type` attribute is an EPUB-specific way to attach publishing semantics (like “chapter” or “footnote”) to elements. **Accessibility metadata** is information in the OPF package document that describes the accessibility characteristics of the publication (for example, whether it has alt text, whether it contains hazards like flashing content, and an accessibility summary). **Ace by DAISY** (often just “Ace”) is a tool that checks EPUBs for accessibility issues and produces a report.

## Why accessibility in EPUB feels different from accessibility on the web

If you come from the web, you might assume accessibility is primarily about the browser. In EPUB, the “browser” is a reading system, and it can vary widely. Some reading systems are based on modern web engines. Others are built on older engines. Some expose accessibility features deeply (screen reader support, landmarks navigation, read-aloud). Others do the minimum.

This variability creates an important mindset shift: you want your EPUB to be accessible because the content itself is well-structured and meaningful, not because a particular reading system happens to compensate for mistakes. That means your primary tool is semantic HTML, supported by EPUB semantics (`epub:type`) and careful metadata. ARIA can help, but it is not the foundation.

## EPUB 3 as an accessibility-forward format: semantics are the foundation

EPUB 3 was designed with modern HTML in mind, and that gives you a huge advantage compared to older book formats. A well-authored EPUB can be navigated like a well-authored website, but with a book-oriented reading model layered on top (spine order, table of contents, landmarks, and pagination).

In practice, “accessible EPUB” usually starts with a small set of habits:

Use real headings (`<h1>`–`<h6>`) in a meaningful hierarchy. Use paragraphs (`<p>`) for paragraphs. Use lists (`<ol>`, `<ul>`, `<li>`) for lists. Use emphasis (`<em>`, `<strong>`) for emphasis. Use tables only when the content is truly tabular, and use the table elements that communicate relationships (`<th>`, `scope`, `caption`).

Those choices are not primarily about screen readers. They are about preserving meaning when the visual presentation changes. EPUB reading systems reflow content, users change font size, and styles can be overridden. Semantic markup keeps your book intelligible under all of those changes.

## Semantic sections and landmarks inside content documents

EPUB content documents are usually XHTML files, and you can use sectioning elements to give the document a clear structure. A typical chapter might use a `<section>` or `<article>` that contains a heading and the chapter content. In EPUB 3, you can add `epub:type` to express publishing semantics alongside HTML semantics:

```xml
<body epub:type="bodymatter">
  <section epub:type="chapter" id="ch04">
    <h1>Accessibility in EPUBs</h1>
    <p>Accessibility is often described as…</p>
  </section>
</body>
```

There are a few things to notice here.

First, the HTML elements (`<section>`, `<h1>`, `<p>`) already communicate a lot. Assistive technologies know what a heading is and how to navigate between headings. Second, `epub:type` adds book-oriented semantics that some reading systems can use for navigation, UI labels, or specialized behavior. Third, the semantics are attached to the content itself, so even if a stylesheet changes the look, the structure remains accurate.

You will sometimes see ARIA roles used for landmarks as well. In EPUB, you should treat ARIA as an enhancement and apply it carefully. Native semantics are usually more reliable than ARIA, and reading systems vary in how they expose ARIA to assistive technology. When you do use ARIA, the goal is to clarify the role of a region, not to replace good HTML.

## Reading order: the spine is necessary, but not sufficient

The OPF spine defines the publication’s high-level reading order across content documents. But accessibility also depends on reading order within each document. Screen readers and other assistive technologies generally follow the DOM order of the XHTML. If you create layouts that visually reorder content (for example, positioning a sidebar before the main text or floating content that appears “earlier” visually), you can accidentally create a confusing reading experience.

This is one reason EPUB styling advice tends to be conservative. In a reflowable book, the safest approach is to keep the DOM order aligned with the reading order and use CSS mainly for typography and spacing rather than complex two-column layouts that rely on positioning tricks.

## Language declarations: tell reading systems how to pronounce your content

Language is a core accessibility signal. Screen readers use it to choose a voice and pronunciation rules. Hyphenation and line-breaking can also depend on it. In EPUB, you generally declare language in two places:

In the OPF metadata, you set the publication language with `dc:language` (for example, `en` or `en-US`). In each XHTML content document, you declare language on the root `<html>` element using both `xml:lang` and `lang` for broad compatibility:

```xml
<html xmlns="http://www.w3.org/1999/xhtml"
      xmlns:epub="http://www.idpf.org/2007/ops"
      xml:lang="en"
      lang="en">
```

This is not just “metadata hygiene.” If you have passages in other languages, you should mark them too. A single French sentence in an English chapter is a common example:

```xml
<p>He shrugged and said, <span xml:lang="fr" lang="fr">c’est la vie</span>.</p>
```

Without this, assistive technology may pronounce the phrase incorrectly. With it, the screen reader has a chance to switch pronunciation appropriately.

## Navigation, table of contents, and landmarks

Chapter 2.5 covered the navigation document from a structural perspective, but accessibility raises the stakes. A good nav document is not just a convenience feature; it is part of how non-visual readers orient themselves in the book.

In EPUB 3, the nav document is an XHTML file that contains `<nav>` elements with different `epub:type` values. The table of contents is expressed as a structured list, which is inherently accessible when done correctly:

```xml
<nav epub:type="toc" id="toc">
  <h2>Contents</h2>
  <ol>
    <li><a href="text/chapter-01.xhtml">Chapter 1: …</a></li>
    <li><a href="text/chapter-02.xhtml">Chapter 2: …</a></li>
  </ol>
</nav>
```

This is valuable because it gives reading systems a reliable, machine-readable hierarchy that can be exposed as navigation UI, keyboard navigation, or assistive navigation.

Landmarks are also important. They identify major regions of the book such as the cover, table of contents, and main body. A landmarks nav looks similar, but uses landmark semantics:

```xml
<nav epub:type="landmarks" id="landmarks">
  <h2>Landmarks</h2>
  <ol>
    <li><a epub:type="cover" href="text/cover.xhtml">Cover</a></li>
    <li><a epub:type="toc" href="nav.xhtml#toc">Table of contents</a></li>
    <li><a epub:type="bodymatter" href="text/chapter-01.xhtml">Start of content</a></li>
  </ol>
</nav>
```

The details of which landmarks you include can vary, but the core idea is that you are labeling the book in a way that lets a reader jump to the parts they care about. This is especially helpful in books with front matter, long introductions, or appendices.

## Images: alt text, figures, and long descriptions

Images are one of the most visible accessibility requirements because they demand editorial judgment. The goal is not “put some text in `alt`.” The goal is “make sure a reader who can’t see the image does not lose meaning.”

For many images, good alt text is short and specific:

```xml
<img src="../images/zip-diagram.png"
     alt="A diagram showing an .epub file as a ZIP container with a mimetype file, META-INF, and OEBPS content." />
```

This works when the image’s meaning can be expressed in a sentence. If the image is purely decorative, the best practice is an empty alt attribute so assistive tech can skip it:

```xml
<img src="../images/ornament.png" alt="" />
```

If the image is complex (a chart, a detailed diagram, a map), a single sentence may be insufficient. In EPUB, a common pattern is to provide a visible caption and a separate longer description that can be referenced. One practical approach is to use a `<figure>` with a `<figcaption>`, and connect the image to a longer description with `aria-describedby`. For example:

```xml
<figure class="figure" id="fig-accessibility-metadata">
  <img src="../images/metadata-graph.svg"
       alt="A graph comparing accessibility metadata fields."
       aria-describedby="desc-accessibility-metadata" />
  <figcaption>Figure 4.1: Accessibility metadata fields used in EPUB.</figcaption>
</figure>

<section id="desc-accessibility-metadata" epub:type="rearnotes">
  <h2>Figure 4.1 description</h2>
  <p>The diagram shows four schema.org properties—accessMode, accessibilityFeature, accessibilityHazard, and accessibilitySummary—grouped into “what the book is,” “what it contains,” “what to watch out for,” and “human summary.”</p>
</section>
```

The important idea is not the exact mechanics of `aria-describedby` (support varies), but the editorial structure: short alt text plus an available longer description. If a reading system does not use the ARIA link, the description still exists in the book and can be reached by normal navigation.

Also remember the biggest red flag: images that contain important text. If the text matters, prefer real text in XHTML so it can reflow, be searched, be read aloud, and be resized independently.

## Links, cross-references, and footnotes

Accessible linking in a book has the same basic rule as on the web: the link text should make sense when read out of context. “Click here” is not meaningful in a book, and it is especially unhelpful when a screen reader user is navigating links as a list.

For example, this link is vague:

```xml
<p>For details, <a href="text/chapter-07.xhtml">see this section</a>.</p>
```

This link is clearer:

```xml
<p>For details, see <a href="text/chapter-07.xhtml">Chapter 7, “Validation and Debugging”</a>.</p>
```

Footnotes and endnotes are also a common accessibility hot spot. EPUB 3 defines semantics for note references and the notes themselves. A typical pattern uses `epub:type="noteref"` on the link and `epub:type="footnote"` (or another note type) on the note content:

```xml
<p>
  EPUB is a ZIP file<a href="notes.xhtml#n1" epub:type="noteref">1</a>.
</p>
```

In the notes document:

```xml
<section id="n1" epub:type="footnote">
  <p><a href="chapter-01.xhtml#ref-n1">1</a> ZIP is the container format; `.epub` is a convention over it.</p>
</section>
```

This is a good example of “spec meets reality.” Some reading systems provide special note popovers or “return to text” behavior when these semantics are present. Others treat them as normal links. Your job is to make sure both experiences work: the note should be reachable, readable, and easy to return from.

## Tables: use real table semantics (or don’t use a table)

Tables are not inherently inaccessible, but they require accurate markup. If your content is truly tabular, use `<table>`, include a `<caption>` when it helps, and use `<th>` with `scope` to express header relationships:

```xml
<table>
  <caption>Accessibility tools for EPUB</caption>
  <thead>
    <tr>
      <th scope="col">Tool</th>
      <th scope="col">What it checks</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <th scope="row">EPUBCheck</th>
      <td>Package validity and spec conformance</td>
    </tr>
    <tr>
      <th scope="row">Ace</th>
      <td>Accessibility signals and WCAG-aligned issues</td>
    </tr>
  </tbody>
</table>
```

This kind of markup helps a screen reader announce headers correctly when a user navigates cell by cell. If you are using a table only for layout (for example, to align a block quote and an image), that is a sign you should not use a table at all. In reflowable EPUBs, layout tables often degrade poorly on small screens and can confuse assistive technologies.

## Audio, video, and interactive content

EPUB 3 supports multimedia, but accessibility requires equivalents. If you include video, you should expect to provide captions. If you include audio, you should expect to provide a transcript. If your content relies on interaction or scripting, you should expect to provide a usable fallback, because many reading systems restrict or disable JavaScript for security and consistency reasons.

This is where the EPUB mindset becomes very practical: assume progressive enhancement. If the interactive version works, great. If it does not, the book should still be readable and understandable.

## Accessibility metadata in the OPF package document

Accessible content is the most important thing, but metadata plays a supporting role. It helps libraries, distributors, and readers discover accessible books and understand what accessibility features are present. In EPUB, accessibility metadata is commonly expressed using schema.org properties in the OPF metadata, along with an accessibility summary written for humans.

In EPUB 3, the `prefix` attribute on the `<package>` element is often used to declare prefixes like `schema:` and `a11y:`. A simplified example looks like this:

```xml
<package xmlns="http://www.idpf.org/2007/opf"
         version="3.0"
         unique-identifier="pub-id"
         prefix="schema: http://schema.org/ a11y: http://www.idpf.org/epub/vocab/package/a11y/#">
  <metadata xmlns:dc="http://purl.org/dc/elements/1.1/">
    <dc:identifier id="pub-id">urn:uuid:...</dc:identifier>
    <dc:title>Example Book</dc:title>
    <dc:language>en</dc:language>

    <meta property="schema:accessMode">textual</meta>
    <meta property="schema:accessibilityFeature">alternativeText</meta>
    <meta property="schema:accessibilityFeature">structuralNavigation</meta>
    <meta property="schema:accessibilityHazard">none</meta>
    <meta property="schema:accessibilitySummary">
      This publication includes a structured table of contents, heading hierarchy, and alternative text for meaningful images. Decorative images are marked as decorative. No flashing content is present.
    </meta>
  </metadata>
</package>
```

Let’s unpack what this is doing.

The `schema:accessMode` value describes the primary sensory mode needed to consume the content (for many books, “textual” is the core). `schema:accessibilityFeature` can list features like alt text and navigation structure. `schema:accessibilityHazard` describes hazards such as flashing content, sound hazards, or motion simulation; “none” is meaningful if it’s true. `schema:accessibilitySummary` is a human-readable statement that sets expectations. It is not a marketing blurb. It should be honest.

The exact set of properties you include depends on your book. The point is that metadata should reflect reality. If the book has diagrams without alt text, you should not claim it has “alternativeText.” If the book contains content that could trigger seizures, it should not claim “none” for hazards.

## Checking your work: a practical validation workflow

Accessibility work benefits from tools because humans are bad at spotting systematic issues in large documents. Two tools are commonly used together.

EPUBCheck validates EPUB packaging and conformance issues. It will catch problems like missing manifest items, invalid media types, broken XHTML well-formedness, and other issues that can make a book unstable across readers. While EPUBCheck is not an accessibility checker, it is a prerequisite: a book that fails to parse reliably cannot be reliably accessible.

Ace by DAISY focuses on accessibility signals and common problems. It can identify missing alt attributes, heading structure issues, language declaration problems, and other patterns. It will also produce a report that you can use as a checklist. Like any automated tool, Ace cannot judge whether your alt text is meaningful, but it can tell you where alt text is missing, duplicated, or suspicious.

The best workflow is iterative. Validate the EPUB so you know the container is correct, then run accessibility checks, then do human review on the areas where automation cannot decide (alt text quality, reading order coherence, clarity of links, correctness of tables, and usefulness of navigation).

## Common pitfalls and gotchas

The first gotcha is confusing “looks correct” with “is structured correctly.” A chapter title that is a centered `<p>` with large font styling may look fine, but it is not a heading. Screen reader users often navigate by headings; if you remove headings, you remove one of the most powerful navigation tools in an EPUB.

The second gotcha is broken heading hierarchy. Jumping from `<h1>` to `<h4>` because you like the visual size is a common web mistake, and it carries over into EPUB. Use CSS to style headings however you want, but keep the semantic levels meaningful. A clean hierarchy makes the book navigable and makes automated checks calmer.

The third gotcha is missing or incorrect language declarations. If you forget `xml:lang`/`lang` in XHTML, or if you never set `dc:language`, you are leaving pronunciation and hyphenation up to guesswork. If you have multilingual content, mark the language shifts explicitly.

The fourth gotcha is treating alt text as an afterthought. Alt text that repeats the caption, alt text that says “image,” or alt text that dumps a paragraph of irrelevant detail all create poor experiences. Aim for context-specific meaning. If the image is decorative, use empty alt text. If the image is complex, provide a longer description in the content, not just in the `alt` attribute.

The fifth gotcha is relying on color alone to communicate meaning. A sentence like “Items in red are required” may make sense visually, but it fails for many readers (color blindness, night mode, custom themes, and non-visual use). If something is required, say it in text.

The sixth gotcha is using layout hacks that reorder the reading experience. CSS that moves content around can create a mismatch between what a sighted reader sees and what a screen reader reads. In reflowable EPUBs, keep the DOM order aligned with intended reading order, and be cautious with complex multi-column layouts.

The seventh gotcha is inaccessible fixed-layout content. Fixed-layout EPUBs can be beautiful, but they tend to be harder to make accessible because they often encode meaning in position and images. If you use fixed layout, you must be even more deliberate about text alternatives, reading order, and navigation. Many accessibility expectations are easier to meet in reflowable content.

The eighth gotcha is overstating accessibility metadata. It is tempting to declare every accessibility feature you hope you have, but accessibility metadata should be a truthful summary of what is actually present. Incorrect claims can create legal and trust issues, and they can mislead readers who depend on accessible features.

Finally, remember that EPUB accessibility lives in the gap between the spec and real reading systems. Some semantics are honored widely; others are inconsistently exposed. The most reliable strategy is boring but effective: write clean XHTML, use semantic HTML, declare language, keep navigation solid, provide meaningful text alternatives, and validate with both EPUBCheck and an accessibility-focused tool like Ace. When you do that, your EPUB becomes more robust for everyone, not only for a particular assistive technology.
