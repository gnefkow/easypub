# 2.3. The Overall File Structure of an EPUB

## Key vocabulary

**Reading system** is the app or device that opens and renders an EPUB (Apple Books, Kobo, Thorium Reader, Google Play Books, etc.). “Reader” is often used casually, but the spec term is “reading system.”

**OCF (Open Container Format)** is the outer “container” layer of an EPUB: the fact that an EPUB is a ZIP archive with a required `mimetype` file and a `META-INF/` directory that tells the reading system where the publication’s package document lives.

**Container** (in EPUB context) means the ZIP-based wrapper that holds everything. It is not just “a folder.” It’s a packaging format with rules that let software reliably locate the book’s metadata and content.

**Package document** is the file (usually an `.opf` file) that describes what the EPUB contains. It is the “inventory + reading order + metadata” for the publication.

**OPF** stands for Open Packaging Format. In practice, people say “the OPF” to mean “the `.opf` package document.”

**Manifest** is the section inside the OPF that lists every resource in the EPUB (XHTML, CSS, images, fonts, nav document, NCX, etc.) with a path (`href`) and a media type.

**Spine** is the section inside the OPF that defines the primary reading order. If the manifest is “everything that exists,” the spine is “what you read, in what order.”

**Content document** is an XHTML (EPUB 2) or HTML/XHTML (EPUB 3) file that contains the actual book content (chapters, sections, title page, etc.). These are the files readers paginate and display.

**Navigation document** (often called “the nav doc”) is the EPUB 3 XHTML file that provides the table of contents (and sometimes landmarks and page lists) in a machine-readable way.

**NCX** is the EPUB 2 navigation file format (an XML file, typically `toc.ncx`) that provides the table of contents structure for older reading systems.

**Rendition** is the way a publication is presented (for example, reflowable vs fixed layout). You’ll see rendition-related metadata and properties that influence layout behavior.

## The high-level mental model: three layers

The easiest way to stay sane with EPUBs is to stop thinking of them as “a book file” and start thinking of them as three nested layers, each with a different job.

The outermost layer is the **container layer**. This is the part that makes an EPUB “an EPUB” instead of “a random ZIP of HTML.” The container layer includes the required `mimetype` file and the `META-INF/` directory, and it answers one critical question for the reading system: *where is the package document that describes this publication?*

Inside the container, you have the **package layer**. This layer is centered around the **OPF package document**. The OPF is the “brain” of the EPUB. It contains metadata (title, author, language, identifiers), a manifest (every file in the publication), and a spine (the reading order). If you only learn one file deeply, learn the OPF.

Finally, you have the **content layer**. This is the material you think of as “the book”: content documents (chapters), stylesheets, images, fonts, and any other assets. This layer is web-like (XHTML/HTML + CSS + media), but it exists inside the constraints and linking rules imposed by the container and package layers.

If you like diagrams, here’s the “nesting doll” view:

```text
------------------------------+
  EPUB (.epub) = ZIP archive  |   <-- Container layer (OCF)
  - mimetype                  |
  - META-INF/container.xml    |
  - (everything else)         |
       +------------------+   |
        OPF package (.opf) |   <-- Package layer
        - metadata         |
        - manifest         |
        - spine            |
             +----------+  |
              Content    |  <-- Content layer
              - XHTML    |
              - CSS      |
              - images   |
              - fonts    |
             +----------+  |
       +------------------+ |
------------------------------+
```

This framing is not just conceptual. It maps directly to how reading systems actually open books: they start at the container, discover the OPF, then use the OPF to find everything else.

## How the pieces find each other (the “pointer chain”)

An EPUB is a bundle of files, so the number one engineering problem is discovery: how do you reliably locate the “entry point” of the book without guessing filenames or folder structures?

EPUB solves this with an explicit chain of references:

1) The reading system opens the ZIP and reads the `mimetype` file to confirm it is an EPUB container.

2) The reading system looks inside `META-INF/container.xml`. That file points to the path of the OPF package document.

3) The OPF’s manifest lists every resource (including content documents, navigation, styles, and assets).

4) The OPF’s spine selects a subset of manifest items and orders them into the linear reading flow.

If you remember nothing else from this chapter, remember this chain:

```text
META-INF/container.xml  -->  package.opf/content.opf  -->  manifest + spine  -->  XHTML content
```

This is why EPUB tooling can be robust. A reading system doesn’t need to guess “is the OPF named `content.opf` or `book.opf`?” It reads `container.xml`. And it doesn’t need to scan the whole ZIP to find chapters; it reads the spine.

## The container layer: `mimetype` and `META-INF/`

When you unzip an EPUB, the first thing you typically see is a `mimetype` file and a `META-INF/` directory at the top level. That top level is special: it is the root of the container.

The `mimetype` file is intentionally boring. It usually contains exactly:

```text
application/epub+zip
```

What matters is not the content (which is obvious), but the rule: the `mimetype` file must be the first file in the ZIP and must be stored without compression. This is one of those “seems pedantic until it breaks your book” constraints. The practical reason is that some reading systems want to identify the file type quickly without reading a large central directory or decompressing data. Later we’ll treat this as a packaging gotcha, but it belongs in your mental model of the container layer.

The `META-INF/` directory contains metadata about the container. The most important file inside it is `container.xml`. A typical `META-INF/container.xml` looks like this:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<container
  version="1.0"
  xmlns="urn:oasis:names:tc:opendocument:xmlns:container">
  <rootfiles>
    <rootfile
      full-path="OEBPS/content.opf"
      media-type="application/oebps-package+xml"/>
  </rootfiles>
</container>
```

Even if you’ve never seen this before, you can read it like a map. The critical attribute is `full-path`. That string is the path (inside the ZIP) to the OPF package document. Everything else in the publication will be discovered through that OPF.

One subtle point: EPUB allows *multiple* `<rootfile>` entries. That means, in theory, a container can include multiple package documents (think: multiple books in one container, or alternate renditions). Many tools never use this, but it explains why robust EPUB software treats “the OPF path” as a discovered value rather than a hardcoded assumption.

## The package layer: the OPF as the publication’s “index”

Once the reading system has the OPF path, it loads the OPF. This is where an EPUB stops being “a zip of files” and becomes “a publication.”

The OPF has three big responsibilities.

First, it provides **metadata**. This is where the title, author, language, unique identifier, publication date, and other descriptive data live. Metadata is used for library views, searching, and storefront ingestion, but it’s also used for technical features like accessibility metadata and rendition properties.

Second, it provides a **manifest**, which is a complete inventory of the resources in the publication. An important detail here is that EPUB does not want reading systems to “just scan directories.” A reading system should be able to rely on the manifest as the authoritative list of what exists and what its media types are.

Third, it provides a **spine**, which defines the default reading order. The spine is what turns a bag of content documents into a coherent “start here, then read this, then that” experience.

Here is a deliberately small, simplified sketch of the manifest + spine relationship. You don’t need to memorize the syntax yet; you just need to see the idea:

```xml
<manifest>
  <item id="nav" href="nav.xhtml" media-type="application/xhtml+xml" properties="nav"/>
  <item id="css" href="styles/book.css" media-type="text/css"/>
  <item id="ch1" href="text/chapter-1.xhtml" media-type="application/xhtml+xml"/>
  <item id="ch2" href="text/chapter-2.xhtml" media-type="application/xhtml+xml"/>
</manifest>

<spine>
  <itemref idref="ch1"/>
  <itemref idref="ch2"/>
</spine>
```

Notice the key mechanism: the spine does not list filenames directly. It references manifest items by ID (`idref="ch1"`). That indirection is intentional. It lets the OPF serve as the single source of truth about resources, and it gives reading systems a consistent way to resolve “what is this thing?” (path, media type, properties) for each spine entry.

## The content layer: what “pages” actually are in EPUB

If you come from web development, a tempting mistake is to imagine the “pages” of an EPUB as something like DOM pages in a browser. In EPUB, the closest equivalent to a “page file” is a **content document**.

A content document is typically a single XHTML file containing a section of the book. Most commonly, that means “one chapter per file,” but there are many valid patterns. Some EPUBs use one content document per chapter. Others split chapters into smaller files for performance or tooling convenience. Others put front matter (title page, copyright page) into separate content documents.

What matters is this: the reading system paginates and displays content documents according to the spine. A user may experience “pages,” but those pages are generated dynamically (reflow) from the content documents. The file boundaries are structural boundaries, not page boundaries.

Content documents link to stylesheets and assets using relative paths, just like a website. For example, a content document might have:

```xml
<link rel="stylesheet" type="text/css" href="../styles/book.css"/>
```

And it might reference an image like:

```xml
<img src="../images/figure-1.png" alt="A diagram of the EPUB layers."/>
```

The big difference from the open web is that these resources are expected to be present inside the EPUB container and listed in the OPF manifest.

## A typical on-disk structure (and what is “required” vs “conventional”)

One confusing part for beginners is that EPUB has strict rules, but it is not strict about *folder names*. The spec does not require you to use `OEBPS/` or `Text/` or `Styles/`. What it requires is that the container can locate the OPF, and the OPF can locate and declare everything else.

That said, the ecosystem has conventions, and you will see them repeatedly.

It is common to place the main publication contents under a single directory such as `OEBPS/` (historically common) or `EPUB/` (common in EPUB 3 examples and some toolchains). Inside that directory, it is common to separate content documents, stylesheets, images, and fonts into subfolders so that paths stay predictable.

Here is a realistic example structure you might see after unzipping:

```text
MyBook.epub
├─ mimetype
├─ META-INF/
│  └─ container.xml
└─ OEBPS/
   ├─ content.opf
   ├─ nav.xhtml            (EPUB 3 navigation document)
   ├─ toc.ncx              (EPUB 2 navigation document, often included for compatibility)
   ├─ text/
   │  ├─ titlepage.xhtml
   │  ├─ chapter-01.xhtml
   │  └─ chapter-02.xhtml
   ├─ styles/
   │  └─ book.css
   ├─ images/
   │  ├─ cover.jpg
   │  └─ figure-1.png
   └─ fonts/
      └─ SomeFont-Regular.otf
```

Nothing about the names `text/`, `styles/`, or `images/` is magical. You could rename them to `chapters/`, `css/`, and `img/` and still be valid, as long as the OPF `href` values match and everything is present in the manifest.

This is an important distinction to carry into debugging. When something fails to load, don’t ask “is it in the correct folder?” first. Ask “does the OPF manifest list it, and does every `href` and link resolve correctly from where it is referenced?”

## The spine: “reading order” is explicit (and it can differ from your table of contents)

The **spine** deserves special attention because it’s where EPUB becomes “book-like” rather than “website-like.”

On the web, users can land on any page first. A website might have a homepage, but the architecture assumes multiple entry points. In contrast, books have a default reading flow. EPUB represents that flow with the spine.

The spine is a list of references to manifest items, ordered the way the book should be read. Reading systems use it to determine what to show when the user taps “next page” across file boundaries and to decide what “beginning of the book” means.

A subtle but practical detail is that the spine can include items that are marked as non-linear (in EPUB 2) or otherwise treated as auxiliary. For example, an index, acknowledgments, or bonus material might exist in the package but not be part of the main reading flow. Different reading systems handle these nuances differently, which is why you should treat the spine as your primary “truth” of reading order, but not assume every feature is implemented identically.

Another subtlety is that the spine and the navigation table of contents do not have to match one-to-one. The spine is the reading order. The table of contents is the user-facing map. Many books have sections that are read but not shown as ToC entries (for example, a blank page, a half-title, a decorative separator, or a copyright page). Conversely, a ToC may point into the middle of a content document using a fragment identifier (`chapter-01.xhtml#section-3`) rather than pointing to a whole file.

For an editor tool like EasyPub, this distinction matters: when you split or merge content documents, you are not merely moving files around. You are editing the spine-level structure of the publication, which affects how the book progresses in every reading system.

## EPUB 2 and EPUB 3 differences you see in the structure

At the file-structure level, EPUB 2 and EPUB 3 feel similar because both share the same container concept and both use an OPF package document. The practical differences show up in navigation and in the “content document” expectations.

EPUB 2 typically uses XHTML 1.1-ish constraints and relies on an **NCX** file (`.ncx`) for the table of contents. EPUB 3 introduces the **navigation document** (`nav.xhtml`) as the primary ToC mechanism and aligns content documents more closely with HTML5-era semantics (while still usually being authored as XHTML for strictness and tool compatibility).

In the real world you will often see hybrid books: an EPUB 3 package that includes both `nav.xhtml` and `toc.ncx` so that older reading systems still have a ToC. This is not “cheating.” It is a pragmatic response to mixed device support.

The key point for your mental model is that “EPUB version” is not a cosmetic label. It changes which navigation file a reading system expects and which features are safe to rely on.

## A practical debugging approach: locate these three things first

When you unzip an EPUB and you’re trying to understand or debug it, you can reduce the chaos by locating three things in order.

First, locate `META-INF/container.xml` and read the `full-path` to find the OPF. If you can’t find the OPF, nothing else matters yet.

Second, open the OPF and identify the manifest and spine. If the manifest is missing resources that are referenced by content documents, you’ve found a structural bug. If the spine references an ID that doesn’t exist in the manifest, you’ve found a structural bug. If the spine’s order is wrong, you’ve found a “book logic” bug that will show up as chapters appearing in the wrong sequence.

Third, identify the navigation structure (nav doc and/or NCX) and check whether it points to real content. A broken ToC often comes down to a path mismatch, a missing fragment ID, or a content document being renamed without updating the nav references.

This workflow is simple, but it maps to how reading systems load publications. It also prevents you from spending an hour staring at CSS when the real problem is that a chapter isn’t in the spine.

## Common pitfalls and gotchas

The most common pitfall in EPUB structure is confusing “the files exist” with “the book is valid.” An EPUB can contain perfectly fine XHTML and images, and still fail in reading systems because the OPF manifest doesn’t list a resource, the spine points at the wrong thing, or `container.xml` points to an OPF path that doesn’t exist. The package layer is not optional bookkeeping; it is how the book is defined.

Another common pitfall is forgetting that the container has weird ZIP rules. The `mimetype` file being first and uncompressed is a famously annoying requirement, and it is easy to violate accidentally if you “just zip the folder” using a naive GUI tool. Some readers forgive this; some do not; validators will complain; storefront pipelines may reject it.

Path and casing bugs are also disproportionately common. EPUB paths are just strings inside XML, and they must match exactly. If your OPF says `Text/Chapter-01.xhtml` but the file is actually `text/chapter-01.xhtml`, you may get a book that “works on my machine” in one reader and breaks in another depending on how strictly it resolves paths and how it treats case sensitivity. This is especially nasty when EPUBs are produced on case-insensitive filesystems (like many macOS setups) and then validated or deployed elsewhere.

A related pitfall is thinking the directory structure is standardized. You’ll see `OEBPS/` a lot, but it is not guaranteed. If you write tooling that assumes `OEBPS/content.opf`, it will work on some books and then fail mysteriously on others. Always follow the discovery chain: start at `META-INF/container.xml`, then follow `full-path`.

Finally, be careful about the difference between the spine and navigation. A ToC that looks correct does not guarantee the reading order is correct, and a correct spine does not guarantee the ToC is correct. Good EPUBs keep them consistent, but they can drift apart during edits and conversions. When you change structure (splitting, merging, renaming content documents), you must think in terms of “what did I change in the spine?” and “what did I change in navigation?” as two separate responsibilities.
