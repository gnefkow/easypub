# 3.1 XHTML vs HTML: what’s the difference, and why it matters (especially for EPUB)

When people say “EPUB uses HTML,” they usually mean **XHTML** (an XML-serialized form of HTML). In EPUB 2 and EPUB 3, the *content documents* you edit (chapters/sections) are typically `.xhtml` files that must be **well‑formed XML**. That single constraint—*well‑formedness*—is the practical difference that matters for tools like EasyPub.

## The core difference in one sentence

- **HTML (text/html)**: forgiving; browsers recover from many markup mistakes.
- **XHTML (XML)**: strict; the document must be parseable as XML or it can fail to parse/render correctly in reading systems and tooling.

## Why EPUB cares about XHTML

EPUB is a ZIP package with well-defined, machine-validated documents (OPF, navigation, content). Reading systems, validators (like epubcheck), and processing tools commonly treat EPUB content documents as **XML** (or at least *expect XML-like correctness*).

That matters because:

- **Validation**: A single unclosed tag or malformed attribute can break EPUB validation.
- **Parsing/manipulation**: Tools that manipulate the DOM (like EasyPub’s server-side edits via Cheerio) behave more predictably when documents are well-formed.
- **Interoperability**: Different eReaders vary in how forgiving they are. XHTML reduces “works on my device” surprises.

## What “well‑formed XML” means in practice

If your EPUB content documents are XHTML, you should assume the following rules always apply:

### Tags must be properly nested and closed

HTML might “fix” this:

```html
<p><em>Hello</p></em>
```

XHTML must be properly nested:

```xml
<p><em>Hello</em></p>
```

### No optional end tags

In HTML, you can often omit closing tags. In XHTML, you generally **cannot**.

Bad (HTML-tolerated, XHTML-invalid):

```html
<p>First paragraph
<p>Second paragraph
```

Good (XHTML):

```xml
<p>First paragraph</p>
<p>Second paragraph</p>
```

### Attribute values must be quoted

Bad:

```html
<p class=lead>
```

Good:

```xml
<p class="lead">
```

### Case sensitivity is real (XML rule)

XML is case-sensitive. In XHTML, you should use lowercase element and attribute names consistently.

Risky:

```xml
<P Class="lead">Text</P>
```

Preferred:

```xml
<p class="lead">Text</p>
```

### Empty elements must be self-closed (XML rule)

Bad:

```html
<br>
<img src="cover.jpg">
```

Good:

```xml
<br />
<img src="cover.jpg" alt="Cover" />
```

### Entities must be valid XML/HTML entities

XHTML supports `&amp;`, `&lt;`, `&gt;`, `&quot;`, `&apos;` and named HTML entities **only if they’re defined/understood** by the processor. Safer approach: use the standard ones (or numeric entities) and always escape `&`.

Bad:

```xml
<p>Tom & Jerry</p>
```

Good:

```xml
<p>Tom &amp; Jerry</p>
```

## What an EPUB XHTML file usually looks like

Most EPUB 3 content documents resemble this (details vary):

```xml
<?xml version="1.0" encoding="utf-8"?>
<!DOCTYPE html>
<html xmlns="http://www.w3.org/1999/xhtml" xml:lang="en" lang="en">
  <head>
    <meta charset="utf-8" />
    <title>Chapter 1</title>
    <link rel="stylesheet" type="text/css" href="../styles/book.css" />
  </head>
  <body>
    <h1>Chapter 1</h1>
    <p>…</p>
  </body>
</html>
```

Key things to notice:

- **XML declaration** may be present.
- The root `<html>` often includes the **XHTML namespace** (`xmlns="http://www.w3.org/1999/xhtml"`).
- Empty tags are **self-closed** (`<meta />`, `<link />`).

## Why this matters for EasyPub specifically

From the EasyPub overview:

- EasyPub displays EPUB *content documents* (“Sections”) in the Viewer.
- EasyPub represents many elements as `TextBlock`s (e.g., `<p>`, `<h1>`, `<blockquote>`).
- EasyPub queues edits and then applies them in a server-side batch using DOM manipulation.

Because those “Sections” are typically XHTML:

- **Edits must preserve well‑formedness**. If an action deletes a node, appends one block into another, or splits a section, the resulting document must still be valid XML/XHTML.
- **Small formatting mistakes can cascade**. For example, appending raw text that contains an unescaped `&` can make the entire file invalid.
- **Parsing mode matters**. Libraries may parse differently in HTML vs XML mode (e.g., handling of self-closing tags, namespaces, and optional tags). When you treat XHTML as “just HTML,” you can accidentally introduce invalid markup that browsers would hide but EPUB tools will reject.

## Common pitfalls when converting “web HTML” into EPUB XHTML

- **Missing closing tags** (`</p>`, `</li>`, `</span>`)
- **Unquoted attributes** (`class=lead`)
- **Void elements not self-closed** (`<br>`, `<img>`)
- **Unescaped ampersands** (`AT&T` → `AT&amp;T`)
- **Invalid nesting** (block elements inside `<p>`, overlapping inline tags)
- **Namespace removal** on the root `<html>` (can break strict tooling)

## Practical rule of thumb

If you can parse the content document as XML without errors, you’re usually on the right track. If you rely on “the browser fixes it,” you’re in HTML-land—and that’s where EPUB bugs and validation failures tend to come from.
