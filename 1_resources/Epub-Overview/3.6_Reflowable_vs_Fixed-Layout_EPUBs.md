# 3.6 Reflowable vs Fixed-Layout EPUBs

If you only remember one thing about EPUB, make it this: EPUB’s default superpower is that it is **reflowable**. The text is not “painted onto a page.” It is laid out at reading time, based on the device’s screen size, the reader’s chosen font size, margins, line spacing, and even theme. That flexibility is why EPUB works on phones, tablets, e-ink devices, and desktop readers.

But not every book is primarily text. Some books are inherently visual: picture books, comics, magazines, cookbooks, design portfolios, and complex textbooks with tight relationships between images and text. For these, the “reflow everything” model can be a poor fit. That’s where **fixed-layout** EPUBs come in. Fixed-layout (often abbreviated **FXL**) gives you a page that behaves more like a designed canvas: content is positioned intentionally, and the reading system is asked to preserve that layout.

This chapter gives you a mental model for both approaches, shows how fixed-layout is declared in the OPF and in content documents, and helps you reason about the trade-offs. If you are building tools that generate or edit EPUBs, this chapter is also about making correct decisions: when to prefer reflowable, when fixed-layout is justified, and how to avoid “hybrid” mistakes that fail on real devices.

## Key vocabulary (for this chapter)

**Reflowable** content is content whose layout is determined dynamically by the reading system. Text rewraps, pagination changes, and the user’s settings can dramatically alter the page count and line breaks. **Fixed-layout** (FXL) content is content whose layout is intended to be preserved: the book declares a fixed viewport, and content is positioned relative to that viewport.

**Pre-paginated** is the standard term used in EPUB metadata to mean “fixed-layout pages” (as opposed to dynamically paginated reflowable text). A **viewport** is the coordinate space that a fixed-layout page is designed for, often declared with a viewport meta tag. A **rendition** is essentially “a way this publication wants to be rendered,” and the EPUB ecosystem uses **rendition properties** like `rendition:layout` to describe those preferences. A **synthetic spread** is when a reading system shows two pages side by side (like an open book), often relevant for fixed-layout comics and picture books.

## Reflowable EPUB: the default model

Reflowable EPUB is what most people mean when they say “EPUB.” It behaves like a web document in a reading app: it’s structured HTML/XHTML content styled with CSS. The key difference from the open web is that the reading system usually exposes strong user controls: font size, font face, line spacing, margins, and sometimes even justification, hyphenation, and theme. Those controls don’t just change the appearance—they change the layout.

That means reflowable layout has a few important consequences.

First, page numbers are not stable. When a user increases font size, the same chapter may go from “10 pages” to “25 pages.” Any workflow that tries to treat reflowable EPUB like a print layout will fight the format.

Second, precise alignment between text and images is not guaranteed. You can insert images, but their position relative to nearby text can shift because the text rewraps and pagination changes.

Third, reflowable EPUB is generally the best option for accessibility. Because the content is real text, assistive technologies can navigate it, read it aloud, and allow the user to adjust it. Reflowable EPUB is also usually more resilient across devices and apps.

If you’re editing a typical novel, memoir, nonfiction text, or any book where the primary experience is reading paragraphs, reflowable is almost always the correct choice.

## Fixed-layout (FXL): when the page is the product

Fixed-layout EPUB changes the mental model. Instead of “a flowing document,” you have “a set of designed pages.” Each page has a defined size (a viewport), and elements are placed intentionally within that space. It’s closer to a PDF in spirit, but it is still packaged as EPUB: it still has an OPF, a manifest, a spine, and content documents.

FXL is most appropriate when layout conveys meaning. Comics are a clear example: speech bubbles and panels have spatial relationships that are part of comprehension. Children’s picture books often have text integrated into illustrations. Some textbooks and cookbooks are designed so that specific text must appear next to a specific image.

Fixed-layout is not automatically “higher quality.” It is a trade. You get layout control, but you usually give up some of EPUB’s best properties: adaptability to small screens, user customization, and accessibility.

## Declaring fixed-layout in the OPF with `rendition:layout`

In EPUB 3, the key declaration is the `rendition:layout` property. Conceptually, it answers the question: “Should this publication (or a given spine item) be treated as reflowable or pre-paginated?”

At the package level, you typically declare it in OPF metadata:

```xml
<package xmlns="http://www.idpf.org/2007/opf"
         unique-identifier="pub-id"
         version="3.0">
  <metadata xmlns:dc="http://purl.org/dc/elements/1.1/">
    <dc:identifier id="pub-id">urn:uuid:12345678-1234-1234-1234-123456789abc</dc:identifier>
    <dc:title>Example Fixed-Layout Book</dc:title>

    <meta property="rendition:layout">pre-paginated</meta>
  </metadata>
  <!-- manifest and spine... -->
</package>
```

The value `pre-paginated` is the fixed-layout signal. The value `reflowable` is the reflowable signal. If you omit the property, reading systems generally assume reflowable.

This package-level setting is the “default.” Many fixed-layout books are entirely pre-paginated, so a single declaration is enough.

### Overriding layout per spine item

Sometimes a publication is mostly fixed-layout but still includes a reflowable section, like a copyright page, an introduction, or a long reading section that’s better as text. EPUB supports per-item preferences through the `properties` attribute on `spine` `itemref` elements (using the rendition properties vocabulary).

Conceptually, it looks like this:

```xml
<spine>
  <itemref idref="cover" properties="layout-pre-paginated" />
  <itemref idref="page-001" properties="layout-pre-paginated" />
  <itemref idref="page-002" properties="layout-pre-paginated" />

  <itemref idref="afterword" properties="layout-reflowable" />
</spine>
```

The details of your spine structure depend on your workflow, but the idea is powerful: you can build a hybrid book where the parts that need a fixed canvas get it, and the parts that are better as real text remain reflowable.

## The viewport meta tag in fixed-layout content documents

A fixed-layout page needs a coordinate system. On the web, you might think of this as “the design width and height.” In fixed-layout EPUB, a common way to express this is the viewport meta tag in the content document’s `<head>`.

Here’s a simplified fixed-layout XHTML page:

```xml
<html xmlns="http://www.w3.org/1999/xhtml"
      xml:lang="en"
      lang="en">
  <head>
    <meta charset="utf-8" />
    <title>Page 1</title>
    <meta name="viewport" content="width=1200, height=1600" />
    <link rel="stylesheet" type="text/css" href="../styles/fxl.css" />
  </head>
  <body>
    <div class="page">
      <img class="background" src="../images/page-001.jpg" alt="" />
      <p class="caption">A short caption placed precisely on the page.</p>
    </div>
  </body>
</html>
```

The viewport declaration is the clue to the reading system: this page is designed for a specific width and height. The reading system will typically scale that page to fit the device’s screen, often allowing zooming and panning depending on the app.

Note the empty `alt` on the background image. That’s a reminder that many fixed-layout pages are built as large images with text layered on top. This can look great, but it has serious accessibility implications: if the “real content” is baked into an image, it is not naturally available to screen readers and it does not reflow.

## How fixed-layout pages are typically built

There are a few common patterns for fixed-layout pages.

One pattern is a full-page background image (a scanned or exported page design) with text positioned over it using CSS. This is common in picture books and some comics workflows. The advantage is visual fidelity. The disadvantage is accessibility and file size.

Another pattern is to build the page with HTML elements and CSS positioning directly, using images only where needed. This can be more accessible if the text is real text and the reading order is sensible, but it is harder to author and more likely to run into rendering differences across reading systems.

A third pattern uses SVG heavily. Because SVG is vector-based, it can represent a whole page layout with scalable shapes and text. This can work well for diagrams and design-heavy pages, but it also moves you into a more specialized toolchain and a more variable support landscape.

## Choosing between reflowable and fixed-layout

The easiest way to choose is to ask what you are optimizing for.

If your top priority is readability across devices, user control, accessibility, and long-form text, reflowable is the right default. It’s what EPUB is best at.

If your top priority is preserving a designed page where spatial relationships are part of meaning, fixed-layout is justified. That tends to be true for comics, children’s books, and certain textbooks.

It’s also common to make a strategic compromise: keep the “story” or long narrative parts reflowable, and use fixed-layout only for sections where it’s truly necessary.

## Trade-offs: accessibility, compatibility, file size, and user control

Fixed-layout often reduces accessibility. Users can’t reliably change font size without breaking the layout, and if text is embedded in images, it may not be readable by assistive technologies. Even when text is real text, the reading order can be confusing if the layout is primarily visual.

Fixed-layout also reduces compatibility. Many reading systems support FXL to some degree, but behavior varies: zooming, spreads, orientation handling, and page scaling can all differ between apps. Reflowable books have differences too, but they are generally more predictable.

File size is another common cost. A reflowable novel might be a few megabytes. A fixed-layout picture book with dozens of full-page images can be hundreds of megabytes if not carefully optimized.

Finally, fixed-layout reduces user control, which can be a negative even for users who are not using assistive tech. Readers often want to increase font size, adjust margins, or switch fonts. Fixed-layout intentionally constrains those options.

These trade-offs are why FXL should be chosen deliberately, not as a way to “make EPUB look like print.”

## Common pitfalls and gotchas

The first gotcha is trying to force reflowable content to behave like fixed-layout. You can do a lot with CSS, but reflowable EPUB is fundamentally a dynamic layout environment. If your design requires exact pixel placement of text relative to an image, you are fighting the format. The result is usually an EPUB that looks “almost right” in one reader and broken in another.

The second gotcha is declaring fixed-layout inconsistently. If the OPF metadata says `rendition:layout` is `pre-paginated` but your pages don’t have a sensible viewport, different reading systems will guess differently. If you mix reflowable and fixed-layout items, be explicit about per-item layout properties so readers don’t inherit the wrong behavior.

The third gotcha is accessibility regression. A fixed-layout page built as a single image is effectively a poster. If that poster contains important text, you need an alternative: real text on top (with correct reading order), meaningful descriptions, or a reflowable equivalent section. FXL can be made more accessible, but it requires deliberate effort.

The fourth gotcha is performance. Full-page images are expensive: they increase file size, decode time, and memory usage. On lower-end devices, that can mean blank pages, crashes, or long loading delays. If you choose FXL, budget time for image optimization and device testing.

The fifth gotcha is spread and orientation behavior. Synthetic spreads (two-page views) and orientation rules differ across reading systems. A comic that “must” be two-page spreads in landscape may be shown as single pages on some devices. The best fixed-layout publications tolerate these differences rather than assuming one perfect display mode.

Finally, remember that EPUB is not one renderer. The spec provides vocabulary like `rendition:layout` and the viewport meta tag, but the gap between “declared intent” and “actual behavior” is real. When fixed-layout is essential, you test on multiple reading systems early. When it isn’t essential, you usually choose reflowable and let EPUB do what it does best.
