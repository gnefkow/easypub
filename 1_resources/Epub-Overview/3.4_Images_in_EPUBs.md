# 3.4 Images in EPUBs

Images are where EPUB starts to feel less like “a bunch of HTML files in a ZIP” and more like a real publishing format. Covers, illustrations, diagrams, icons, and decorative flourishes are all just files inside the EPUB package, referenced from your XHTML, styled with CSS, and declared in the OPF manifest so the reading system knows they exist.

If you already know the web, the basic mechanics will feel familiar: you use `<img>` with a `src` attribute, you can use CSS to control sizing, and you can use SVG when you want crisp vector graphics. What changes in EPUB is the ecosystem. Your “browser” is a reading system with its own constraints, image decoding support, memory limits, pagination model, and accessibility features. A technique that works in one reader might be ignored or behave differently in another, so the job is not just “render the image,” but “render it reliably and meaningfully across devices.”

This chapter builds a practical mental model for images in EPUB 3: which formats are safe, how image files are referenced and packaged, how cover images are represented, how to size images in reflowable layout, when SVG is a good idea, and what accessibility expectations look like.

## Key vocabulary (for this chapter)

An **asset** (or **resource**) is any non-text file packaged in the EPUB, such as an image, font, audio file, or stylesheet. A **core media type** is a file format that EPUB reading systems are expected to support broadly (for images, that’s typically JPEG, PNG, GIF, and SVG). The **OPF manifest** is the “inventory list” in the package document (`content.opf`) where every file in the EPUB should be declared with a media type. A **viewport** is the visible reading area on the device; in reflowable EPUBs it changes dramatically depending on screen size, font size, margins, and orientation. **Raster** images (JPEG, PNG, GIF) are pixel-based; **vector** images (SVG) are shape-based and scale cleanly. **Alt text** is text provided via the `alt` attribute to describe an image for screen readers (and other non-visual uses). A **figure** is a semantic container for an image and its caption, usually expressed with `<figure>` and `<figcaption>`.

## Core image formats: what “supported” really means

Most EPUB 3 workflows treat JPEG, PNG, GIF, and SVG as the practical baseline. You will also see WebP and AVIF on the modern web, but EPUB reading systems are not the modern web. Even if a format is “allowed” in some contexts, it may not decode everywhere you care about.

JPEG is the common choice for photographic images because it compresses well. PNG is common for images that need sharp edges or transparency (logos, UI-like diagrams, icons), though it can be much larger than JPEG for photo-like content. GIF appears mostly as legacy content or for simple animation; some reading systems support animated GIFs, others display only the first frame, and some reduce animation for performance or battery life. SVG is a vector format, which makes it ideal for diagrams and line art that need to remain crisp at any size.

The key mindset is: pick formats that are boring. In EPUB, boring is reliable.

## Where images live inside the EPUB

An EPUB can store images anywhere as long as the paths are consistent and the files are declared in the manifest. In practice, you’ll often see a folder like `images/`:

```text
OEBPS/
  content.opf
  text/
    chapter-01.xhtml
    chapter-02.xhtml
  styles/
    book.css
  images/
    cover.jpg
    fig-01.png
    diagram.svg
```

Nothing about `images/` is magical. It’s just a convention that makes the package easier to understand. What *is* magical (in the sense of “the reader cares”) is that the image files are listed in `content.opf` and referenced with correct relative paths from the XHTML that uses them.

## Referencing images from XHTML content documents

The simplest and most common way to include an image is the `<img>` element:

```xml
<img src="../images/fig-01.png" alt="A simple diagram showing EPUB as a ZIP container." />
```

There are three details here that matter a lot in EPUB work.

First, the path in `src` is a relative path from the XHTML file to the image file. If your XHTML is in `text/` and your images are in `images/`, you usually need `../images/...`. Broken relative paths are the number one reason images “randomly” fail in a reader.

Second, because EPUB content documents are typically XHTML, the tag must be well-formed. That means the empty element is self-closing (`/>`) and the attribute value is quoted.

Third, `alt` is not optional in practice. A missing or meaningless `alt` might not prevent rendering, but it hurts accessibility and can cause automated checkers (including accessibility checkers) to flag your book.

### Using `<figure>` and `<figcaption>` for captions

If an image has a caption that the reader should see, it’s helpful to use HTML semantics instead of inventing ad-hoc markup:

```xml
<figure class="figure">
  <img src="../images/diagram.svg" alt="A flow diagram: container.xml points to content.opf, which lists content documents." />
  <figcaption>Figure 3.1: How an EPUB points to its content.</figcaption>
</figure>
```

This is a good example of EPUB-friendly design: the markup expresses meaning (this is a figure with a caption), and the styling can be done in CSS. If a reader ignores your CSS or restyles captions, the structure still makes sense.

## Declaring images in the OPF manifest

In EPUB, it’s not enough to put an image file in the ZIP and reference it from XHTML. You should also declare it in the OPF manifest. The manifest is the complete inventory of files in the package, and it’s how many reading systems validate “is this a legitimate resource in the book?”

Here is a typical manifest snippet:

```xml
<manifest>
  <item id="css" href="styles/book.css" media-type="text/css" />

  <item id="cover-image" href="images/cover.jpg" media-type="image/jpeg" properties="cover-image" />
  <item id="fig-01" href="images/fig-01.png" media-type="image/png" />
  <item id="diagram" href="images/diagram.svg" media-type="image/svg+xml" />

  <item id="ch1" href="text/chapter-01.xhtml" media-type="application/xhtml+xml" />
  <item id="ch2" href="text/chapter-02.xhtml" media-type="application/xhtml+xml" />
</manifest>
```

The important idea is that `href` values in the manifest are relative to the OPF file’s location (often the `OEBPS/` folder). This is a common source of confusion: XHTML resolves image paths relative to the XHTML file, but the manifest resolves paths relative to the OPF. Both must be correct.

Also notice the media types. EPUB cares about these strings. A `.jpg` is usually `image/jpeg`, a `.png` is `image/png`, and SVG should be `image/svg+xml`. If the media type is wrong, some validators will complain, and some reading systems will refuse to load the resource or will behave unpredictably.

## Cover images: two things, not one

When people say “the cover,” they often mean two related but distinct things.

The first is the **cover image file** (a JPEG or PNG) that stores the artwork. In EPUB 3, the modern convention is to mark that manifest item with `properties="cover-image"`, as shown above.

The second is a **cover content document** (an XHTML file) that displays the cover image when the reader navigates to the cover as part of the spine reading flow. Many reading systems want both: the metadata cover image so they can show a thumbnail in a library view, and an actual cover page so the cover appears as the first “page” when reading.

An example cover XHTML document might look like this:

```xml
<html xmlns="http://www.w3.org/1999/xhtml"
      xmlns:epub="http://www.idpf.org/2007/ops"
      xml:lang="en"
      lang="en">
  <head>
    <meta charset="utf-8" />
    <title>Cover</title>
    <link rel="stylesheet" type="text/css" href="../styles/book.css" />
  </head>
  <body epub:type="cover">
    <section class="cover">
      <img class="cover-image" src="../images/cover.jpg" alt="Cover of the book." />
    </section>
  </body>
</html>
```

And then the OPF would include the cover XHTML in the manifest and likely in the spine:

```xml
<item id="cover" href="text/cover.xhtml" media-type="application/xhtml+xml" />
```

The details vary by production workflow, but the mental model is stable: “cover in metadata” and “cover as a page” are related, and you usually want both for broad compatibility.

## Image sizing in reflowable EPUBs: think “responsive by default”

On the web you might design for a particular viewport width. In EPUB you don’t really get to pick. The viewport depends on device size, orientation, font size, margins, and the reading system’s UI chrome. An image that looks perfectly sized on an iPad might overflow on a small phone, and an image that fits on a phone might look tiny on a desktop reader.

For that reason, the default strategy in reflowable EPUBs is simple: make images scale down to the available width, and keep their aspect ratio.

A common CSS pattern looks like this:

```css
img {
  max-width: 100%;
  height: auto;
}
```

This tells the reading system: if the image is wider than the viewport, shrink it to fit. If it is smaller, leave it at its natural size. The `height: auto` part preserves aspect ratio so the image doesn’t become stretched.

If you want a figure to be centered and have comfortable spacing, you can style the container:

```css
.figure {
  margin: 1em 0;
  text-align: center;
}

.figure img {
  display: block;
  margin: 0 auto;
}

.figure figcaption {
  font-size: 0.9em;
  margin-top: 0.5em;
}
```

Two practical notes: many reading systems apply their own default margins and font sizing, so treat this as a starting point, not a guarantee. Also, avoid hardcoding pixel widths in reflowable layout unless you have tested widely, because fixed pixel sizes fight user-controlled font scaling and small screens.

### Should you set `width` and `height` attributes?

On the web, `width` and `height` can help avoid layout shifts. In EPUB, they can still be useful as hints, but they also can backfire if they force a fixed size. If you do include them, pair them with responsive CSS so the image can still shrink:

```xml
<img src="../images/fig-01.png"
     alt="A diagram of an EPUB folder layout."
     width="1200"
     height="800" />
```

With `max-width: 100%` in CSS, the image can still scale down when needed. Without it, a reading system might try to honor the width too literally.

## SVG in EPUBs: great for diagrams, with a few rules

SVG shines in EPUB because diagrams and line art often need to remain crisp at multiple sizes. A raster diagram can look blurry on a high-density screen unless it’s shipped at a very large pixel size, which increases file size. An SVG diagram can stay sharp while remaining relatively small.

The simplest and most compatible way to use an SVG is as the `src` of an `<img>` element:

```xml
<img src="../images/diagram.svg" alt="A diagram showing how content.opf lists all assets in the EPUB." />
```

You will sometimes see inline SVG embedded directly in the XHTML. That can work, but it makes the XHTML more complex and increases the odds of running into quirks in strict XHTML parsing or reading system SVG support. If your workflow doesn’t require inline SVG, a referenced SVG file is often the safer route.

When you author or edit SVG for EPUB, avoid relying on external resources (like web fonts or linked images) unless you are sure they are packaged and declared properly. Think of an SVG as a self-contained asset that should render without the network and without assumptions about the environment.

## Accessibility: images must have a text story

In an accessible EPUB, images are not “extra.” They are part of the content, and they need an equivalent experience for readers who can’t see them.

For most images, the starting point is a good `alt` attribute. “Good” means it communicates the image’s purpose in context. If the image is purely decorative and contributes no meaning, the best practice is an empty alt text so screen readers skip it:

```xml
<img src="../images/ornament.png" alt="" />
```

If an image is complex (a detailed chart, a dense diagram), short alt text may not be enough. In those cases, you can provide a longer description in nearby text, in a caption, or in a referenced description section. The right choice depends on editorial goals, but the principle is consistent: readers should not lose meaning because they can’t see a particular image.

Also remember that “accessibility” includes more than screen readers. Low-vision readers may zoom. Dyslexic readers may prefer larger text and different fonts. Images that contain important text (for example, a screenshot of a paragraph) are often a red flag because the text inside the image won’t reflow, can’t be resized as easily, and isn’t naturally exposed to assistive technology. If the text matters, prefer real text in XHTML whenever possible.

## File size and performance: images can make or break an EPUB

Images are frequently the largest contributor to an EPUB’s file size. Large images also increase memory usage when decoding, which can cause slow page turns, blank pages while loading, or outright failures on constrained devices.

A practical EPUB image workflow is to compress images intentionally. For photos, that often means JPEG with a reasonable quality setting and dimensions that match likely display sizes. For diagrams, that often means SVG or a carefully optimized PNG. For images with transparency, PNG may be necessary, but you still want to control dimensions.

In reflowable books, a common mistake is shipping a cover image that is far larger than necessary because it came from a print workflow. The cover might be 6000 pixels wide even though no reading system will display it at that resolution. The result is a huge file that slows down the book for no benefit.

## Common pitfalls and gotchas

The most common failure mode is a broken path. The `src` in an XHTML file is resolved relative to that XHTML file, while the `href` in the OPF manifest is resolved relative to the OPF. If you move files between folders or rename `images/` to `Images/`, you can end up with a situation where the manifest is correct but the XHTML is wrong (or vice versa). Case sensitivity matters on many reading systems, even if your development machine is forgiving.

The second gotcha is forgetting the manifest entry. Some reading systems will render an image that is referenced but not declared, but you should not rely on that. Treat the manifest as the authoritative list of allowed resources, and keep it in sync.

The third gotcha is incorrect media types. A `.svg` declared as `image/svg` instead of `image/svg+xml`, or a `.jpg` declared as `image/jpg`, can trigger validation warnings and can break rendering on strict systems.

The fourth gotcha is unhelpful alt text. Alt text like “image” or “diagram” is almost as bad as no alt text at all, because it adds noise without meaning. On the other hand, overly long alt text can also be unpleasant. Aim for “what the reader needs to know,” and push truly long explanations into nearby text.

The fifth gotcha is forcing fixed sizes. Hardcoded `width: 600px` styles, tables built as images, and images that assume a specific viewport all tend to fail on small devices or with large user font settings. In reflowable EPUBs, prefer flexible rules like `max-width: 100%` and let the reading system paginate naturally.

The sixth gotcha is SVG quirks. SVG support varies, and an SVG that looks perfect in a modern browser can still behave oddly in a reading system. Keep SVGs simple, avoid scripting, avoid external dependencies, and test on at least a couple of different readers when SVG is important to the book’s meaning.

Finally, remember the EPUB mindset: images should enhance the reading experience, but the book should remain usable if an image renders smaller than you expected, if a reader restyles captions, or if a device is underpowered. When you package images with conservative formats, correct manifest entries, responsive sizing, and thoughtful alt text, you get EPUBs that hold up in the real world.
