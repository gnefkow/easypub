# 2.1. An EPUB Is a ZIP File

## Key vocabulary

**ZIP archive** is a single file that contains a directory tree of other files, plus a table of contents that lets tools extract those files. A ZIP can also compress its contents, but compression is optional.

**Open Container Format (OCF)** is the EPUB “container” specification: the rules that say how a ZIP becomes a valid EPUB container. OCF covers things like the required `mimetype` file and the `META-INF` directory.

**`mimetype`** is a required file at the root of an EPUB whose content identifies the archive as an EPUB. It must contain the string `application/epub+zip`, and it has special packaging rules.

**`META-INF/`** is a required directory at the root of an EPUB container. It contains container-level metadata, most importantly `container.xml`.

**`container.xml`** is the file that tells a reading system where the OPF package document lives. The container does not assume a fixed path like `OEBPS/content.opf`; it must be discoverable through `container.xml`.

**Package document (OPF)** is the XML file (often named `content.opf` or `package.opf`) that defines the publication’s metadata, the list of resources (manifest), and the reading order (spine). We’ll go deep on this in Chapter 2.4, but in this chapter you just need to know it’s the “entry point” for book content.

## The foundational insight: an `.epub` is a renamed `.zip`

If you take one of the best selling novels on your phone and email yourself the `.epub` file, you can rename it to `.zip` and open it with everyday tools. This is not a hack or a loophole. It’s the design.

This is a big deal for two reasons.

First, it means EPUB is a *bundle format* rather than a single monolithic document. Your “book” is actually a collection of XHTML files (chapters), CSS (styling), images, fonts, and a few XML documents that describe how those pieces fit together.

Second, it means the way you debug and build EPUBs is often the way you debug and build websites: inspect files, follow references, and keep paths consistent. The difference is that an EPUB has stricter rules than the web, because reading systems can’t afford to guess.

## A ZIP, but with rules: what OCF adds on top

Not every ZIP file is an EPUB, even if you name it `something.epub`. EPUB is “ZIP + constraints.” Those constraints come from **OCF**, the Open Container Format.

The two constraints you’ll run into immediately are:

The EPUB must contain a file named `mimetype` at the root of the archive, and it must be packaged in a specific way.

The EPUB must contain a `META-INF/container.xml` file that points to the OPF package document.

If either of those is wrong, many reading systems won’t open the book. Some will open it anyway. That inconsistency is one of the themes you’ll see throughout this book: the spec is strict, and the real world is “strict-ish.”

## Unzipping and inspecting an EPUB on any operating system

Before we talk about the required files, you should be comfortable with a simple workflow: take an EPUB, unzip it into a folder, and explore.

On macOS and Linux, you can do this with the command line. Suppose your file is `book.epub`.

```bash
mkdir -p book_unzipped
unzip book.epub -d book_unzipped
```

If you just want to peek at the file list without extracting everything, you can list the archive:

```bash
unzip -l book.epub
```

On Windows, you have a few options.

The “quick and dirty” way is to duplicate the file and rename the copy from `book.epub` to `book.zip`, then open it in File Explorer. This works because the container format is ZIP. If Windows hides extensions, be careful: you can end up with `book.zip.epub` without realizing it. (A safer approach is to enable file extensions in Explorer settings.)

If you prefer a tool that’s common in developer workflows, [7-Zip](https://www.7-zip.org/) is excellent for inspecting EPUBs. It also makes it easier to see compression methods and file ordering when you need to debug packaging details.

If you’re comfortable with PowerShell, you can also extract an EPUB with the built-in archive command:

```powershell
New-Item -ItemType Directory -Force -Path .\book_unzipped | Out-Null
Expand-Archive -Path .\book.epub -DestinationPath .\book_unzipped -Force
```

One important practical note: “opening” a ZIP-like archive in a file manager and editing files inside it is usually a recipe for subtle breakage. Many tools will repackage the archive when you save, and they may not preserve EPUB-specific constraints like the `mimetype` rules. As a habit, extract to a folder, edit there, then re-zip using a deliberate packaging process.

## The required `mimetype` file (and why it has weird constraints)

Once you unzip a valid EPUB, the first thing you should notice is a file named `mimetype` at the root (not inside a folder). Its contents are usually just a single line:

```text
application/epub+zip
```

This file exists so that software can identify an EPUB container quickly and reliably. But EPUB goes further than “the file exists.” OCF requires two extra constraints that surprise new developers:

The `mimetype` file must be the **first file** in the ZIP archive.

The `mimetype` file must be **uncompressed** (stored, not deflated).

At first glance this looks like unnecessary ceremony. Why should the order of files in a ZIP matter? Why should one file not be compressed?

The reason is that reading systems and other tooling sometimes want to identify the file type without reading (or decompressing) the entire archive. If the `mimetype` entry is first and stored, a tool can read a small fixed part of the ZIP structure, retrieve that entry’s content cheaply, and decide “yes, this is an EPUB.”

From an engineering perspective, it’s similar to having a magic number at the start of a binary format, except in this case the “magic” is a tiny file inside a container.

Because this is a packaging rule, it can be broken accidentally by normal ZIP tools. A file can be “correct on disk” but “wrong in the archive.” That’s why EPUB validation tooling often complains about `mimetype` even when the extracted file content looks fine.

## How to re-zip an EPUB without breaking `mimetype`

If you ever need to rebuild an EPUB from an extracted folder, the key idea is: **add `mimetype` first, without compression, then add everything else normally**.

On macOS and Linux, if your extracted files are in `book_unzipped/` and the `mimetype` file is at the root of that folder, you can produce a new EPUB like this:

```bash
cd book_unzipped

# Create the archive and add mimetype first, stored (no compression)
zip -X0 "../book_fixed.epub" mimetype

# Add the rest of the EPUB with normal compression
zip -Xr9D "../book_fixed.epub" META-INF OEBPS
```

There’s a lot of “zip flag lore” floating around, so here’s the intent in plain language.

The `-0` (or `-X0` in the combined form above) means “store this file without compression,” which is what OCF requires for `mimetype`.

The `-X` flag strips extra filesystem metadata that some tools include by default. This helps you avoid platform-specific junk inside the archive.

The `-r` flag adds folders recursively, and `-9` uses stronger compression for everything except `mimetype`.

The `-D` flag avoids adding separate directory entries. Many EPUBs work fine without it, but it can make archives a little cleaner and can help avoid weird corner cases in older tooling.

On Windows, the exact command depends on your tooling. The same principle applies with 7-Zip: add `mimetype` first with “store” compression, then add the remaining folders with normal compression. The detail that matters is the resulting ZIP entry order and compression method, not which UI you clicked.

## `META-INF/` and `container.xml`: how a reader finds the OPF

The other container-level requirement is the `META-INF/` directory. Inside it, the most important file is `container.xml`.

When you’re new to EPUB, you might assume the package document lives at a fixed path like `OEBPS/content.opf`. Many EPUBs do follow that convention, but it’s not required. Instead, the reading system finds the OPF by reading `META-INF/container.xml`.

Here is a minimal, realistic `container.xml`:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<container version="1.0"
  xmlns="urn:oasis:names:tc:opendocument:xmlns:container">
  <rootfiles>
    <rootfile
      full-path="OEBPS/content.opf"
      media-type="application/oebps-package+xml"/>
  </rootfiles>
</container>
```

There are three ideas to take from this:

The root element is `<container>`, and it uses a namespace. (Namespaces are a normal XML detail you’ll see throughout EPUB.)

The `<rootfiles>` element contains one or more `<rootfile>` entries. Most EPUBs have exactly one, but the format allows multiple.

The `full-path` attribute is the key. It is the authoritative location of the OPF package document inside the ZIP.

You can treat `container.xml` as the container’s “index file.” It’s the bridge between “this is a ZIP of files” and “this is a publication with a defined structure.”

## Walkthrough: unzip a real EPUB and follow the chain of pointers

Let’s do a mental walkthrough of a very common EPUB layout. After extraction, you might see something like this:

```text
book_unzipped/
  mimetype
  META-INF/
    container.xml
  OEBPS/
    content.opf
    nav.xhtml
    toc.ncx
    text/
      titlepage.xhtml
      chapter-01.xhtml
      chapter-02.xhtml
    styles/
      book.css
    images/
      cover.jpg
      figure-01.png
```

Now follow the chain the way a reading system does.

It starts at the container root. It expects `mimetype` and `META-INF/container.xml` to exist. If either is missing, the file might not even be treated as an EPUB.

Then it reads `META-INF/container.xml` to discover the OPF path. In this example, `full-path="OEBPS/content.opf"` says: the package document lives at `OEBPS/content.opf`.

Then it opens the OPF. The OPF lists every resource that matters to the publication (manifest) and defines the reading order (spine). It also identifies navigation resources like `nav.xhtml` (EPUB 3) and sometimes `toc.ncx` (EPUB 2 compatibility). We’ll unpack those details in Chapters 2.4 and 2.5, but the core concept is: **the OPF is where the “book-ness” begins**.

Finally, when the OPF points to content documents like `text/chapter-01.xhtml`, the reading system loads those XHTML files and applies referenced stylesheets like `styles/book.css`. At that point, you’re in “web content land,” with EPUB-specific constraints around XHTML correctness and reader support.

The highest-value mental model is this three-step pointer chain:

`container.xml` points to the OPF, and the OPF points to everything else.

Once you internalize that, you can debug many issues quickly. If a reader can’t open the book, suspect the container layer. If a reader opens the book but chapters are missing, suspect the OPF manifest/spine. If text looks wrong, suspect the XHTML/CSS layer and reader limitations.

## Common pitfalls and gotchas

**Breaking the `mimetype` constraints** is the classic packaging failure. The content of the file might be correct, but if it isn’t the first entry in the ZIP or if it is compressed, validators will complain and some reading systems will refuse the book. This is one reason EPUB packaging is usually done by specialized tooling rather than “whatever ZIP app is installed.”

**A hidden UTF-8 BOM or extra whitespace in `mimetype`** can trip up strict tooling. The `mimetype` file should effectively be plain ASCII text containing `application/epub+zip`. If an editor inserts a byte order mark (BOM) at the start, or if conversion tooling introduces odd whitespace, you can end up with a file that looks correct when viewed casually but fails an exact check.

**Accidentally adding extra files when repackaging** can also cause problems. On macOS, it’s easy to introduce metadata artifacts like `.DS_Store` or `__MACOSX/` when zipping a folder from Finder. Some reading systems ignore these. Some don’t care but EPUBCheck may warn. If you want predictable output, use a deliberate packaging step rather than a drag-and-drop ZIP workflow.

**Zipping the wrong folder level** is another common mistake. If you zip the containing directory instead of the directory’s contents, you’ll end up with an EPUB where everything is under an extra top-level folder (for example, `book_unzipped/OEBPS/...` becomes `SomeFolder/OEBPS/...` inside the ZIP). That shifts paths and can break expectations. A reading system looks for `mimetype` and `META-INF/` at the root of the archive, not one level down.

**Assuming the OPF lives at a fixed path** will eventually break your tooling. Some EPUBs use `OEBPS/content.opf`, some use `EPUB/package.opf`, some put it at the root, and some use layouts that reflect publisher pipelines. The only portable approach is to start at `META-INF/container.xml` and follow `full-path`.

**Case sensitivity can bite you** when moving between operating systems. Many macOS and Windows setups use case-insensitive filesystems by default, while many Linux systems are case-sensitive. Inside an EPUB, a reference to `Images/Cover.jpg` will not match `images/cover.jpg` on a strict system. Some readers run on platforms that behave more like Linux than like your local dev machine. When you see “it works here but not there,” case mismatches are a frequent culprit.

**The gap between “opens” and “is valid” is real.** Some reading systems are forgiving. Others are strict. A broken container might still open in one app and fail in another. When you’re producing EPUBs professionally (or building a tool that manipulates them), you should treat validation as part of your development loop, not an optional final step. Chapter 4.2 will give you the practical workflow for that, but the key takeaway here is simple: being “just a ZIP” makes EPUB approachable, and being “ZIP + constraints” makes it reliable.
