# 5.2 Tools for Creating and Inspecting EPUBs

If you are new to EPUBs, the hardest part is not learning what an OPF file is or why XHTML has to be well-formed. The hardest part is building a workflow where you can reliably create an EPUB, inspect what is actually inside it, validate it, and iterate without guesswork. Tools matter because EPUBs are “web content in a container,” and you often need both kinds of tooling: something that understands books (spines, manifests, navigation) and something that understands the web (HTML, CSS, images, fonts).

This chapter is a practical tour of the most common tools you will encounter for creating and inspecting EPUBs. The goal is not to turn you into a power user of every tool. The goal is to give you a mental map: which tools are good at which parts of the job, where they fit in a workflow, and how to avoid being fooled by a tool preview that does not match the real world.

## Key vocabulary (for this chapter)

A **WYSIWYG editor** (“what you see is what you get”) is an editor that lets you edit content visually rather than directly editing XHTML and CSS. A **source editor** is an editor where you directly edit the EPUB’s files (XHTML, OPF, CSS) as text. A **conversion tool** takes one format (like Markdown, Word, or HTML) and generates an EPUB. A **round-trip** is when you convert to EPUB, edit the EPUB, then convert back or regenerate and hope changes survive; in EPUB workflows, round-trips are often fragile. An **inspector** is any tool or technique that lets you open an EPUB and see its internal structure (manifest, spine, navigation, assets) rather than only the rendered pages. A **validator** checks your EPUB against rules from the spec (and practical ecosystem rules) and reports errors and warnings. A **toolchain** is your repeatable end-to-end workflow: “source format → build/export → validate → inspect → test in readers.”

## A mental model: authoring tools versus inspection tools

It helps to separate tools into two categories.

Authoring tools are what you use to create or edit content: WYSIWYG EPUB editors, conversion tools, layout/design software, or your own code. These tools focus on productivity: “how do I get a book made?”

Inspection tools are what you use to understand what you actually produced: unzipping the EPUB, checking the OPF, verifying that the nav document matches the spine, seeing which CSS files are linked, checking that images and fonts exist, and running validators. These tools focus on correctness and predictability: “what is really in the artifact I am shipping?”

In practice, you need both. Many EPUB problems happen because authors rely on a single authoring tool’s preview, never inspect the internal structure, and only discover issues when a distributor rejects the file or a particular reading app behaves differently.

## Sigil: a practical EPUB-first editor

Sigil is an open-source EPUB editor that is “book aware.” It understands that an EPUB is not just a directory of HTML files: it has an OPF package document, a spine, a table of contents, and metadata. That makes Sigil useful when you want to work at the EPUB layer directly.

Sigil’s superpower is that it makes the internal structure visible and editable. You can open an EPUB and quickly see content documents, stylesheets, images, and metadata. You can edit XHTML directly, fix broken links, adjust CSS, and update the table of contents. For a junior developer, Sigil is often the fastest way to connect the concepts from Part II (OPF, manifest, spine, nav) to a real file you can click around in.

Sigil is also a good “repair shop” tool. A common workflow is to generate an EPUB from another system (for example, export from a writing tool, or convert from Markdown), then use Sigil to clean up the output: normalize heading structure, fix nav document problems, remove unused files, correct metadata, and make sure links and IDs behave consistently.

The limitation to keep in mind is that Sigil’s preview is not a promise of how every reading system will render your book. Sigil can help you author and inspect, but the final truth still comes from validation and testing in representative readers (Chapter 5.1). Treat Sigil as a high-leverage editor, not as your only rendering oracle.

## Calibre: library management plus powerful conversion (and an editor)

Calibre is best known as an e-book library manager, but for EPUB workflows it is often used for three things: conversion, inspection, and quick viewing.

First, Calibre’s conversion pipeline can take many inputs (including some formats you might encounter in the wild) and emit an EPUB. This is useful when you inherit content from a publisher, a client, or an older toolchain and you need to get it into “EPUB world” quickly. The trade-off is that conversion is never perfect. Calibre can get you a usable baseline, but you should expect to do cleanup afterward, especially around semantic structure, heading hierarchy, and CSS.

Second, Calibre includes an EPUB editor. Like Sigil, it can open the internal file structure. Many people use Calibre’s editor for targeted fixes and for its search/replace capabilities across the book. If you already use Calibre to manage and view books, its editor can be a convenient “one app” choice.

Third, Calibre’s viewer is a quick way to open an EPUB and sanity-check that it basically reads. It is not the same as testing on mobile or e-ink, but it is a fast feedback loop during development. The right way to use Calibre viewer is as a “smoke test”: does the book open, does navigation exist, do images appear, do the pages look roughly correct?

As with Sigil, the key limitation is that a desktop viewer is not a universal rendering standard. Calibre is great for iterating quickly, but you should still validate and test in the reading systems that matter for your audience.

## Pandoc: programmatic conversion from Markdown/HTML/DOCX into EPUB

Pandoc is a format conversion tool that appeals to developers because it is scriptable and repeatable. If your source material is Markdown, HTML, or Word documents, Pandoc can produce an EPUB as an automated build step. That makes it a good fit for “docs-like” books, technical writing, and workflows where you want to keep the source in version control and generate release artifacts deterministically.

Here is a minimal example of converting Markdown to an EPUB:

```bash
pandoc book.md -o book.epub
```

That one line can be surprisingly productive, but it is also where many people get stuck. A good EPUB is not just “text converted to XHTML.” You will almost always care about metadata, a cover image, a stylesheet, and consistent structure across chapters. Pandoc supports many of those concerns through flags and templates, which is powerful but can feel like a small build system.

The most important mindset with Pandoc is to treat your output as generated code. You usually do not want to hand-edit the EPUB after Pandoc generates it, because you will eventually regenerate and lose your edits. Instead, you adjust the source documents and the Pandoc options (or templates) so the next build produces the right EPUB. This is the difference between a fragile workflow (“manual EPUB cleanup forever”) and a robust workflow (“fix the generator once”).

Pandoc also makes it easy to adopt a “build pipeline” discipline. You can run conversion, then run validation, then package and test. If you are trying to create a repeatable release process (the theme of Chapter 4.3), Pandoc is one of the cleanest ways to do it when your inputs are text-centric.

## Adobe InDesign: industry-standard layout, with EPUB export trade-offs

InDesign is widely used in traditional publishing because it is extremely strong at layout and typography for print-like workflows. It can export EPUB, and in many teams that is “how the EPUB gets made” even when the final product is a reflowable e-book.

The practical reality is that InDesign’s export is often a starting point rather than a final artifact. InDesign is fundamentally a layout tool, and reflowable EPUB is fundamentally a structure-first format. When a layout-centric system exports into a structure-centric format, you can get markup that is technically valid but semantically messy: too many nested spans, CSS that encodes visual intent rather than meaning, headings that are styled paragraphs instead of real heading elements, or page-break hacks that make sense in print but not in reflow.

If your workflow starts in InDesign, a common approach is to export, then use an EPUB editor (Sigil or Calibre editor) as a cleanup stage. You focus cleanup on a few high-value areas: correct heading structure, sane CSS, navigation that matches chapters, correct cover metadata, and removal of unused or duplicated assets. The goal is not to “perfectly hand-author everything.” The goal is to transform the export into a resilient EPUB that behaves predictably across reading systems.

It is also worth being honest about what InDesign is best at. If you truly need pixel-level control and the book is more like a designed page than a flowing text, you may be in fixed-layout territory (Chapter 3.6). InDesign can be appropriate there, but the compatibility and accessibility trade-offs become more severe, and you should test aggressively.

## Programmatic generation: building EPUBs with code (and why it can be worth it)

There is a class of EPUB workflows where you do not “author an EPUB” so much as you generate one from structured data. This is common in documentation systems, learning platforms, content management systems, and any product that needs to produce many books with consistent structure.

In Python, a commonly referenced library is `ebooklib`. In JavaScript and other ecosystems you will find similar libraries, or teams build custom generators. The details vary, but the benefits tend to be the same: consistency, repeatability, and the ability to enforce best practices automatically. For example, a generator can ensure every content document has a consistent XHTML boilerplate, every file is included in the manifest with the correct media type, the nav document matches the spine, and internal links are generated correctly.

The trade-off is that you are now building software, not just a book. You need tests (at least “does it validate?”), you need a packaging step that obeys container rules, and you need an inspection workflow for when something goes wrong. The good news is that EPUB is a file format made of plain text and predictable assets. That makes it a surprisingly good target for automated generation once you understand the structure.

If you go down this path, you should design your generator so that the source of truth is not “the EPUB file.” The source of truth should be your structured content and your templates, and the EPUB should be a build artifact that can be regenerated at will. This mirrors the Pandoc mindset, but with more control.

## Inspection fundamentals: unzip, read the OPF, and treat the EPUB like a web project

Even if you never write a line of EPUB generation code, you will become much faster at debugging if you get comfortable inspecting EPUBs directly.

The simplest inspection workflow is to unzip the EPUB and look at the internal files. This connects directly to Chapter 2.1: an EPUB is a ZIP archive, and reading it “from the inside” is often the fastest way to understand what a tool produced. You do not need special magic for this; you need the habit of opening the OPF and confirming the manifest and spine, then opening the nav document (or NCX) and confirming navigation, then checking that referenced files exist and paths match exactly.

Because the content documents are XHTML, many web debugging instincts carry over. You can open an XHTML file in a browser to check whether it is well-formed and whether your CSS selectors behave. You can use browser developer tools to inspect computed styles and find out why a particular element is not picking up a rule. This does not fully simulate an EPUB reading system (Chapter 5.1 explains why), but it is excellent for debugging your own markup and CSS logic.

The biggest mental shift is remembering that the EPUB container adds extra constraints. An image can exist in the folder and still fail in a reader if it is not in the OPF manifest or if the media type is wrong. A link can work in a browser but fail in a reader because of path base differences or because the referenced file is not a spine item. Inspection is about catching those “EPUB-specific” problems early.

## Reader apps as inspection tools: use them as “rendering probes”

It can feel strange to call a reading app a “tool,” but in EPUB work, real reading systems are some of your most important inspectors. A desktop reader like Thorium Reader (Readium-based) can act as a modern baseline viewer. Adobe Digital Editions is still common in library and publishing workflows and can reveal different classes of issues. Apple Books and Google Play Books matter because they reflect major consumer ecosystems.

The right way to use these apps during development is as rendering probes. You are not trying to perfectly tune the book for one app. You are checking whether the structure and assets you shipped behave in the ecosystem: does the table of contents appear, does the cover show up in the library view, do internal links work, do fonts load when allowed, and do common user overrides still produce a readable book?

## Validation tools: use them as part of your build, not as a last resort

While this chapter focuses on creation and inspection, it is hard to talk about tools without emphasizing validation as a habit. A validator is your first line of defense against the class of problems that are invisible in a visual preview but fatal in the ecosystem.

The most important validator is EPUBCheck, which is the de facto standard for structural validation. Accessibility-specific checking is often done with Ace by DAISY (Chapter 4.1), and many teams run both. The key point here is workflow: you want to validate early and often, ideally after every export or generation step, and especially on the final `.epub` artifact you plan to ship (not just an unpacked folder).

If you are using validators from the command line, the actual invocation is usually simple. EPUBCheck is commonly run as a single command against the final `.epub` file:

```bash
epubcheck book.epub
```

Ace workflows vary, but the core idea is the same: run it on the final EPUB artifact and treat the report as something you fix systematically rather than something you skim once. Even if you use a GUI wrapper, you want the habit of “export/generate → validate → fix → repeat” to be part of your normal toolchain.

If you treat validation as a late step, error messages will feel mysterious because you are no longer close to the edit that caused them. If you validate continuously, error messages become actionable: “I just changed the OPF; this error is probably about that change.” This is the same discipline you use in software development when you run tests frequently rather than only at the end.

## Choosing the right tool for your workflow

The right tool is less about “what is best” and more about “what is your source of truth.”

If your source is already EPUB and you need to fix or inspect it, an EPUB-first editor like Sigil (or Calibre’s editor) is usually the fastest path because it speaks the format’s native concepts. If your source is Markdown or structured text and you care about reproducible builds, Pandoc (or a custom generator) tends to be a better foundation than manual editing. If your source is a layout-driven design file and your organization is already committed to that workflow, InDesign export may be unavoidable, but you should plan for a cleanup stage and treat the export as generated output rather than as your final hand-authored product.

In all cases, your workflow becomes dramatically more reliable when you separate “creation” from “inspection.” Create with whatever tool fits your organization, but inspect and validate with a consistent, repeatable set of steps: unzip when needed, read the OPF and nav, validate, then test in representative readers.

## Common pitfalls and gotchas

The first gotcha is assuming that a visual preview in an editor is equivalent to how real reading systems behave. Previews are useful, but they run in a specific engine with specific defaults. A book can “look fine” in a tool preview and still fail validation, fail ingestion, or render differently on a phone or e-ink device.

The second gotcha is unintentional round-tripping. If you generate an EPUB from Pandoc, then hand-edit the EPUB in Sigil, then regenerate from Pandoc later, you will overwrite your fixes unless you pushed those fixes back into the generator inputs or templates. This is one of the most common sources of “why did this bug come back?” in EPUB production. Decide what your source of truth is, and make sure fixes land there.

The third gotcha is using conversion tools as if they produce semantically clean documents by default. Converters can be excellent, but they often produce markup that is technically correct yet structurally noisy. If the output is full of spans and inline styles, your CSS becomes harder to reason about, and accessibility can suffer because visual intent is encoded instead of meaning. When you inherit converted output, prioritize cleanup of headings, lists, and landmarks first, because those drive navigation and readability.

The fourth gotcha is treating EPUB problems as “CSS bugs” when they are actually packaging or OPF problems. Missing images, missing fonts, or stylesheets that do not load are often caused by incorrect manifest entries, wrong media types, or path mismatches. Inspection (unzip, check OPF, confirm paths) is usually faster than tweaking CSS at random.

The fifth gotcha is tool lock-in. Every major tool has a slightly different way of representing metadata, navigation, and cover conventions. If you switch tools late in a project, you can accidentally change how the book is packaged or how the TOC is generated. If you do switch, validate the final artifact again and re-check the specific behaviors you care about: cover display, TOC structure, internal links, and embedded font behavior.

Once you treat tools as interchangeable parts in a deliberate toolchain—create, inspect, validate, test—you stop depending on any single app’s “magic.” That is the point where EPUB development starts to feel like engineering rather than trial and error.
