# 4.3 Export and Packaging Best Practices

By the time you are ready to “export an EPUB,” most of your work is already done. Your chapters are written as XHTML content documents, your CSS is in place, your images and fonts are included, and your OPF ties it all together. Export and packaging is the final step that turns that folder of files into a single `.epub` artifact that other tools can validate and real reading systems can open reliably.

This step sounds mundane, but it is where a lot of production EPUBs fail in ways that feel unfair: everything looks correct in your editor, but the book refuses to open on a particular device, or a distributor rejects it, or a Kindle conversion pipeline behaves oddly. In those moments, the issue is often not your HTML or CSS at all. The issue is the container: the way you zipped, which files you included, and whether the package is clean and consistent.

In this chapter, you will learn what “good packaging” means in practice, how to build a clean EPUB archive, and how to treat export as a repeatable release step rather than a one-off “zip the folder” action.

## Key vocabulary (for this chapter)

**Packaging** is the act of taking the EPUB’s internal directory structure and turning it into a single EPUB container file (a ZIP archive with specific rules). **OCF** (Open Container Format) is the container layer of the EPUB standard: it defines the ZIP-based rules, the `mimetype` file requirement, and the `META-INF` directory. **Deflate** is the common ZIP compression method used for most files; **stored** (sometimes called “store method”) means the file is included with no compression. The EPUB `mimetype` file must be **stored** (uncompressed) and must be the **first entry** in the ZIP. **Manifest generation** is the process of ensuring that every resource file in your EPUB is declared in the OPF manifest with the correct media type. A **directory structure convention** is the set of predictable folders and naming patterns you use inside the EPUB so references are stable and understandable. A **release artifact** is the final `.epub` file you ship to a store, distributor, or customer.

## Why “export” is not the same as “zipping a folder”

If you come from web development, it is tempting to treat EPUB export as “bundle the assets and ship.” That mindset is close, but EPUB adds a strict container layer on top of the files. A reading system does not scan your ZIP archive arbitrarily and try to guess where your book starts. It expects a very specific chain of discovery: the archive must declare itself as an EPUB, `META-INF/container.xml` must point to the OPF package document, and the OPF must inventory every resource and define a reading order.

That chain is why packaging mistakes can be catastrophic. A small ZIP detail can prevent a reading system from even recognizing the file as an EPUB. A missing manifest item can cause a stylesheet or image to silently disappear in one app but not another. And hidden extra files can make your EPUB “messy” in ways that confuse ingestion pipelines.

The goal of best-practice export is to produce an EPUB that is (1) structurally correct by spec, (2) conservative and compatible in the ecosystem, and (3) reproducible: you can run the same export process tomorrow and get the same quality result.

## Build a clean, boring internal structure

EPUB does not require a specific folder layout for your content, but a predictable structure makes your life easier and reduces mistakes. The most common pattern is to keep content documents, stylesheets, images, and fonts in separate subfolders under a single “content root” folder that the OPF lives in.

For example, you might have an EPUB where the OPF is at `OEBPS/content.opf`, and the resources are organized like this: `OEBPS/text/` for XHTML, `OEBPS/styles/` for CSS, `OEBPS/images/` for raster and SVG images, and `OEBPS/fonts/` for embedded fonts. The exact folder names do not matter; what matters is that you commit to a convention and apply it consistently. When you later need to debug a missing image, you want that path to be obvious and stable.

Naming is also part of “structure.” A surprisingly large number of EPUB issues come from file names rather than markup. The safest approach is to use lowercase ASCII file names, avoid spaces, avoid punctuation that has special meaning in URLs, and keep names short. `chapter-01.xhtml` is rarely a problem. `Chapter 1 (Final!).xhtml` is an invitation for quoting issues, URL encoding issues, and case-sensitivity surprises when you move between operating systems.

The other “cleanliness” rule is to avoid shipping accidental files. The most common culprits are OS-generated metadata files (like `.DS_Store` on macOS), temporary backups, and editor swap files. These can sneak into the archive if you zip a folder without filtering. They usually do not break the EPUB spec by themselves, but they make the package harder to reason about and can trigger warnings or store ingestion quirks. Treat your EPUB content directory as a build input, not a junk drawer.

## The `mimetype` file: tiny, strict, and non-negotiable

If you remember only one packaging rule, remember this: the `mimetype` file must be the first file in the ZIP archive and it must not be compressed.

The contents of the `mimetype` file are simple. It is plain text containing exactly the MIME type string for an EPUB:

```text
application/epub+zip
```

The reason it is strict is that some reading systems and distribution pipelines use it as a fast signature check. Because it must be uncompressed and first, those systems can read a small fixed portion of the file and quickly decide “this is an EPUB container” without doing a full ZIP scan. If you compress it, or if it appears later in the archive, a strict system may reject the file even though everything else inside is perfect.

This is also why “right-click → Compress” in a file manager is risky. Many generic ZIP workflows compress everything and do not preserve a deliberate file entry order. For EPUBs, you want a packaging step that explicitly controls compression and ordering.

## A reliable command-line packaging pattern (and what it does)

One of the most common “production-grade” ways to package an EPUB is to build the ZIP in two steps: add `mimetype` first without compression, then add the rest of the directories with normal compression. Here is a typical pattern:

```bash
cd path/to/epub-root
zip -X0 ../book.epub mimetype
zip -Xr9D ../book.epub META-INF OEBPS
```

The first command creates (or updates) `book.epub` by adding the `mimetype` file with no compression (`-0`). The second command recursively adds the rest of the EPUB contents. The flags vary across workflows, but the intent is consistent.

The `-X` flag tells `zip` to exclude extra file attributes that can differ between machines. This is not about correctness in the EPUB spec so much as portability and reproducibility. If you are working on macOS, extra metadata can appear in the archive in ways that make “the same book” look different across builds, and sometimes that metadata can confuse older tooling. You will see many teams use `-X` as a general “keep it clean” habit.

The `-r` flag means recursive: include everything in those directories. The `-9` flag asks for maximum compression. Compression does not usually matter much for text-heavy EPUBs compared to image-heavy ones, but it is a reasonable default for a release artifact.

The `-D` flag suppresses directory entries in the ZIP. This is one of those pragmatic flags you will see in real-world packaging scripts because it can avoid odd edge cases in older readers and keep the ZIP structure simpler. Whether you need it depends on your toolchain, but the broader idea is important: packaging is not purely about “it unzips”; packaging is about what reading systems in the wild will accept.

If you prefer not to memorize flags, the best long-term approach is to use a known-good packaging script (or the packaging function of a mature EPUB tool) and treat it as part of your build pipeline. What matters is that your output obeys the `mimetype` rule and that you do not introduce platform-specific junk.

## Be explicit about what files belong to the publication

Inside an EPUB, “exists in the ZIP” is not the same as “belongs to the book.” The OPF manifest is the publication’s official inventory. Reading systems often use it to decide what can be loaded and how it should be interpreted. This is why packaging best practices are closely tied to manifest best practices: you want the ZIP and the OPF to describe the same world.

When you add or remove a resource, you should think in two steps. First, does the file exist in the folder you plan to package? Second, is the file declared in the OPF manifest with the correct `href` and `media-type`? If either step is wrong, you can end up with unpredictable behavior.

Here is a minimal example of what an OPF manifest item looks like:

```xml
<item id="chapter01" href="text/chapter-01.xhtml" media-type="application/xhtml+xml" />
```

The `href` is relative to the OPF file’s location. That detail matters during export. If you move your OPF file or reorganize directories, your manifest `href` values may need to change even if your XHTML file references do not. This “two different bases for paths” is one of the most common sources of broken packages when people restructure projects late in the process.

If you are generating EPUBs programmatically, manifest generation becomes a first-class packaging concern. A robust export pipeline usually walks the content directory, selects “real publication resources,” assigns deterministic IDs, and emits manifest entries with accurate media types. It is better to generate the manifest from the actual file set than to maintain it manually in a large project, because manual maintenance tends to drift.

## Strip what you do not want to ship (and why it matters)

In production, export is often also a cleanup step. During authoring, you may have intermediate files, drafts, or unused images sitting in the project folder. If your packaging step blindly zips the entire directory tree, you can end up shipping megabytes of unused assets. That increases file size, slows downloads, and makes store ingestion and device performance worse for no reader-visible benefit.

Even more importantly, shipping “extra stuff” increases risk. Some ingestion systems scan the archive and apply heuristics or security checks. Some reading systems have bugs triggered by unusual file sets. The safest EPUB is the one that contains only what the OPF declares and what your content actually uses.

You can think of this as the publishing equivalent of a production web build that removes source maps, dev-only assets, and temporary files. It is not about hiding anything; it is about shipping a minimal, coherent artifact.

## Optimize assets before packaging, not after

Export is a good moment to make sure your assets are appropriate for the medium. EPUB readers run on everything from high-end tablets to low-power e-ink devices. A huge PNG that looks fine on your laptop can make page turns sluggish on an older device.

For images, “right size and right format” is the main principle. Photos are usually best as JPEG. Flat-color illustrations or UI-like graphics often compress better as PNG. Diagrams that benefit from scalability can work well as SVG, but SVG support varies, so you should test and keep fallbacks in mind when your audience includes older readers.

For fonts, the main concerns are licensing and necessity. Embedded fonts can add a lot of weight. If you only need a font for a small portion of the book (like a title treatment), you might be better off using system fonts or limiting font embedding to the scripts and glyph ranges you truly need, depending on the font’s licensing terms and the font format.

The reason to do this before packaging is practical: the OPF manifest and your CSS will point to the specific files you ship. If you change assets after packaging, you can easily drift into a mismatch between “files in the ZIP” and “files referenced by the OPF and XHTML.” Asset optimization is part of building a coherent publication, not a post-processing trick.

## Treat export as a release pipeline: validate, package, validate again

Chapter 4.2 emphasized the validate/unpack/fix loop. Export is where you apply that mindset to the final artifact. A common mistake is to validate an unpacked folder or a draft EPUB and then assume the final packaged EPUB is equivalent. In reality, the packaging step itself can introduce problems, especially around the `mimetype` rule, accidental extra files, or incorrect path casing.

A robust workflow validates the final `.epub` output, not just the source folder. It also validates after every major packaging change. If you switch tools (for example, from an editor’s “export EPUB” feature to a custom script), you should assume the packaging behavior changed and revalidate. Think of validation as an invariant you keep re-checking, not a one-time gate you passed once.

## Test across multiple readers before you publish

Export and packaging is also the moment to shift from “does this build?” to “does this read?” EPUB is an ecosystem format, and the ecosystem is messy. Two reading systems can interpret the same valid EPUB differently, especially with CSS, embedded fonts, and advanced navigation features.

The practical goal is not to test on every device; it is to test on representative engines. A good baseline is to test in a Readium-based reader (because Readium is widely used as a foundation) and at least one major consumer ecosystem that matches your distribution goals. If you care about Apple Books, test there. If you care about Google Play Books, test there. If your audience is heavy on dedicated e-ink devices, test with a Kobo or a comparable app.

You should also remember that “Kindle support” usually means “conversion support.” Amazon’s ecosystem typically converts your uploaded book into its own internal formats. That means your EPUB packaging quality still matters, because conversion pipelines behave better on clean, spec-conformant inputs, but you should also test the converted output if Amazon is a target. Export is not only about shipping an EPUB; it is about shipping an EPUB that converts predictably.

## A note on determinism and versioning

Once you start shipping EPUBs to stores or clients, you will care about repeatability. If someone reports a bug, you want to know exactly which EPUB artifact they used. That is harder than it sounds if your packaging step produces slightly different ZIP metadata each time, or if you manually zip on different machines.

The best practice here is to put export under the same discipline you would put a software release under. Use a consistent toolchain. Generate the EPUB in the same way each time. Keep a version identifier in metadata where appropriate, and store the exact release artifact you shipped. Even if you are a team of one, this practice pays off the first time you need to reproduce a defect.

## Common pitfalls and gotchas

The first pitfall is assuming that a valid folder of EPUB files automatically becomes a valid `.epub` when zipped. The `mimetype` file rule makes EPUB packaging unlike generic ZIP packaging. If you zip the folder casually, you can end up with a book that looks normal on your machine but fails on a strict device or ingestion system. If a book suddenly breaks after “just repackaging,” suspect the archive order and compression of `mimetype` before you suspect your XHTML.

The second pitfall is shipping hidden OS files and editor artifacts. These usually do not improve anything, but they do increase noise and risk. They can also make debugging harder, because when you unpack the EPUB later, you have to mentally filter out junk before you can see the publication’s real structure.

The third pitfall is path inconsistency, especially case differences. It is easy to accidentally reference `Images/Cover.jpg` in one place and `images/cover.jpg` in another, and your development machine might not punish you for it. Some reading systems will. Export is a good time to normalize paths and names, and to treat “case-consistent naming” as part of correctness.

The fourth pitfall is drifting between “files in the ZIP” and “files in the OPF manifest.” It is possible to ship an EPUB that contains a resource but does not declare it properly, or declares a resource that is no longer present. This creates unpredictable behavior across readers. The cure is to treat manifest generation and packaging as linked steps: the archive should contain what the OPF declares, and the OPF should declare what the archive contains.

The fifth pitfall is believing that packaging is purely a technical concern and testing is purely a visual concern. In practice, packaging and testing are connected. Navigation documents, cover conventions, and resource declarations affect not just validity but how a book appears in a library view, how the table of contents is presented, and whether search and accessibility features work as expected.

The sixth pitfall is treating export as a one-time action rather than a repeatable pipeline. If your release process is “zip it somehow, hope it works,” you will eventually produce an EPUB that fails for a reason you cannot reproduce. A simple, scripted export step that you can run again and again is one of the highest-leverage improvements you can make once your EPUB projects grow beyond toy examples.

If you treat export as “build the cleanest possible container,” and you validate and test the final artifact the same way you validate and test any other release, packaging becomes boring. In EPUB production, boring is a feature.
