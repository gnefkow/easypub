# 3.2 Style and Formatting in EPUBs

If you know CSS from the web, you already understand most of how EPUB styling works. EPUB content documents are (usually) XHTML, and those XHTML files can link to CSS stylesheets in almost exactly the same way a web page does. The important difference is not the syntax, but the environment: there is no single “browser” for EPUB. There are many reading systems, each with its own rendering engine, its own default stylesheet, and—crucially—its own rules about how much control the user gets to override your choices.

This chapter will give you a practical mental model for EPUB styling: where CSS lives inside the ZIP, how it gets applied to content documents, how to think about reader overrides, and which patterns tend to survive across devices.

## Key vocabulary (for this chapter)

When you read EPUB styling discussions, you’ll see a few terms come up repeatedly. A **stylesheet** is a `.css` file packaged inside the EPUB and referenced by one or more content documents. The **cascade** is the set of rules CSS uses to decide which declarations “win” when multiple rules apply to the same element. A **reading system** is an EPUB app or device (Apple Books, Kobo, Thorium, Adobe Digital Editions, and many others) that loads the package and renders the content. A **user override** is when the reading system applies user preferences—font face, font size, margins, line spacing, night mode—on top of (or instead of) the publisher’s CSS. And a **page break** is a pagination hint you can express in CSS so chapters start on new pages or certain blocks don’t split awkwardly.

We’ll also talk about **inline styles** (a `style="..."` attribute on an element), **class-based styling** (using `class="..."` and a stylesheet rule like `.className { ... }`), and **@page** rules (special CSS used mostly in paginated contexts; support varies widely).

## Where styles live inside an EPUB

Most EPUBs keep stylesheets in a folder such as `styles/` or `css/`, but the folder name is just a convention. The only thing that matters is that the stylesheet file exists in the ZIP and is declared in the OPF manifest with the correct media type (`text/css`). If a stylesheet is not in the manifest, some reading systems will still load it, but you should treat that as luck rather than design.

In practice, you’ll often see a structure like this:

```text
OEBPS/
  content.opf
  text/
    chapter-01.xhtml
    chapter-02.xhtml
  styles/
    book.css
    fonts.css
  images/
    cover.jpg
```

The key idea is that your XHTML file and your CSS file are just two files in the same ZIP. Linking them is mostly about getting the relative path right and keeping the package metadata consistent.

## How stylesheets are linked from XHTML content documents

The most common and most reliable pattern is a `<link>` element in the document’s `<head>`. Here is a typical EPUB 3 content document head:

```xml
<head>
  <meta charset="utf-8" />
  <title>Chapter 1</title>
  <link rel="stylesheet" type="text/css" href="../styles/book.css" />
</head>
```

Let’s unpack what each piece means.

`<meta charset="utf-8" />` declares the character encoding. In EPUB content documents, you should assume UTF‑8. `<title>` is metadata for the document; many readers ignore it, but it’s still part of good hygiene. The `<link>` element is the important part: `rel="stylesheet"` means “treat the linked file as CSS,” and `href="../styles/book.css"` is a relative path from this XHTML file to the stylesheet inside the EPUB. The `type="text/css"` attribute is commonly included for compatibility, even though modern HTML doesn’t require it.

You will sometimes also see `@import` used inside CSS:

```css
@import url("fonts.css");

body {
  line-height: 1.4;
}
```

This can work, but it adds another layer of “will this reader load it the same way?” If you’re editing an EPUB, prefer explicit `<link>` tags unless you have a reason to keep imports.

## The biggest difference from the web: reader defaults and user overrides

On the web, you usually assume the author (you) controls the stylesheet and the user agent (the browser) applies a fairly consistent default base. In EPUB, reading systems tend to be much more opinionated, and they often put the user in charge. That means two things can be true at the same time:

First, your CSS can be perfectly valid and still not produce the result you expect on a particular device, because the reading system has its own default styles and layout decisions. Second, a user can intentionally override your choices, and the reading system will respect that override even if your stylesheet says otherwise.

This is why “publisher control” in EPUB is mostly about choosing what is essential for meaning (headings, emphasis, block quotes, poetry structure) and what is optional decoration (exact fonts, precise margins, pixel-perfect spacing).

For a junior developer, the most useful way to think about EPUB CSS is: you are proposing a default. A reading system may accept it, partially accept it, or replace it with the user’s preferred experience.

## Supported CSS: “it’s CSS, with caveats”

EPUB 3 is based on web technologies, but the real compatibility target is not “latest Chrome.” Many reading systems are based on older WebKit builds, custom engines, or constrained environments. In addition, some readers intentionally disable or limit certain features (for performance, security, or consistency).

You can still use CSS selectors, classes, inheritance, and the cascade in the normal way. But you should be conservative about advanced layout features unless you are prepared to test on multiple reading systems. As a general rule, typography and spacing rules tend to be safer than complex layout rules.

For example, rules like these are usually safe:

```css
body {
  margin: 0;
  padding: 0;
}

p {
  margin: 0 0 1em 0;
}

h1 {
  font-size: 1.6em;
  margin: 1.2em 0 0.6em 0;
}
```

They rely on basic units (`em`) and common properties (margin, font-size). Using `em` is helpful because it scales with the user’s chosen font size. If a user increases text size, the spacing scales too.

By contrast, modern features like multi-column layouts, complex flexbox grids, and aggressive positioning can be inconsistent. Sometimes they work beautifully; sometimes they break in ways that are hard to predict.

## Inline styles vs external stylesheets

Inline styles look tempting because they are local and obvious. For example:

```xml
<p style="text-align: center; margin-top: 2em;">* * *</p>
```

This will often render as intended, but inline styles have costs. They are harder to maintain at scale, they make your content documents noisier, and they can fight the user override system more aggressively than you want. Inline styles also make batch editing harder because you have to update many individual attributes instead of one rule in one stylesheet.

Class-based styling is usually the better long-term choice. The same formatting can be expressed like this:

```xml
<p class="scene-break">* * *</p>
```

And the stylesheet can define:

```css
.scene-break {
  text-align: center;
  margin-top: 2em;
  margin-bottom: 2em;
}
```

Now you can change the look of all scene breaks by editing one CSS rule. More importantly, you can keep your XHTML focused on structure and meaning, and keep “presentation knobs” in CSS where they belong.

## Pagination and page-level styling

Most readers paginate reflowable EPUBs into “screens” or “pages,” even though the content is not fixed-layout. CSS includes properties that can influence pagination, but support varies, and different reading systems interpret them differently.

Historically, EPUB 2-era CSS used `page-break-before` and `page-break-after`:

```css
h1 {
  page-break-before: always;
}
```

Newer CSS defines the `break-*` family (`break-before`, `break-after`, `break-inside`). Some reading systems support one set, some support both, and some treat them as hints rather than commands. If you are authoring or editing for broad compatibility, you’ll often see both:

```css
h1 {
  page-break-before: always;
  break-before: page;
}
```

The intent here is “start a new page before each chapter heading.” Whether that becomes a true “new page” depends on the reading system’s pagination model.

The `@page` rule is even more variable. It is designed for paged media (printing, some paginated engines), and it can express margins and other page box properties:

```css
@page {
  margin: 1.2em;
}
```

Some readers ignore this entirely. Others apply it only in certain modes. When you use `@page`, treat it as progressive enhancement.

## Common styling patterns that work well in EPUBs

The EPUB world has a set of “classic” formatting problems—drop caps, poetry, block quotes—that show up in many books. These patterns are common because they are expressible with simple markup and modest CSS.

### Block quotes

Semantically, a block quote should use the `<blockquote>` element:

```xml
<blockquote>
  <p>We are what we repeatedly do. Excellence, then, is not an act, but a habit.</p>
</blockquote>
```

The styling is usually just indentation and spacing:

```css
blockquote {
  margin: 1em 0;
  padding-left: 1.2em;
  border-left: 0.2em solid #999;
}

blockquote p {
  margin: 0;
}
```

Not every reader will render the border exactly the same way, but the meaning of the quote is preserved by the structural markup even if the styling is simplified.

### Centered text and ornamental breaks

Centered text is typically used for ornamental breaks, dedication pages, or short epigraphs. The EPUB-friendly approach is a class:

```xml
<p class="centered">For my family.</p>
```

```css
.centered {
  text-align: center;
}
```

If a reader overrides fonts and margins, centered alignment usually remains centered, which is why it’s a good “low-risk” styling choice.

### Drop caps (with realistic expectations)

Drop caps are a classic “looks great when supported, degrades acceptably when not” feature. One common pattern is to wrap the first letter in a `span`:

```xml
<p><span class="dropcap">T</span>his is the first paragraph of the chapter.</p>
```

And then style it:

```css
.dropcap {
  float: left;
  font-size: 3em;
  line-height: 1;
  padding-right: 0.1em;
}
```

Some reading systems handle floats well; others can be quirky, especially with large fonts and narrow screens. The important thing is that if the float is ignored, the text still reads correctly. That is the EPUB mindset: make the meaning survive even when the decoration does not.

### Poetry and line breaks

Poetry is tricky because whitespace matters, and EPUB content documents are XHTML, which collapses whitespace like HTML. The typical approach is to represent each line explicitly, either with `<br />` or with separate elements.

A simple pattern is:

```xml
<p class="poem">
  Two roads diverged in a yellow wood,<br />
  And sorry I could not travel both<br />
  And be one traveler, long I stood<br />
</p>
```

And then style the block to indent or adjust spacing:

```css
.poem {
  margin-left: 1.5em;
  margin-right: 1.5em;
}
```

Another approach is to use `<pre>` for preformatted text, but `<pre>` can produce awkward wrapping on small screens and may interact poorly with user font overrides. The `<br />` approach is common because it is explicit and simple.

## A note on “pixel-perfect layout” in reflowable EPUBs

Reflowable EPUBs are not a canvas. The user can change the font size, rotate the device, switch to night mode, increase line spacing, or choose a dyslexic-friendly font. All of those changes reflow the content. That means layout techniques that assume a fixed width or fixed line length tend to break.

As you edit EPUBs—especially with an app like EasyPub that lets you split, merge, and reorganize content documents—you get the best results when you separate structure from presentation. Use semantic elements and headings for structure, use classes for styling hooks, and keep CSS focused on typography rather than rigid positioning.

## Common pitfalls and gotchas

The gap between “the spec allows it” and “real readers support it” is where most EPUB styling pain lives.

The first gotcha is assuming your CSS will always win. Many reading systems allow users to override fonts and sometimes margins and line spacing. If a user chooses a different font, `font-family` declarations can be ignored. If a user chooses “publisher fonts,” they may be respected. If a user chooses “system fonts,” they may not. Your job as an editor is to make sure the content still looks good and remains readable under both.

The second gotcha is relying on advanced layout features without testing. CSS support differs widely between Apple Books, Kobo, ADE, Google Play Books, and open-source Readium-based readers. Even when a feature “works,” it may have edge cases (tables overflowing, floats overlapping, hyphenation behaving unexpectedly). In EPUB work, conservative CSS is not a sign of low skill; it is a sign you are optimizing for the real ecosystem.

The third gotcha is breaking your own links. Stylesheets are referenced by relative paths. If you move a chapter file from `text/` to `Text/`, or you rename `styles/book.css` to `styles/main.css`, you must update every referencing `<link>` and also update the OPF manifest entry. Case sensitivity matters on many systems, and what “works” on your machine can fail on a device.

The fourth gotcha is overusing inline styles. Inline styles make it harder to maintain consistency across a book, and they can create messy interactions with user overrides. If you find yourself repeating the same `style="..."` attribute over and over, that is a strong signal to introduce a class and put the styling in the stylesheet.

The fifth gotcha is whitespace and XHTML correctness. Because EPUB content documents are typically XHTML, mistakes like `<br>` instead of `<br />` or an unescaped `&` can turn a styling tweak into a validation failure. Styling work in EPUBs is never “just CSS”; it always touches the XHTML/XML rules of the container.

Finally, be careful with fonts and dark mode. Some books hardcode colors like `color: #000` and `background: #fff`. That can make night mode unreadable. A safer approach is to avoid forcing foreground/background colors in the main reading text unless you have a strong reason, and to test in both light and dark themes when your target readers support them.
