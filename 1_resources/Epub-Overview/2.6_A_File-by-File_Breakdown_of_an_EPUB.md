# 2.6. A File-by-File Breakdown of an EPUB

## Key vocabulary

**Resource** is any file that is part of the EPUB package: content documents, stylesheets, images, fonts, navigation files, audio, video, and so on. In EPUB, a “resource” is only truly part of the book if it is declared in the OPF manifest.

**Asset** is a resource that is typically referenced by content documents (images, fonts, audio) rather than being a “top-level” structural file like the OPF.

**Content document** is an XHTML (or HTML/XHTML in EPUB 3) file that contains actual readable book content (chapters, title page, front matter). Content documents are what the spine sequences for reading order.

**Stylesheet** is a CSS file that controls presentation of content documents. In EPUB it behaves like web CSS, but with reader constraints and overrides.

**Core media type** is a file format that reading systems are expected to support as part of the baseline standard (for example, XHTML, CSS, JPEG/PNG/GIF/SVG). In practice, “core” is the safe set you can rely on across most readers.

## Why this chapter is useful (and how to use it)

Up to this point, we’ve talked about the EPUB container as layers: a ZIP container that points to an OPF package document, which in turn defines the reading order and inventory of content and assets. This chapter is a practical companion to that mental model. The goal here is that when you unzip an EPUB and see a dozen unfamiliar files, you can quickly answer three questions about each one: what it is, how it is discovered, and what breaks when it is wrong.

There are two important framing rules for this walkthrough.

The first rule is that filenames and folder names are mostly conventions, not requirements. The spec cares about relationships and references, not whether you used `OEBPS/` or `EPUB/`.

The second rule is that, inside the package layer, the OPF manifest is the authoritative “inventory.” Many reading systems can be forgiving, but if you want consistent behavior across devices, treat “is it declared in the manifest?” as a hard requirement.

## A typical EPUB directory tree

Let’s start with a representative example. Your EPUB won’t always look exactly like this, but the roles are common even when names differ.

```text
MyBook.epub
├─ mimetype
├─ META-INF/
│  └─ container.xml
└─ OEBPS/
   ├─ content.opf
   ├─ nav.xhtml
   ├─ toc.ncx
   ├─ text/
   │  ├─ titlepage.xhtml
   │  ├─ chapter-01.xhtml
   │  └─ chapter-02.xhtml
   ├─ styles/
   │  └─ book.css
   ├─ images/
   │  ├─ cover.jpg
   │  └─ figure-1.png
   └─ fonts/
      └─ SomeFont-Regular.otf
```

In the next sections we’ll walk down this tree from the outside in, following the same “pointer chain” a reading system uses: container → OPF → navigation → content → assets.

## `mimetype`

The `mimetype` file is a tiny text file at the root of the EPUB container. Its contents are usually exactly:

```text
application/epub+zip
```

What makes `mimetype` important is not what it says, but how it is packaged. The EPUB container rules require that `mimetype` be the first file in the ZIP and that it be stored uncompressed. This is one of those details that feels arbitrary until you ship a book that opens in one app but fails validation or fails in another.

From a debugging perspective, `mimetype` is a “does this even look like an EPUB?” check. If a validator complains about it, fix it early. It is a container-level correctness issue, not a content issue.

## `META-INF/container.xml`

Inside `META-INF/`, the key file is `container.xml`. This file answers the reading system’s first question: “Where is the OPF package document?”

A typical `container.xml` looks like:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<container
  version="1.0"
  xmlns="urn:oasis:names:tc:opendocument:xmlns:container">
  <rootfiles>
    <rootfile
      full-path="OEBPS/content.opf"
      media-type="application/oebps-package+xml"/>
  </rootfiles>
</container>
```

The only part you usually care about is `full-path`. That is the authoritative location of the OPF. If you can’t find the OPF, or the OPF moved but `container.xml` wasn’t updated, many reading systems will treat the EPUB as broken before they even try to render a single chapter.

You may also see multiple `<rootfile>` entries. That’s legal, and it’s one reason robust tooling should always read `container.xml` rather than assuming a default path.

## The OPF package document (`content.opf` / `package.opf`)

The OPF is the “brain” of the EPUB, and we covered it in depth in Chapter 2.4. In this chapter, treat it as the file that ties everything else together. It contains metadata, a manifest (inventory), and a spine (reading order).

The practical way to approach an OPF in a real EPUB is to scan it for three kinds of information.

First, identify the manifest items for navigation, especially the EPUB 3 nav document (`properties="nav"`) and any NCX item included for compatibility.

Second, identify the spine and confirm that the content documents you expect to be read are listed there, in the correct order.

Third, scan for items that are commonly mis-declared: images and fonts, which often fail because of missing items or wrong media types.

Here’s a small reminder of what a manifest entry looks like:

```xml
<item id="ch1" href="text/chapter-01.xhtml" media-type="application/xhtml+xml"/>
```

And a spine entry that references it:

```xml
<itemref idref="ch1"/>
```

If those IDs don’t line up, the book’s structure is wrong even if the file exists on disk.

## The NCX file (`toc.ncx`) (EPUB 2 navigation)

If an EPUB contains a `.ncx` file, it is usually there for EPUB 2 navigation support, or for compatibility with older reading systems. The NCX is a data file that defines a hierarchical ToC using `<navMap>` and nested `<navPoint>` entries.

A navPoint links into the book using a `<content src="...">` reference, which can include a fragment identifier. For example:

```xml
<navPoint id="navpoint-2-1" playOrder="3">
  <navLabel><text>Chapter 2, Section A</text></navLabel>
  <content src="text/chapter-02.xhtml#section-a"/>
</navPoint>
```

If you edit chapters (rename files, split chapters, change IDs), NCX is one of the first things that drifts out of sync. That’s why many EPUB toolchains regenerate navigation late in the pipeline: it’s “derived data” that must match the final file names and IDs.

For EPUB 3-first publications, NCX is optional. For compatibility-driven publications, it is common to include both NCX and the nav document, as described in Chapter 2.5.

## The nav document (`nav.xhtml`) (EPUB 3 navigation)

In EPUB 3, the primary navigation structure is an XHTML file often called `nav.xhtml`. It contains one or more `<nav>` blocks, including a ToC nav (`epub:type="toc"`) and optionally landmarks (`epub:type="landmarks"`) and a page list (`epub:type="page-list"`).

The nav document is only “the nav document” if two things are true.

It must be declared in the OPF manifest with `properties="nav"`.

And it must contain the correct semantics inside the XHTML (especially `epub:type="toc"` for the ToC).

A minimal ToC in a nav document looks like a nested list of links:

```xml
<nav epub:type="toc" id="toc">
  <h1>Table of Contents</h1>
  <ol>
    <li><a href="text/chapter-01.xhtml">Chapter 1</a></li>
    <li><a href="text/chapter-02.xhtml#section-a">Chapter 2, Section A</a></li>
  </ol>
</nav>
```

The nav document is one of the nicest parts of EPUB 3 for web developers because it is essentially “normal HTML navigation.” The downside is that it is also easy to accidentally break, because it relies on correct links and stable fragment IDs.

## XHTML content documents (the actual chapters)

The “meat” of an EPUB is a set of content documents: XHTML files containing the book’s text and structure (headings, paragraphs, lists, images, footnotes, and so on). These are the files listed in the OPF spine as the primary reading order.

Footnotes and endnotes are not a separate “note format” inside the container. They live in the same XHTML world as chapters do. In some books, each chapter contains its own note bodies in-place. In other books, note bodies are collected into one or more separate XHTML files (for example `text/notes.xhtml` or `text/endnotes.xhtml`) and chapters link into them using fragment IDs. In either case, the notes XHTML file is still a normal content document: it should appear in the OPF manifest, and it may appear in the OPF spine (often with `linear="no"`) depending on whether you want it to be part of the primary reading flow or only reachable via links.

At a high level, you can think of a content document as a web page that happens to be packaged inside a ZIP. It links to CSS, references images, and uses anchors for internal navigation. But it lives under stricter rules than the open web, especially if it’s authored as XHTML.

A content document typically begins like this:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<html xmlns="http://www.w3.org/1999/xhtml" xml:lang="en">
  <head>
    <title>Chapter 1</title>
    <link rel="stylesheet" type="text/css" href="../styles/book.css"/>
  </head>
  <body>
    <h1>Chapter 1</h1>
    <p>This is the first paragraph.</p>
  </body>
</html>
```

The important thing to notice is how web-like it is, and how path-sensitive it is. The `href="../styles/book.css"` path is resolved relative to the content document’s location, not relative to the EPUB root. When you reorganize folders, the most common breakage is that relative links silently become wrong.

Content documents are also where XHTML strictness shows up. If you make an XML well-formedness mistake (like forgetting to close a tag, mismatching quotes, or nesting tags incorrectly), some reading systems will refuse to render the file, and others will render it unpredictably. Part III of this book goes deeper on that, but it’s worth noting here because it’s one of the most common “it worked in a browser, but not in an EPUB reader” surprises.

## CSS stylesheets

EPUB CSS is “normal CSS” in syntax, but it runs in a constrained environment. Reading systems vary widely in how much CSS they support, and many allow the reader to override publisher styles (font family, font size, margins, line height, night mode, and more).

In an EPUB, a stylesheet is typically referenced from content documents with a `<link>` element, and the stylesheet itself is declared in the OPF manifest:

```xml
<item id="css" href="styles/book.css" media-type="text/css"/>
```

The most important practical lesson is that CSS in EPUB is often best-effort. You should aim for robust typography and layout, but avoid relying on fragile browser-specific behavior. Some readers strip certain properties. Some ignore embedded fonts. Some treat margins and page-break rules differently. That “gap between spec and reality” is not theoretical; you’ll feel it immediately when you test across readers.

Part III (Chapter 3.2) will go deep on CSS support patterns. For now, treat CSS as an important file type that frequently causes cross-reader differences even when the EPUB is structurally valid.

## Images

Images in EPUB are ordinary image files stored inside the container and referenced from content documents. Common formats include JPEG and PNG, and many EPUBs also use SVG.

The two structural rules that matter are consistent references and correct manifest entries. An `<img>` tag in a content document might look like:

```xml
<img src="../images/figure-1.png" alt="A diagram explaining the EPUB layers."/>
```

But the image should also be declared in the OPF manifest with the correct media type:

```xml
<item id="fig1" href="images/figure-1.png" media-type="image/png"/>
```

In practice, “missing images” bugs often come from one of three things: the file isn’t in the container at the expected path, the `src` path is wrong (relative path mistakes), or the manifest entry is missing or has the wrong media type.

Cover images deserve special mention. EPUB 3 uses `properties="cover-image"` on the manifest item to declare the cover image resource. Many books also include a “cover page” XHTML file that displays the cover as a content document, because some reading systems expect a visual cover page in the spine.

## Embedded fonts

Fonts are resources (often `.otf`, `.ttf`, or `.woff`) stored inside the EPUB. They are declared in the manifest and referenced from CSS using `@font-face`.

A manifest entry might look like:

```xml
<item id="font-regular" href="fonts/SomeFont-Regular.otf" media-type="font/otf"/>
```

And the CSS might include:

```css
@font-face {
  font-family: "SomeFont";
  src: url("../fonts/SomeFont-Regular.otf");
}
```

Fonts are one of the most “spec meets reality” file types. Some reading systems respect embedded fonts, some ignore them, and some only partially support them. There are also licensing constraints: many fonts cannot be legally embedded unless you have the right license. This becomes a practical engineering issue when you build an EPUB pipeline for real publishing.

Some workflows also use font obfuscation, which is a lightweight mechanism intended to discourage casual extraction. It’s not strong DRM, but it does affect how the font file is stored and referenced. Chapter 3.5 focuses on this in detail.

## Other assets: audio, video, and JavaScript

EPUB 3 allows richer media types like audio and video, and it allows JavaScript in some forms. Whether those features work reliably depends heavily on the reading system.

From a file-structure perspective, these assets follow the same pattern as everything else: they live inside the container, they are declared in the OPF manifest with correct media types, and they are referenced from content documents.

An audio file might be declared like:

```xml
<item id="audio1" href="audio/track-01.mp3" media-type="audio/mpeg"/>
```

And referenced from a content document with something like:

```xml
<audio controls="controls" src="../audio/track-01.mp3">
  <p>Your reading system does not support audio.</p>
</audio>
```

Video is similar. JavaScript is trickier. Many reading systems restrict or disable it, and some storefront pipelines reject interactive EPUBs even if the spec allows them. The practical rule is: if you include multimedia or JavaScript, treat it as progressive enhancement and design the EPUB to remain readable and navigable without it.

## Common pitfalls and gotchas

The first and most notorious gotcha is packaging correctness at the container layer. The `mimetype` file must be first and uncompressed. Many “zip this folder” workflows get this wrong without realizing it. Some reading systems forgive it; validators and distribution pipelines often do not.

The second gotcha is manifest drift. If a resource exists in the ZIP but isn’t declared in the OPF manifest, you have created a book that depends on reader forgiveness. Some readers will still load the resource if referenced directly. Others won’t. EPUBCheck will complain. The safe rule is: every referenced resource should have a manifest item.

The third gotcha is media-type correctness. The manifest is where you declare MIME types. Small mistakes here can produce failures that look random across reading systems. If you declare an SVG incorrectly, or a font with the wrong media type, or an XHTML file as `text/html`, you may see inconsistent behavior and confusing validation output.

The fourth gotcha is path resolution. EPUB is full of relative paths: OPF `href`s are relative to the OPF location, content document links are relative to the content document location, and nav/NCX links are relative to their own file locations. Reorganizing folders is one of the quickest ways to break a book without changing a single line of “content.”

The fifth gotcha is XHTML strictness. Browsers recover from a lot of broken HTML. EPUB readers often do not, especially when they treat content documents as XML. A single unclosed tag can cause a whole chapter to disappear or render partially. This is why validation and well-formedness checks matter in EPUB workflows.

Finally, remember that “valid” does not mean “consistent rendering.” CSS support, font support, navigation UI, and interactive features vary widely across reading systems. The file-by-file structure gives you correctness and predictability, but the last step is always real-world testing across the readers that matter for your audience.
