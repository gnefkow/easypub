# 3.5 Embedded Fonts

Fonts are one of the most emotionally loaded parts of digital publishing. A font can make a book feel classic, modern, playful, academic, or “like a brand.” If you come from the web, embedding fonts in an EPUB looks familiar: you bundle font files and reference them from CSS using `@font-face`. What makes EPUB different is that many reading systems intentionally let the user override your typography choices. That means embedded fonts are often best thought of as “strong suggestions,” not absolute commands.

This chapter explains why you might embed fonts, how embedded fonts work mechanically inside an EPUB package, what “font obfuscation” is and why it exists, and what the real-world support landscape looks like. We’ll also cover the least glamorous but most important piece: licensing. Many font problems in EPUB are not technical—they’re legal.

## Key vocabulary (for this chapter)

An **embedded font** is a font file (such as `.otf`, `.ttf`, `.woff`, or `.woff2`) packaged inside the EPUB so the book can render with that font even when the device doesn’t already have it installed. **@font-face** is the CSS rule that declares a font family name and points to the font file’s URL. A **glyph** is a specific shape in a font (for example, the letter “A,” the em dash “—,” or a Chinese character). **Font fallback** is what happens when the requested font can’t render a character or isn’t available; the reading system chooses another font. **Subsetting** is the process of generating a smaller font file that includes only the glyphs a book needs.

**Obfuscation** (in EPUB font discussions) is a lightweight, reversible transformation applied to a font file to discourage casual extraction. It is not DRM and it is not secure encryption. You will also see **WOFF** and **WOFF2** (web-optimized font wrappers), **OTF** (OpenType), and **TTF** (TrueType). Finally, **user override** is the reading system feature that allows the reader to pick a preferred font, size, and spacing, often disabling embedded fonts entirely.

## Why embed fonts at all?

There are a few good reasons to embed fonts in an EPUB, and they are usually tied to meaning rather than decoration.

One reason is **character coverage**. If your book includes non-Latin scripts, uncommon diacritics, mathematical symbols, or special punctuation, relying on a device’s built-in fonts can produce missing glyph boxes (“tofu”) or inconsistent rendering. Embedding a font that includes the glyphs you need makes the text portable.

Another reason is **typographic intent**. Some books use a specific font to convey tone (a children’s book, a branded nonfiction title, or a book with design-heavy headings). Embedding a font can help preserve that intent, especially for titles, headings, or ornamental elements.

A third reason is **consistency across platforms**. EPUB is a multi-reader ecosystem. When you don’t embed fonts, your book will be rendered with whatever default serif/sans font each reading system chooses. That can be acceptable (and often desirable), but if you need more predictable typography, embedding can narrow the range of outcomes.

The counterweight to all of this is that many readers prefer to choose their own fonts for comfort and accessibility. In practice, you get the best results when you embed fonts selectively and make sure the book still looks good when those fonts are ignored.

## Font formats in EPUB: OTF/TTF vs WOFF/WOFF2

At a file level, you are packaging font binaries inside a ZIP. The most common “raw” formats are OpenType (`.otf`) and TrueType (`.ttf`). WOFF and WOFF2 are wrappers designed for the web that often reduce file size through compression and metadata structure.

From a compatibility perspective, `.otf` and `.ttf` are broadly supported across many reading systems, but they are often larger. WOFF is also widely supported in modern EPUB readers, while WOFF2 support is more mixed. If you care about broad compatibility, a common strategy is to package WOFF (and optionally WOFF2) and test on your target readers.

One important practical detail is that “the font file exists” does not guarantee “the font will render.” Fonts can be malformed, can have unusual tables, or can trigger bugs in specific engines. EPUB work rewards conservative choices: use well-known, well-tested fonts when possible, and avoid exotic builds unless you’re prepared to test.

## How fonts are bundled in an EPUB package

Bundling a font is mechanically similar to bundling images: you place the file in the EPUB directory structure, declare it in the OPF manifest with the correct media type, and make sure it is referenced by something in the content (in this case, referenced from CSS).

A typical layout looks like this:

```text
OEBPS/
  content.opf
  text/
    chapter-01.xhtml
  styles/
    book.css
    fonts.css
  fonts/
    Merriweather-Regular.woff
    Merriweather-Italic.woff
    Merriweather-Bold.woff
```

You can put fonts in any folder, but a dedicated `fonts/` directory makes the package easier to reason about.

### Declaring fonts in the OPF manifest

Fonts should be listed in the `<manifest>` section of your OPF file, just like stylesheets and images. The `media-type` value depends on the font format, and this is one of those places where EPUB’s “spec vs reality” gap shows up. Reading systems and validators expect valid media types, but different production pipelines have historically used different strings.

Here is a practical example:

```xml
<manifest>
  <item id="css-book" href="styles/book.css" media-type="text/css" />
  <item id="css-fonts" href="styles/fonts.css" media-type="text/css" />

  <item id="font-regular" href="fonts/Merriweather-Regular.woff" media-type="font/woff" />
  <item id="font-italic" href="fonts/Merriweather-Italic.woff" media-type="font/woff" />
  <item id="font-bold" href="fonts/Merriweather-Bold.woff" media-type="font/woff" />

  <item id="ch1" href="text/chapter-01.xhtml" media-type="application/xhtml+xml" />
</manifest>
```

The key idea is that the manifest is your publication’s inventory. The reading system should not have to “discover” fonts by crawling the ZIP. If the font isn’t declared, you may get lucky in some readers and fail in others, and validators will often flag it.

## Referencing fonts from CSS with `@font-face`

Fonts are usually applied through CSS, not directly in XHTML. You declare the font family using `@font-face`, and then you use that family name in normal `font-family` declarations.

Here is a minimal `@font-face` rule:

```css
@font-face {
  font-family: "Merriweather";
  src: url("../fonts/Merriweather-Regular.woff") format("woff");
  font-weight: 400;
  font-style: normal;
}
```

Let’s unpack what each part means.

`font-family` is the name you will use later in your CSS. It does not need to match the file name, and it does not need to match the font’s internal “PostScript name.” In EPUB workflows it’s usually best to choose a simple, consistent family name.

`src` is the path to the font file. Like every other URL in EPUB, it is resolved relative to the CSS file that contains it. If `fonts.css` is in `styles/` and fonts are in `fonts/`, you commonly need `../fonts/...`.

`font-weight` and `font-style` are crucial. They tell the reading system which font file corresponds to which style. If you omit them, some readers will guess, and some will not. If you declare a regular font as `font-weight: 400` but then request `font-weight: 700` in your styles, the reader may synthesize bold (fake it), or it may fall back to a different font entirely.

### Declaring multiple faces (regular, italic, bold)

Real books use italic and bold constantly, so you almost always need multiple `@font-face` rules for a family:

```css
@font-face {
  font-family: "Merriweather";
  src: url("../fonts/Merriweather-Regular.woff") format("woff");
  font-weight: 400;
  font-style: normal;
}

@font-face {
  font-family: "Merriweather";
  src: url("../fonts/Merriweather-Italic.woff") format("woff");
  font-weight: 400;
  font-style: italic;
}

@font-face {
  font-family: "Merriweather";
  src: url("../fonts/Merriweather-Bold.woff") format("woff");
  font-weight: 700;
  font-style: normal;
}
```

Now when your content uses `<em>` or `<strong>`, or when your CSS applies `font-style: italic` or `font-weight: bold`, the reading system has a real font file to use instead of synthesizing.

### Applying the embedded font (with sane fallbacks)

Once declared, you can use the font like any other:

```css
body {
  font-family: "Merriweather", serif;
}
```

That trailing `serif` matters. If the embedded font is ignored or fails to load, the reader still has a reasonable fallback. A robust EPUB stylesheet treats embedded fonts as an enhancement, not as the single point of failure for legibility.

In practice, many publishers embed fonts only for headings and keep body text in the reader’s default font, because body text is where user preferences matter most.

## Glyph coverage and “missing character” surprises

Even with embedded fonts, you can still run into missing characters. If you embed a font that doesn’t include the glyphs for your content (for example, curly quotes, em dashes, or accented characters), the reader may fall back to another font for just those characters. That can look like inconsistent typography within a single line.

Subsetting can make this problem worse if you subset too aggressively. A subset font might cover the letters in your current manuscript but not the characters added later during editing (a new language quote, a new symbol, or a new name with a diacritic). Subsetting can be a powerful file-size optimization, but it needs a careful workflow.

## The reality of font support: readers, settings, and overrides

From the EPUB spec perspective, embedded fonts are a normal feature. From the real world perspective, embedded fonts live at the intersection of reader preference and platform policy.

Many reading systems expose a “Publisher font” toggle. When it’s on, `@font-face` is honored (more often). When it’s off, embedded fonts may be ignored entirely, and the reading system will render with the user’s chosen font. Some systems always allow user choice, especially for body text. Others partially honor embedded fonts, but still adjust line height, margins, or hyphenation in ways that change how the font feels.

As an EPUB editor, the practical goal is not “force my font everywhere.” The goal is “ensure the book is readable and attractive when the font is used, and still excellent when it isn’t.”

## Font obfuscation: what it is and why it exists

Font licensing often allows embedding fonts inside a publication but prohibits redistributing the raw font file for general use. Because an EPUB is a ZIP file, font files can be extracted with basic tools. Obfuscation exists to add a small barrier: it makes the font file not immediately installable after extraction, while still being recoverable by reading systems that know how to reverse the transformation.

It’s important to be clear about what obfuscation is not. It is not strong encryption. It is not DRM. It does not stop a determined user. It is a “stumbling block” intended to satisfy certain licensing expectations.

There are two commonly encountered obfuscation approaches in the EPUB ecosystem:

The IDPF method (historically documented for EPUB 2/OCF) transforms the first 1040 bytes of the font using XOR with a key derived from the publication’s unique identifier (a SHA‑1 digest of the identifier). The obfuscation is declared in `META-INF/encryption.xml` using an `EncryptionMethod` algorithm URI of `http://www.idpf.org/2008/embedding`.

Some Adobe tools and workflows have historically used an Adobe-specific method identified in `encryption.xml` by an algorithm URI such as `http://ns.adobe.com/pdf/enc#RC`. You will most often see this when an EPUB is exported from Adobe InDesign with fonts included by default.

If you ever see an `encryption.xml` file in `META-INF/`, that’s a sign that something in the package is encrypted or obfuscated. In the case of embedded fonts, it usually means the fonts are obfuscated and tied to the publication identifier. This has a practical consequence: changing the publication identifier after export can break the ability to de-obfuscate the fonts, causing fonts to stop rendering in some readers.

In day-to-day EPUB work, obfuscation is usually something you do only when you must. If your fonts are licensed for embedding without obfuscation, you generally prefer to keep the EPUB simple and compatible.

## Licensing: the non-negotiable part

Not all fonts permit embedding in ebooks. Some require a specific ebook license. Some allow embedding only if the font is obfuscated. Some prohibit embedding entirely. And “I own the font on my computer” is not the same as “I have the right to redistribute it inside an EPUB.”

The practical best practice is: treat fonts like third-party code dependencies. Check the license terms, keep a record of what you’re allowed to do, and choose fonts whose licensing matches your distribution goals. If you’re producing EPUBs for commercial distribution, “safe” often means using fonts that explicitly allow ebook embedding or using open-source fonts with clear embedding rights.

## Common pitfalls and gotchas

The first gotcha is assuming embedded fonts will always be used. Many reading systems let users disable publisher fonts, and some readers will override fonts for accessibility by default. Your typography needs to degrade gracefully, which usually means using good fallbacks and avoiding layout that depends on a specific font’s metrics.

The second gotcha is missing a face. If you embed only the regular font file but the text contains italics and bold, the reading system may synthesize those styles (with ugly results) or fall back to a different font for emphasized text. Declaring the full set of faces you actually use is the difference between “professional” and “almost right.”

The third gotcha is path confusion. The `src: url(...)` in `@font-face` is resolved relative to the CSS file, not relative to the XHTML file and not relative to the OPF. A font can be correctly declared in the OPF manifest and still fail to load because the CSS path is wrong.

The fourth gotcha is incorrect or inconsistent media types in the OPF manifest. Validators and some reading systems care. If you change font formats (say, from `.otf` to `.woff`) and forget to update the media type, you can create subtle failures.

The fifth gotcha is glyph coverage, especially with subsetting. A subset font that works today can fail tomorrow when content changes. Missing glyphs often show up as boxes or as random fallback characters from a different font. If you subset, make sure your workflow can regenerate the subset after edits and that it covers all languages and punctuation you use.

The sixth gotcha is font obfuscation side effects. If `META-INF/encryption.xml` ties an obfuscation key to the publication identifier, changing `dc:identifier` (or the OPF `unique-identifier` target) after export can make the font unreadable. Obfuscation can also reduce compatibility with readers that don’t implement that specific method. Unless you have a licensing or distribution requirement, keeping fonts unobfuscated is often the safer option.

Finally, remember that EPUB is a real-world ecosystem, not a lab environment. Embedded fonts are useful, but they are also one of the most inconsistent features across readers. When you treat embedded fonts as progressive enhancement—supported when possible, ignored when necessary—you end up with books that feel polished everywhere instead of perfect in one app and broken everywhere else.
